Unit 1 · Getting started

1overview

What Java is

In one line: Java is a general-purpose language that compiles to portable bytecode, which runs on any machine that has a Java Virtual Machine.

You write Java in plain text. A compiler turns it into bytecode — not machine code for one specific chip, but instructions for a virtual machine. Any device with a JVM (Windows, Mac, Linux, Android, servers) can run that same bytecode. That's the famous promise: write once, run anywhere.

Everyday analogy

Bytecode is like a recipe written in a universal cooking language. You don't rewrite it for every kitchen — each kitchen just needs a translator (the JVM) who reads the universal recipe and works the local stove.

Comparison

Practice problem

You compile a program on a Mac and email the .class file to a friend on Windows. Will it run? Why or why not?

Show solution
Yes — bytecode is platform-independent. As long as your friend has a compatible JVM installed, the same .class file runs unchanged. That's the whole point of "write once, run anywhere."

2setup

JDK, JRE, JVM

In one line: The JVM runs bytecode, the JRE is the JVM plus core libraries, and the JDK is the JRE plus the tools you need to build Java.

These three acronyms confuse every beginner. Think of them as nested boxes. To just run a finished Java app you only need a JVM/JRE. To write and compile Java, you need the full JDK, because it contains javac (the compiler) and other developer tools. Install the JDK — it includes everything.

Everyday analogy

The JVM is the engine. The JRE is the engine plus a fueled, drivable car. The JDK is that car plus the whole garage — wrenches, lift, and diagnostics — so you can also build and repair cars, not just drive them.

A teammate says "I installed Java but javac isn't found." What did they most likely install, and what should they install instead?

Show solution
They likely installed only a JRE (or a runtime-only package). javac ships with the JDK. Install a full JDK (e.g. from Adoptium / Temurin) and javac will be available.

3first code

Hello, World

In one line: Every classic Java program lives inside a class and starts at a special method called main.

Let's read the smallest real program line by line. Don't memorize the ceremony yet — just see the shape. The main method is the door the JVM walks through to start your program.

Everyday analogy

main is the front door of a building. No matter how many rooms (methods) exist inside, visitors (the JVM) always enter through that one labeled door.

java · Hello.java
// a class — the container for your code
public class Hello {
    // the entry point — the JVM starts here
    public static void main(String[] args) {
        System.out.println("Hello, World!");
    }
}

What each part means

PieceMeaning
public class HelloDefines a class named Hello. The file should be named Hello.java.
public static void mainThe exact signature the JVM looks for to start running.
String[] argsCommand-line arguments passed to your program (an array of text).
System.out.println(...)Prints a line of text to the console.
Practice problem

Change the program so it prints your name on one line and "Learning Java" on a second line.

Show solution
Add a second println call inside main: System.out.println("Ada"); then System.out.println("Learning Java");. Each println ends with a newline, so they appear on separate lines.
4workflow

Compile & run

In one line: Use javac to compile your .java file into a .class file, then java to run it.

This two-step cycle — compile, then run — is the heart of working in Java. Modern JDKs also let you run a single source file in one step, which is great for learning.

terminal · the classic two steps
# 1) compile: Hello.java  →  Hello.class (bytecode)
javac Hello.java

# 2) run: note there's NO .class extension here
java Hello
# → Hello, World!

# shortcut (Java 11+): compile + run a single file at once
java Hello.java
What to notice

You run java Hello (the class name), not java Hello.class. A very common beginner error is adding the extension. The JVM wants the class name and finds the .class file itself.

jshell — try Java with no file at all

Run jshell in your terminal to get an interactive Java prompt. Type 2 + 2 and press Enter — it answers instantly. It's a fantastic scratchpad for testing a single expression without the class/main ceremony. Type /exit to leave.

Practice problem

You run javac Hello.java and get no output, no .class file appears, and no error. What's the most likely explanation?

Show solution
Silent success does produce a .class file. If none appears, you're probably in the wrong directory, or the file name/case doesn't match. Check with ls (Mac/Linux) or dir (Windows) that Hello.java is really in the current folder.

Unit 2 · Variables & types

6basics

Variables

In one line: A variable is a named, typed box that holds a value you can read and change.

In Java you must state the type when you first declare a variable. After that, the box can only ever hold that kind of value. You can reassign the value, but not the type.

Everyday analogy

Labeled jars on a shelf. The "sugar" jar holds sugar; you can refill it, but you can't suddenly put soup in it. The label (type) is fixed; the contents (value) can change.

java · declaring and using variables
int age = 30;          // declare + assign in one line
double price = 9.99;
String name = "Ada";
boolean isOnline = true;

age = 31;                // reassigning the value is fine
// age = "old";        // ERROR: can't put text in an int box

System.out.println(name + " is " + age);  // Ada is 31
What to notice

The + with a String joins text together (concatenation). When one side is text, Java converts the number to text and glues them. This is the everyday way to build output.

Practice problem

Declare a variable for the number of items in a cart, set it to 3, then increase it to 4 and print "Cart: 4".

Show solution
int items = 3; then items = 4; (or items = items + 1;), then System.out.println("Cart: " + items);.

In one line: Java has eight built-in "primitive" types for raw values like whole numbers, decimals, true/false, and single characters.

Primitives are the simplest values — they aren't objects, they hold the value directly. You'll use int, double, boolean, and char constantly; the rest are situational.

Everyday analogy

Primitives are like denominations of cash — fixed, simple units. A char is a single coin ('A'); a String (later) is a whole wallet of coins arranged in order.

java · the everyday four
int count = 10;

Predict the output: System.out.println(10 / 3); and System.out.println(10.0 / 3);

Show solution
First prints 3 (integer division throws away the remainder). Second prints 3.3333333333333335 because a double is involved, so the calculation keeps decimals.
8text

Strings

In one line: A String is a sequence of characters; it's an object (not a primitive) and it's immutable — methods return new Strings rather than changing the original.

Strings are everywhere: names, messages, file contents. Two things trip people up. First, they're objects with useful methods. Second, they never change in place — every "modifying" method hands you a brand-new String.

Everyday analogy

A String is like a printed page. To "change" a word you don't erase it — you print a fresh page with the edit. The old page still exists unchanged.

java · common String methods
String s = "Hello";

System.out.println(s.length());        // 5
System.out.println(s.toUpperCase());   // HELLO  (new String)
System.out.println(s.charAt(0));      // H
System.out.println(s.substring(1, 4)); // ell
System.out.println(s);                 // Hello — original is unchanged

// compare CONTENT with .equals, not ==
String a = "hi", b = "hi";
System.out.println(a.equals(b));      // true (correct way)
Practice problem

Why is "hi" == new String("hi") often false, while "hi".equals(new String("hi")) is true?

Show solution
== compares whether two references point to the same object in memory; new String(...) makes a distinct object, so it's a different reference → false. .equals() compares the actual characters, which match → true. Rule: for Strings, always use .equals() to compare content.

9conversion

Type casting

In one line: Going to a bigger type is automatic; squeezing into a smaller type needs an explicit cast and may lose data.

Sometimes you have an int but need a double, or vice versa. Widening (small → big, e.g. int → double) happens silently and safely. Narrowing (big → small) forces you to ask explicitly with a cast, because information can be lost.

Everyday analogy

Pouring a small cup into a big jug always fits (widening). Pouring a big jug into a small cup might overflow — so you must do it deliberately and accept the spill (narrowing cast).

java · widening vs narrowing
int n = 7;
double d = n;            // widening: automatic, 7.0

double pi = 3.99;
int truncated = (int) pi; // narrowing: explicit cast, gives 3 (NOT rounded)

// turning text into a number is parsing, not casting
int parsed = Integer.parseInt("42");   // 42
double dp = Double.parseDouble("3.5"); // 3.5
Practice problem

What does (int) 9.99 produce, and why isn't it 10?

Show solution
It produces 9. Casting a double to int truncates (drops the decimal part) — it does not round. To round, use Math.round(9.99) which gives 10.
10operators

Operators

In one line: Operators combine or compare values — arithmetic does math, comparison asks questions, logical operators chain conditions.

Three families cover almost everything you'll do daily. Comparison operators always produce a boolean (true/false), which feeds straight into the if statements you'll learn next.

FamilyOperatorsExample → result
Arithmetic+ - * / %10 % 3 → 1 (remainder)
Comparison== != > < >= <=5 > 3 → true
Logical&& || !true && false → false
Everyday analogy

Logical && ("and") is a strict bouncer: every condition must pass. || ("or") is a lenient bouncer: any one passing is enough. ! ("not") flips the answer.

java · operators in action
int a = 10, b = 3;
System.out.println(a + b);    // 13
System.out.println(a % b);    // 1  — modulo: the remainder

boolean adult = age >= 18;
boolean canVote = adult && isCitizen;  // both must be true

int x = 5;
x += 3;   // shorthand for x = x + 3  → 8
x++;     // increment by 1            → 9
Practice problem

The modulo operator % is the classic even/odd test. Write a boolean expression that is true when n is even.

Show solution
n % 2 == 0. A number is even exactly when dividing by 2 leaves no remainder. (Odd would be n % 2 != 0.)

Unit 3 · Control flow

11decisions

if / else

In one line: if runs a block only when a condition is true; else if and else handle the other cases.

This is how programs make decisions. The condition in the parentheses must be a boolean. Java checks branches top to bottom and runs the first one whose condition is true, then skips the rest.

Everyday analogy

A series of "if it's raining, take an umbrella; else if it's sunny, take sunglasses; else, just go." You stop at the first matching weather.

java · grading with if/else
int score = 82;

if (score >= 90) {
    System.out.println("A");
} else if (score >= 80) {
    System.out.println("B");   // ← this runs
} else if (score >= 70) {
    System.out.println("C");
} else {
    System.out.println("Try again");
}

// the ternary: a compact if/else that returns a value
String status = (score >= 60) ? "pass" : "fail";
Practice problem

If score = 95, which branch runs in the example above? Why don't the later branches also run?

Show solution
The first branch (score >= 90 → "A") runs. Once a branch matches, the whole if/else-if chain is done — Java never checks the remaining conditions. They're mutually exclusive.
12decisions

switch

In one line: switch picks one of many branches based on a single value — cleaner than a long if/else chain when you're matching exact values.

Use switch when you compare one variable against many fixed options (days, menu choices, enum values). Modern Java added a tidy "switch expression" with -> arrows and no fall-through.

java · classic switch
int day = 3;
switch (day) {
    case 1: System.out.println("Mon"); break;
    case 2: System.out.println("Tue"); break;
    case 3: System.out.println("Wed"); break; // ← runs
    default: System.out.println("Other");
}
java · modern switch expression (Java 14+) — no break needed
String name = switch (day) {
    case 1 -> "Mon";
    case 2 -> "Tue";
    case 3 -> "Wed";
    default -> "Other";
};
System.out.println(name);   // Wed
Practice problem

In the classic switch, what happens if you forget the break after case 1?

Show solution
"Fall-through": execution keeps running into the next case's code until it hits a break or the end. So matching case 1 would also print "Tue". The arrow form (->) avoids this entirely — it never falls through.
13repetition

Loops

In one line: Loops repeat a block of code — for when you know the count, while when you loop until a condition changes, for-each to walk every item in a collection.

Repetition is what makes computers powerful. The for loop bundles three parts: start, keep-going condition, and step. The enhanced for-each is the cleanest way to visit every element.

Everyday analogy

A for loop is "do 10 push-ups" (counted). A while loop is "keep stirring until it thickens" (condition-based). A for-each is "greet every guest in the room."

java · the three loops
// for: start; condition; step
for (int i = 1; i <= 3; i++) {
    System.out.println("count " + i);   // 1, 2, 3
}

// while: repeat as long as the condition holds
int n = 5;
while (n > 0) {
    System.out.print(n + " ");     // 5 4 3 2 1
    n--;
}

// for-each: visit every element, no index needed
String[] pets = {"cat", "dog", "fish"};
for (String pet : pets) {
    System.out.println(pet);
}
Practice problem

Write a for loop that prints the even numbers from 2 to 10.

Show solution
for (int i = 2; i <= 10; i += 2) { System.out.println(i); } — stepping by 2 hits 2, 4, 6, 8, 10. (Alternatively loop 1..10 and print only when i % 2 == 0.)
14loop control

break & continue

In one line: break exits a loop entirely; continue skips to the next iteration.

These give you fine control inside a loop. break means "I'm done, leave now." continue means "skip the rest of this round, go to the next one."

Everyday analogy

Going through a stack of mail: break = "found the one bill I needed, stop sorting." continue = "junk mail — toss it and grab the next envelope."

java · break vs continue
// break: stop at the first number divisible by 7
for (int i = 1; i <= 100; i++) {
    if (i % 7 == 0) {
        System.out.println("first multiple of 7: " + i); // 7
        break;
    }
}

// continue: print 1..10 but skip 5
for (int i = 1; i <= 10; i++) {
    if (i == 5) continue;   // skip the rest for i==5
    System.out.print(i + " "); // 1 2 3 4 6 7 8 9 10
}
Practice problem

A while (true) loop never stops on its own. How do you safely exit it from the inside?

Show solution
Put a condition inside the loop that calls break when met — e.g. read input until the user types "quit", then break. The infinite-loop-plus-break pattern is common for menus and game loops; just make sure some path always reaches the break.

Unit 4 · Methods

15methods

Defining methods

In one line: A method is a named, reusable block of code you can call by name as many times as you like.

Methods let you name a job once and reuse it everywhere — the core of keeping code clean (DRY: "don't repeat yourself"). A method has a return type, a name, parameters in parentheses, and a body.

Everyday analogy

A method is a kitchen recipe card. Write "make pancakes" once; then anyone can just say "make pancakes" instead of re-explaining every step each morning.

java · defining and calling a method
public class Greeter {

    // a method that does a job and returns nothing (void)
    static void greet(String name) {
        System.out.println("Hello, " + name + "!");
    }

    public static void main(String[] args) {
        greet("Ada");     // Hello, Ada!
        greet("Linus");   // Hello, Linus!  — reused, no repetition
    }
}
What to notice

void means the method returns no value — it just does something (here, printing). We mark these helpers static for now so main can call them directly; you'll learn what static really means in Unit 7.

Practice problem

Write a void method printLine() that prints a row of 10 dashes, and call it twice.

Show solution
static void printLine() { System.out.println("----------"); } then call printLine(); twice in main. No parameters, no return — it just does one job.
16methods

Parameters & return

In one line: Parameters are inputs you pass in; return hands a value back out.

A method that returns a value declares the type it gives back (instead of void) and ends with a return. This is how methods compute answers you can use elsewhere.

Everyday analogy

A vending machine: you put coins in (parameters), it gives a snack out (return value). The inside steps are hidden — you only deal with inputs and the output.

java · a method that returns a value
// takes two ints, returns their sum
static int add(int a, int b) {
    return a + b;          // hands the result back
}

static double average(int x, int y) {
    return (x + y) / 2.0;  // 2.0 keeps it a double
}

// using the returned values
int total = add(3, 4);          // 7
System.out.println(average(3, 4)); // 3.5
Practice problem

Write a method boolean isEven(int n) that returns true when n is even.

Show solution
static boolean isEven(int n) { return n % 2 == 0; }. The expression n % 2 == 0 is already a boolean, so you can return it directly — no if needed.
17methods

Overloading

In one line: Overloading lets several methods share one name as long as their parameter lists differ.

Java tells overloaded methods apart by their parameters (count or types), not their name. The compiler picks the matching version based on what you pass. This is why println can accept an int, a String, or a double — it's overloaded many times.

Everyday analogy

The word "open" works for a door, a file, or a meeting. Same verb, different object — and everyone understands which "open" you mean from context. Overloading is that, for methods.

java · one name, several signatures
static int area(int side) {            // square
    return side * side;
}
static int area(int width, int height) { // rectangle
    return width * height;
}

System.out.println(area(5));      // 25  → calls the 1-arg version
System.out.println(area(5, 3));   // 15  → calls the 2-arg version
Practice problem

Can you overload two methods that differ only by return type (e.g. one returns int, one returns double, both take the same parameters)?

Show solution
No. Overloading is decided by the parameter list only; return type alone is not enough. Two methods with identical names and identical parameters are a compile error, even if return types differ.
18methods

Recursion

In one line: A recursive method calls itself on a smaller version of the problem until it hits a base case that stops the chain.

Recursion solves problems that break into smaller copies of themselves. Two parts are mandatory: a base case (when to stop) and a recursive case (call yourself on something smaller). Miss the base case and you get infinite recursion — a stack overflow.

Everyday analogy

Russian nesting dolls. To "open all dolls," open this one, then do the same to the doll inside — until you reach the tiny solid doll that can't open (the base case).

java · factorial by recursion
static int factorial(int n) {
    if (n <= 1) return 1;        // base case — stops the recursion
    return n * factorial(n - 1); // recursive case — smaller problem
}

System.out.println(factorial(4)); // 24  (4·3·2·1)
Practice problem

What goes wrong if you remove the if (n <= 1) return 1; line?

Show solution
With no base case, the method calls itself forever (..., fact(0), fact(-1), ...). Each call uses stack memory, so it eventually crashes with a StackOverflowError. Every recursion needs a reachable base case.

Unit 5 · OOP basics

19OOP

Classes & objects

In one line: A class is a blueprint; an object is a real thing built from it with new.

Object-oriented programming models the world as objects that bundle data (fields) with behavior (methods). You define the blueprint once (the class), then stamp out as many objects as you need.

Everyday analogy

A cookie cutter (class) vs the cookies (objects). One cutter, many cookies — each cookie is its own thing with its own sprinkles, but they share the same shape.

java · define a class, create objects
class Dog {
    String name;   // field (data)
    String breed;  // field (data)

    void bark() {      // method (behavior)
        System.out.println(name + " says woof!");
    }
}

// create two distinct objects from the one blueprint
Dog rex = new Dog();
rex.name = "Rex";
rex.bark();   // Rex says woof!

Dog milo = new Dog();
milo.name = "Milo";
milo.bark();  // Milo says woof!
Practice problem

After Dog rex = new Dog(); and Dog milo = new Dog();, does changing rex.name affect milo.name?

Show solution
No. Each new Dog() creates a separate object with its own copy of the fields. They share the blueprint, not the data. Changing one object's fields never touches another's.
20OOP

Fields & methods

In one line: Fields hold an object's data (its state); methods are the things an object can do (its behavior).

Together, fields and methods are a class's members. You access them with dot notation: object.field and object.method(). Methods can read and update the object's own fields.

Everyday analogy

A phone: its fields are battery level and brightness (state); its methods are call, text, take photo (behavior). The behavior often reads or changes the state.

java · methods that use fields
class BankAccount {
    double balance;   // state

    void deposit(double amount) {  // behavior changes state
        balance = balance + amount;
    }

    double getBalance() {           // behavior reads state
        return balance;
    }
}

BankAccount acct = new BankAccount();
acct.deposit(100);
acct.deposit(50);
System.out.println(acct.getBalance()); // 150.0
Practice problem

Add a withdraw(double amount) method. What real-world rule should it check before subtracting?

Show solution
void withdraw(double amount) { if (amount <= balance) balance -= amount; }. It should refuse to overdraw — only subtract if there's enough balance. Guarding state changes like this is exactly what encapsulation (Lesson 23) protects.
21OOP

Constructors

In one line: A constructor is special code that runs when you create an object, used to set up its starting fields.

Setting each field by hand after new is tedious and error-prone. A constructor lets you pass the starting values right when you build the object. It has the same name as the class and no return type.

Everyday analogy

Ordering a custom sandwich: instead of receiving a plain one and adding everything yourself, you state your fillings at the counter and it's built correctly from the start.

java · a constructor sets fields at creation
class Dog {
    String name;
    String breed;

    // constructor: same name as class, no return type
    Dog(String name, String breed) {
        this.name = name;     // this.name = the field
        this.breed = breed;
    }
}

// now creation and setup happen in one clean step
Dog rex = new Dog("Rex", "Lab");
System.out.println(rex.name);  // Rex
What to notice

If you write no constructor, Java gives you a free empty one (the "default constructor") so new Dog() still works. The moment you define any constructor, that free one disappears.

Practice problem

You add the constructor above, then try new Dog(); with no arguments. It won't compile — why?

Show solution
Once you define a constructor that takes parameters, Java no longer provides the free no-argument default. To allow new Dog() again, add a second (overloaded) no-arg constructor explicitly.
22OOP

The this keyword

In one line: this refers to the current object — most often used to tell a field apart from a parameter with the same name.

Inside a method or constructor, this means "the object this code is running on." Its top use is disambiguation: when a parameter is named the same as a field, this.field points to the field and the bare name points to the parameter.

Everyday analogy

In a room of people named Alex, saying "I'll handle my own task" removes all ambiguity. this is the object saying "my" — referring to itself, not the others.

java · this resolves the name clash
class Point {
    int x, y;

    Point(int x, int y) {
        this.x = x;   // this.x = the field;  x = the parameter
        this.y = y;
    }

    void show() {
        System.out.println("(" + this.x + ", " + this.y + ")");
    }
}

new Point(3, 4).show();   // (3, 4)
Practice problem

In the constructor, what would x = x; (without this) do?

Show solution
It assigns the parameter to itself and leaves the field untouched — a silent bug. The closer name (the parameter) wins both sides, so the field x stays at its default 0. That's exactly why this.x = x; is needed.
23OOP

Encapsulation

In one line: Hide fields with private and expose controlled access through methods, so an object protects its own data.

Encapsulation means a class guards its internals. You mark fields private (off-limits from outside) and offer getters to read and setters to change — letting the setter validate before allowing a change. This prevents invalid states.

Everyday analogy

An ATM. You can't reach into the vault directly; you go through buttons (methods) that enforce rules (PIN, balance checks). The cash (private data) stays protected behind a controlled interface.

Why is a validating setter better than just making the field public?

A public field can be set to any value, including invalid ones (like a negative age), with no checks. A setter is a gatekeeper: it can validate, reject bad input, log, or transform the value. Encapsulation keeps objects in valid states.

24OOP

Inheritance

In one line: A subclass extends a superclass to reuse its fields and methods, then add or change its own.

Inheritance models an "is-a" relationship: a Cat is an Animal. The subclass automatically gets everything from the parent and can add more. This avoids copy-pasting shared code.

Everyday analogy

A sports car is a car. It inherits wheels, doors, and an engine from "car," and adds its own features (turbo, spoiler). You don't redesign wheels from scratch.

java · extends gives you the parent's members
class Animal {
    String name;
    void eat() { System.out.println(name + " is eating"); }
}

class Cat extends Animal {   // Cat IS-A Animal
    void meow() { System.out.println(name + " says meow"); }
}

Cat c = new Cat();
c.name = "Milo";
c.eat();    // inherited from Animal → Milo is eating
c.meow();   // Cat's own method        → Milo says meow
Practice problem

Should Car extend Engine? Use the "is-a" test to decide.

Show solution
No — a car is not an engine; a car has an engine. That's a "has-a" relationship, modeled by composition (a field of type Engine inside Car), not inheritance. Use extends only for genuine "is-a" relationships.
25OOP

Polymorphism

In one line: The same method call behaves differently depending on the actual object — a subclass can override a parent method.

"Polymorphism" = "many forms." You can hold different subclass objects in a variable of the parent type, call one method, and each object runs its own version. This lets you write code against the general type and let specifics vary.

Everyday analogy

You tell every employee "make a sound." The dog barks, the cat meows, the duck quacks. One command, many correct behaviors — each responds in its own way.

java · overriding + dynamic dispatch
class Animal {
    String sound() { return "..."; }
}
class Cat extends Animal {
    @Override
    String sound() { return "meow"; }   // replaces the parent's version
}
class Dog extends Animal {
    @Override
    String sound() { return "woof"; }
}

// a parent-typed array can hold any subclass
Animal[] zoo = { new Cat(), new Dog() };
for (Animal a : zoo) {
    System.out.println(a.sound()); // meow, then woof
}
What to notice

The @Override annotation isn't required, but always use it. It tells the compiler "I mean to replace a parent method" — so if you misspell the name or get the signature wrong, you get an error instead of a silent new method.

Practice problem

The variable is typed Animal, but it prints "meow" for a Cat. How does Java know to run Cat's version?

Show solution
Java decides which sound() to run based on the object's actual runtime type (Cat), not the variable's declared type (Animal). This is called dynamic dispatch — the real object's method always wins.
26OOP

Abstract classes

In one line: An abstract class is a partial blueprint you can't instantiate; it can force subclasses to fill in specific methods.

Sometimes a base concept is too general to create directly — what would a plain "Shape" even look like? An abstract class can hold shared code and declare abstract methods (no body) that every subclass must implement.

Everyday analogy

"Vehicle" is abstract — you never buy a generic "vehicle," you buy a car or a bike. But every vehicle must define how it moves; the concept demands it without saying how.

java · abstract base + concrete subclasses
abstract class Shape {
    abstract double area();   // no body — subclasses MUST define it

    void describe() {            // shared, concrete method
        System.out.println("Area is " + area());
    }
}

class Circle extends Shape {
    double r;
    Circle(double r) { this.r = r; }
    double area() { return Math.PI * r * r; }  // required
}

// Shape s = new Shape();   // ERROR: can't instantiate abstract
Shape s = new Circle(2);
s.describe();   // Area is 12.566...
Practice problem

What's the compile error if Circle forgets to implement area()?

Show solution
A concrete subclass must implement every inherited abstract method, or it stays abstract itself. Java reports that Circle isn't abstract and doesn't override the abstract method area(). Either implement it, or mark Circle abstract too.

Interfaces

In one line: An interface is a pure contract — a list of methods a class promises to provide via implements.

Where a class can extend only one parent, it can implement many interfaces. Interfaces describe what something can do without saying how. They're how Java does "multiple inheritance of behavior" and they power much of the standard library.

Everyday analogy

A power socket is an interface: any appliance that fits the plug shape works, no matter what's inside it. The socket promises "supply power"; the appliance promises "accept this shape."

java · implement a contract
interface Drawable {
    void draw();         // abstract by default — no body
}

class Circle implements Drawable {
    public void draw() { System.out.println("○"); }
}
class Square implements Drawable {
    public void draw() { System.out.println("□"); }
}

// code to the interface — works for any Drawable
Drawable[] shapes = { new Circle(), new Square() };
for (Drawable d : shapes) d.draw();   // ○ then □

28OOP

super & the Object class

In one line: super reaches the parent (its constructor or overridden methods), and every class secretly extends Object, the root of all classes.

Two final OOP pieces. super(...) calls the parent constructor; super.method() calls the parent's version of an overridden method. And every class in Java ultimately inherits from Object, which is why every object has methods like toString() for free.

Everyday analogy

Building on a family recipe: super is "start from grandma's base sauce, then add my twist." You reuse the parent's work instead of redoing it.

java · super + overriding toString
class Animal {
    String name;
    Animal(String name) { this.name = name; }
}

class Dog extends Animal {
    String breed;
    Dog(String name, String breed) {
        super(name);          // call Animal's constructor first
        this.breed = breed;
    }
    @Override
    public String toString() {  // override Object's default
        return name + " (" + breed + ")";
    }
}

System.out.println(new Dog("Rex", "Lab")); // Rex (Lab)
What to notice

Overriding toString() controls how your object prints. Without it, println(dog) shows something ugly like Dog@1b6d3586. A good toString() makes debugging far easier.

Practice problem

Why must super(name) be the very first line of the Dog constructor?

Show solution
A subclass object can't be set up until its parent part is initialized. Java requires the super(...) call to come first so the parent's fields exist before the subclass touches anything. If you omit it, Java inserts an implicit super() call to a no-arg parent constructor.

29keyword

static

In one line: static members belong to the class itself, not to any one object — shared by all and callable without creating an instance.

A normal field lives in each object; a static field lives once on the class and is shared. A static method belongs to the class, so you call it on the class name (like Math.sqrt(...)) without making an object. That's why main is static — it runs before any object exists.

Everyday analogy

Each student has their own grade (instance field), but the school's name is shared by everyone (static field). You don't need a specific student to know the school's name.

java · static (shared) vs instance (per-object)
class Counter {
    static int total = 0;  // ONE shared across all objects
    int id;                // each object gets its own

    Counter() {
        total++;          // bump the shared count
        id = total;       // this object's own number
    }
}

new Counter(); new Counter(); new Counter();
System.out.println(Counter.total); // 3 — call on the class

// static methods: no object needed
System.out.println(Math.max(7, 2)); // 7
Practice problem

Can a static method directly use a non-static (instance) field? Why or why not?

Show solution
No. A static method runs without any particular object, so there's no instance whose field it could read — "which object's field?" has no answer. It can only use static members directly, or work with an object you pass in explicitly.
30keyword

final & constants

In one line: final means "cannot change" — a final variable can't be reassigned, a final method can't be overridden, a final class can't be extended.

final locks things down. Its most common use is constants: combine static final for a single shared, unchangeable value. Convention names constants in ALL_CAPS.

Everyday analogy

A constant is like a number etched in stone — the speed of light, the days in a week. Once set, it never changes, and writing it in stone (final) prevents accidents.

java · constants and final variables
static final double PI = 3.14159;   // shared constant, ALL_CAPS
static final int MAX_USERS = 100;

final int birthYear = 1995;
// birthYear = 1996;   // ERROR: cannot reassign a final variable

System.out.println(PI * 2);   // 6.28318
What to notice

For an object reference, final stops you from pointing it at a different object — but the object's own fields can still change. final locks the reference, not the contents.

Practice problem

Why use a named constant MAX_USERS instead of just writing 100 throughout your code?

Show solution
A named constant is self-documenting (the name explains the meaning) and changeable in one place. If the limit becomes 200, you edit one line instead of hunting for every 100 — some of which might mean something else entirely. Avoiding such "magic numbers" prevents bugs.
31special type

enums

In one line: An enum defines a type whose value can only be one of a fixed, named set of options.

When a value must be one of a small, known list — days of the week, order status, card suits — an enum makes that explicit and type-safe. You can't accidentally assign an invalid value, and they pair beautifully with switch.

Everyday analogy

A traffic light has exactly three states: RED, YELLOW, GREEN. There's no "purple" option. An enum encodes that "these and only these" rule into the type system.

java · enum + switch
enum Status { ACTIVE, PAUSED, CLOSED }   // the only valid values

Status s = Status.ACTIVE;

String msg = switch (s) {
    case ACTIVE -> "Running";
    case PAUSED -> "On hold";
    case CLOSED -> "Finished";
};
System.out.println(msg);  // Running

// s = "active";   // ERROR — not a Status; type-safe
Practice problem

Why is an enum safer than using int codes (0 = active, 1 = paused, 2 = closed)?

Show solution
With int codes nothing stops you from assigning 5 (meaningless) or mixing up which number means what. An enum restricts values to the valid set at compile time, reads clearly (Status.ACTIVE vs 0), and lets the compiler warn you if a switch misses a case.

Collections and generics (next units) only work with objects, not raw primitives. Wrapper classes bridge the gap. Java does the conversion for you: putting an int where an Integer is expected is "autoboxing"; the reverse is "unboxing."

Arrays

In one line: An array is a fixed-size, indexed list of values that all share one type.

Arrays are the most basic way to hold many values. Their size is fixed when created, and you access elements by a zero-based index. Asking for arr.length gives the count (a field, not a method).

Everyday analogy

An egg carton with a fixed number of slots, numbered from 0. You can swap what's in slot 3, but you can't add a 13th slot to a 12-egg carton.

java · creating and looping an array
int[] nums = {10, 20, 30, 40};   // literal initializer

System.out.println(nums[0]);      // 10  — first element
System.out.println(nums.length);   // 4   — a field, no ()
nums[2] = 99;                    // change slot 2

for (int x : nums) {            // for-each: visit each value
    System.out.print(x + " ");  // 10 20 99 40
}

// empty array of size 3 (all default to 0)
int[] blank = new int[3];
Practice problem

For the array above, what happens at nums[4]?

Show solution
It throws ArrayIndexOutOfBoundsException. Valid indexes for a 4-element array are 0–3. Index 4 is past the end. This is one of the most common runtime errors — always loop with i < arr.length, not i <= arr.length.

ArrayList (List)

In one line: An ArrayList is a resizable list — like an array that grows and shrinks as you add and remove items.

Arrays have a fixed size; most real programs don't know the size up front. ArrayList handles growth for you. You declare it to the List interface and use methods like add, get, remove, and size. The <String> part is a generic — it says what type the list holds.

Everyday analogy

A shopping list on paper (array) has limited lines. A notes app list (ArrayList) lets you keep adding items forever — it just makes room.

java · a growable list of Strings
import java.util.ArrayList;
import java.util.List;

List<String> tasks = new ArrayList<>();

tasks.add("email");
tasks.add("lunch");
tasks.add("code");

System.out.println(tasks.size());   // 3
System.out.println(tasks.get(0));   // email
tasks.remove("lunch");            // list shrinks to 2

for (String t : tasks) {
    System.out.println(t);          // email, code
}
What to notice

The <> on the right (the "diamond") is empty — Java infers the type from the left side. And declaring the variable as List (the interface) rather than ArrayList is good practice: it keeps your code flexible.

Practice problem

You wrote List<String> names but tried names.add(5). What happens?

Show solution
Compile error. The generic <String> means the list accepts only Strings, so adding an int is rejected at compile time. That type safety is the whole benefit of generics — bugs caught before the program even runs.

HashMap (Map)

In one line: A HashMap stores key→value pairs for fast lookup by key.

When you want to look something up by name rather than by position — a phone book, a count per word, a user by ID — use a Map. Each key is unique and maps to one value. Lookups are very fast.

Everyday analogy

A dictionary: you look up a word (key) to get its definition (value). You don't scan page by page — you jump straight to the entry.

java · key → value lookups
import java.util.HashMap;
import java.util.Map;

Map<String, Integer> ages = new HashMap<>();

ages.put("Ada", 36);     // key "Ada" → value 36
ages.put("Linus", 54);

System.out.println(ages.get("Ada"));          // 36
System.out.println(ages.containsKey("Bob")); // false
System.out.println(ages.getOrDefault("Bob", 0)); // 0 (safe default)

// putting an existing key overwrites its value
ages.put("Ada", 37);    // Ada is now 37
Practice problem

What does ages.get("Nobody") return for a key that was never added?

Show solution
It returns null (no entry for that key). If you then try to use it as an int, you'd hit a NullPointerException on unboxing. Prefer getOrDefault("Nobody", 0) or check containsKey first.

In one line: A HashSet holds unique values — duplicates are silently ignored — and answers "is this in here?" very fast.

Use a Set when you care about membership and uniqueness, not order or position. Adding a value that's already present does nothing. It's the natural tool for deduplicating or checking "have I seen this before?"

Everyday analogy

A guest list where each name appears once. Inviting someone twice doesn't add a second entry — they're already on it.

You have a List with duplicate emails and want only the unique ones. What's the one-line trick?

Generics

In one line: Generics let a class or method work with a type you supply later, while keeping full type safety.

You've already used generics: List<String> is a list parameterized to hold Strings. The angle-bracket part is the type argument. Writing your own generic class uses a placeholder (commonly T) that callers fill in.

Everyday analogy

A shipping box is generic — it doesn't care if it holds books or shoes. But once you label it "BOOKS," you shouldn't toss shoes in. Generics enforce that label.

java · a generic Box class
// T is a placeholder for whatever type the caller picks
class Box<T> {
    private T item;
    void put(T item) { this.item = item; }
    T get() { return item; }
}

Box<String> sbox = new Box<>();
sbox.put("hello");
String s = sbox.get();   // no cast needed — type-safe

Box<Integer> ibox = new Box<>();
ibox.put(42);
// ibox.put("oops");   // compile error — Box is Integer-only
Practice problem

Before generics, lists held plain Object and you had to cast on the way out. What two problems did generics fix?

Show solution
(1) Safety: you could accidentally add the wrong type and only find out at runtime with a ClassCastException — generics catch it at compile time. (2) Convenience: you no longer write manual casts like (String) list.get(0); the type is known.

Lambdas

In one line: A lambda is a tiny anonymous function you can pass around as a value, written with the -> arrow.

Lambdas let you treat a chunk of behavior like data — hand it to a method, store it, run it later. They work with "functional interfaces" (interfaces with a single abstract method). This is the foundation of streams (next lesson).

Everyday analogy

Instead of writing a full instruction manual (a named class), you jot a sticky note: "given x, do this." A lambda is that sticky note — small, inline, ready to use.

java · lambdas replace boilerplate
import java.util.List;
import java.util.function.Predicate;

// a lambda: (input) -> result
Predicate<Integer> isEven = n -> n % 2 == 0;
System.out.println(isEven.test(4));  // true

// passing behavior straight into a method
List<String> names = List.of("Ada", "Bo", "Cy");
names.forEach(name -> System.out.println(name));

// sort with a lambda comparator (by length)
List<String> words = new java.util.ArrayList<>(List.of("bbb", "a", "cc"));
words.sort((x, y) -> x.length() - y.length());
System.out.println(words);  // [a, cc, bbb]
Practice problem

What does the lambda (x, y) -> x + y do, and roughly what kind of interface does it fit?

Show solution
It takes two inputs and returns their sum. It fits a functional interface with one method that accepts two arguments and returns a value — for example BinaryOperator<Integer> or a custom interface with a single apply(x, y) method.
40functional

Streams

In one line: A stream is a pipeline that transforms a collection step by step — filter, map, collect — in a readable, declarative chain.

Streams let you describe what you want, not the loop mechanics. You chain operations: filter keeps matching items, map transforms each one, and a terminal operation like collect or sum produces the result. Streams don't change the original collection.

java · a stream pipeline
import java.util.List;
import java.util.stream.Collectors;

List<Integer> nums = List.of(1, 2, 3, 4, 5, 6);

List<Integer> result = nums.stream()
    .filter(n -> n % 2 == 0)   // keep evens → 2,4,6
    .map(n -> n * n)            // square each → 4,16,36
    .collect(Collectors.toList());
System.out.println(result);     // [4, 16, 36]

// a terminal operation that reduces to one number
int sum = nums.stream().mapToInt(n -> n).sum();
System.out.println(sum);        // 21
Practice problem

Rewrite this as a stream: from a List<String> names, keep names longer than 3 letters and uppercase them.

Show solution
names.stream().filter(n -> n.length() > 3).map(String::toUpperCase).collect(Collectors.toList());. String::toUpperCase is a "method reference" — a shorthand lambda meaning "call toUpperCase on each element."
41functional

Optional

In one line: Optional is a box that explicitly holds "a value" or "nothing," helping you avoid surprise NullPointerExceptions.

Returning null for "not found" is a famous source of crashes — callers forget to check. Optional<T> makes the absence visible in the type, and gives clean ways to provide a fallback or act only if a value exists.

Everyday analogy

A gift box that's clearly labeled "may be empty." You're prompted to check before reaching in — no nasty surprise of grabbing nothing and panicking.

java · safe handling of "maybe a value"
import java.util.Optional;

Optional<String> found = Optional.of("Ada");
Optional<String> missing = Optional.empty();

// provide a fallback instead of crashing
System.out.println(found.orElse("unknown"));   // Ada
System.out.println(missing.orElse("unknown")); // unknown

// run code only if a value is present
found.ifPresent(name -> System.out.println("Hi " + name)); // Hi Ada
Practice problem

How is returning Optional<User> from a "find user" method better than returning null?

Show solution
The return type itself announces "this might be empty," so callers are nudged (and can be forced) to handle the missing case via orElse/ifPresent. A bare null looks like a normal value and is easy to forget to check — leading to NullPointerExceptions deep in the code.

Exceptions: try / catch / finally

In one line: An exception is Java's way of saying "something went wrong here" — you try risky code and catch the problem instead of letting the program crash.

When something fails — a missing file, bad input, dividing by zero — Java throws an exception object. If nobody catches it, the program stops and prints a stack trace. Wrapping risky code in try/catch lets you respond gracefully. finally always runs, whether or not there was an error — perfect for cleanup.

Everyday analogy

A trapeze act: the try is the daring swing, the catch is the safety net for when things go wrong, and finally is the crew tidying up the ring no matter how the act ended.

KindMust you handle it?Examples
CheckedYes — compiler forces a try/catch or throwsIOException, SQLException
Unchecked (runtime)No — but usually a bug to fixNullPointerException, ArithmeticException
java · catching a problem instead of crashing
public class Safe {
  public static void main(String[] args) {
    try {
      int result = 10 / 0;        // throws ArithmeticException
      System.out.println(result);  // never runs
    } catch (ArithmeticException e) {
      System.out.println("Can't divide by zero: " + e.getMessage());
    } finally {
      System.out.println("This always runs.");
    }
  }
}
In practice

Catch the most specific exception you can handle, and avoid empty catch blocks that silently swallow errors — at minimum log them. Catching the broad Exception everywhere hides real bugs.

Practice problem

What is the difference between a checked and an unchecked exception, and which one does the compiler force you to deal with?

Show solution
A checked exception (like IOException) must be either caught or declared with throws — the compiler refuses to build otherwise. An unchecked exception (a RuntimeException such as NullPointerException) does not have to be declared or caught; it usually signals a programming bug you should fix rather than catch.

Throwing & custom exceptions

In one line: You can throw your own exceptions — and make new exception types — so failures carry clear, domain-specific meaning.

Built-in exceptions are generic. When your program has its own rules ("balance can't go negative"), a custom exception communicates intent far better than a vague IllegalArgumentException. You create one by extending Exception (checked) or RuntimeException (unchecked), then throw it when the rule is broken.

Everyday analogy

A referee's specific whistle: instead of just "foul!", they signal exactly which rule was broken. A custom exception is that precise call — everyone immediately knows what went wrong.

java · a meaningful, domain-specific failure
// 1. define your own type
class InsufficientFundsException extends RuntimeException {
  InsufficientFundsException(String message) {
    super(message);   // pass message up to Exception
  }
}

class Account {
  private double balance = 100;

  void withdraw(double amount) {
    if (amount > balance) {
      throw new InsufficientFundsException(
        "Tried to withdraw " + amount + ", only " + balance + " available");
    }
    balance -= amount;
  }
}
In practice

Rule of thumb: extend RuntimeException for programming/validation errors callers usually can't recover from, and Exception (checked) for recoverable, expected conditions you want callers to consciously handle.

Practice problem

Why might throw new InsufficientFundsException("...") be better than throw new RuntimeException("...")?

Show solution
A named type lets callers catch exactly that situation (e.g. catch (InsufficientFundsException e)) and react to it specifically, while still ignoring unrelated errors. It also makes the code self-documenting: the type name states the business rule that was violated. A bare RuntimeException forces callers to inspect the message string, which is fragile.

Reading & writing files

In one line: Modern Java reads and writes files in a line or two using Files and Path, with try-with-resources closing things automatically.

File access can fail (missing file, no permission), so these methods throw the checked IOException — you must handle it. The modern java.nio.file API (Path, Files) is the friendliest: Files.writeString and Files.readString handle whole small files in one call. For big files, stream line by line with a BufferedReader inside try-with-resources, which guarantees the file is closed even if an error occurs.

Everyday analogy

Try-with-resources is a self-closing door: you walk through to grab what you need, and it shuts behind you automatically — you can't accidentally leave the file "open" and leaking.

java · write then read a text file safely
import java.nio.file.*;
import java.io.*;

public class FileDemo {
  public static void main(String[] args) {
    Path path = Path.of("notes.txt");
    try {
      // write the whole file in one call
      Files.writeString(path, "line one\nline two\n");

      // read it all back
      String all = Files.readString(path);
      System.out.println(all);

      // stream big files line by line; auto-closes the reader
      try (BufferedReader r = Files.newBufferedReader(path)) {
        String line;
        while ((line = r.readLine()) != null) {
          System.out.println("> " + line);
        }
      }
    } catch (IOException e) {
      System.out.println("File problem: " + e.getMessage());
    }
  }
}
In practice

Prefer Files.readString/writeString for small config or text files, and the streaming BufferedReader approach for large logs you can't fit in memory. Always wrap closeable resources in try-with-resources.

Practice problem

What does the try (BufferedReader r = ...) form guarantee that a plain try block does not?

Show solution
Try-with-resources automatically calls close() on the reader when the block finishes — normally or via an exception. This prevents resource leaks (open file handles) without you having to write a finally block to close it manually. Any resource implementing AutoCloseable can be declared this way.

Records: data classes without the boilerplate

In one line: A record is a one-line way to make an immutable data carrier — Java writes the constructor, getters, equals, hashCode, and toString for you.

Plain "data holder" classes used to need dozens of lines of repetitive code. A record declares the fields once in its header, and the compiler generates everything sensible automatically. The fields are final — a record is immutable by design, which makes it safe to share.

Everyday analogy

A pre-printed form: you just fill in the blanks (the fields) and all the standard structure — borders, labels, signature line — comes ready-made. No drawing the form by hand each time.

// declares fields x and y, plus constructor + accessors + equals/hashCode/toString

public class Demo {
  public static void main(String[] args) {
    Point p = new Point(3, 4);
    System.out.println(p.x());      // 3  (accessor is x(), not getX())

  }
}
In practice

Reach for records for DTOs, coordinates, API responses, map keys — anything that's "just data." You can still add methods and validate inputs in a compact constructor if a rule must hold.

Practice problem

Name two methods a record generates for you automatically, and state one consequence of its fields being final.

Show solution
It generates (any two of) the canonical constructor, the per-field accessors like x(), equals, hashCode, and toString. Because the fields are final, a record is immutable — you can't change it after creation, which makes it inherently thread-safe and safe to use as a HashMap key.

var, text blocks & switch expressions

In one line: Three modern conveniences — var for inferred types, """ text blocks for multi-line strings, and switch expressions that return a value.

These features cut noise without changing what your code means. var lets the compiler infer a local variable's type from the right-hand side (it's still strongly typed). Text blocks let you write multi-line strings without ugly \n and +. Switch expressions use -> and produce a value directly, with no fall-through bugs.

Everyday analogy

var is like saying "same as that" instead of repeating a long label; a text block is a quote pasted verbatim instead of retyped line by line; a switch expression is a vending machine that hands you an item back, rather than just rearranging shelves.

java · less ceremony, same meaning
// var: compiler infers the type (here, ArrayList<String>)
var names = new ArrayList<String>();
names.add("Ada");

// text block: multi-line string, no \n soup
String json = """
    {
      "name": "Ada",
      "active": true
    }
    """;

// switch expression: returns a value, arrow syntax, no break needed
int day = 3;
String type = switch (day) {
  case 6, 7 -> "weekend";
  default    -> "weekday";
};
System.out.println(type);  // weekday
In practice

Use var when the type is obvious from the right side (it can hurt readability when it isn't). Text blocks shine for embedded JSON, SQL, or HTML. Prefer switch expressions over old switch statements to avoid forgotten break bugs.

Practice problem

Does var x = 10; make x dynamically typed like in Python? Explain.

Show solution
No. var only asks the compiler to infer the type at compile time — here x is fixed as an int permanently. You can't later assign a String to it. Java stays statically and strongly typed; var just saves you from writing the type name explicitly.

Concurrency basics

In one line: Concurrency runs multiple tasks at once; Java gives you threads and higher-level tools, but shared mutable data needs careful coordination.

A thread is an independent path of execution. Running work on several threads can speed things up (downloading files, handling requests). The catch: when two threads touch the same mutable data at the same time, you get race conditions — unpredictable bugs. Rather than managing raw threads, prefer an ExecutorService (a managed pool), and protect shared state with synchronized or atomic/concurrent types.

Everyday analogy

Several cooks in one kitchen are faster — until two reach for the same knife at once. A thread pool is a head chef assigning tasks; synchronized is a rule that only one cook touches the shared station at a time.

java · run tasks on a managed thread pool
import java.util.concurrent.*;

public class PoolDemo {
  public static void main(String[] args) throws Exception {
    ExecutorService pool = Executors.newFixedThreadPool(3);

    for (int i = 1; i <= 5; i++) {
      int id = i;
      pool.submit(() ->
        System.out.println("Task " + id + " on " + Thread.currentThread().getName()));
    }

    pool.shutdown();  // no new tasks; finishes the queued ones
  }
}
In practice

Don't share mutable objects across threads casually. Favor immutable data (records!), ConcurrentHashMap, and atomic classes like AtomicInteger. Modern Java 21+ also adds lightweight virtual threads for massive numbers of concurrent tasks.

Practice problem

What is a race condition, and name one way to prevent it when two threads update the same counter.

Show solution
A race condition is when the program's result depends on the unpredictable timing of threads accessing shared mutable data — for example, two threads reading the same counter, both incrementing the old value, and one update getting lost. Prevent it by synchronizing access (a synchronized block/method) or by using a thread-safe type such as AtomicInteger whose incrementAndGet() is atomic.
Go deeper

Explore virtual threads (Project Loom, stable in Java 21) and CompletableFuture for composing asynchronous work.

equals() & hashCode()

In one line: Override equals to define what "same value" means, and always override hashCode alongside it — or hash-based collections will misbehave.

By default, equals compares object identity (are these the exact same object in memory?). Usually you want value equality (two Point(3,4) objects are equal). When you override equals, you must override hashCode so that equal objects produce the same hash — otherwise HashMap and HashSet will fail to find your objects. Records do all this for you automatically.

Everyday analogy

Two identical house keys: equals says "these open the same lock." hashCode is the labeled hook they hang on. If equal keys hang on different hooks, you'll never find your key in the drawer — that's a broken hashCode.

java · value equality done correctly
import java.util.Objects;

class Point {
  final int x, y;
  Point(int x, int y) { this.x = x; this.y = y; }

  @Override
  public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof Point p)) return false;
    return x == p.x && y == p.y;
  }

  @Override
  public int hashCode() {
    return Objects.hash(x, y);  // consistent with equals
  }
}
In practice

Easiest correct option: use a record, which generates both methods for you. If you write them by hand, use Objects.equals(...) and Objects.hash(...), and include exactly the fields that define equality.

Practice problem

You override equals but forget hashCode, then put your objects in a HashSet. What goes wrong?

Show solution
Two objects that are equals may land in different hash buckets because their default hashCode values differ. The HashSet could then store visual duplicates, and lookups like contains may fail to find an object that's logically present. Equal objects must return the same hash code — that's the contract you broke.

Immutability & common pitfalls

In one line: Prefer immutable data and learn Java's classic traps — == vs equals, null, and shared mutable references — to avoid most beginner bugs.

Immutable objects can't change after creation, so they're predictable and thread-safe. Beyond that, a handful of pitfalls catch nearly everyone: comparing objects with == (which checks identity, not value), forgetting that an object can be null, and exposing internal mutable collections that callers can secretly modify.

import java.util.*;

// 1) == compares identity, not contents
String a = new String("hi");
String b = new String("hi");
System.out.println(a == b);       // false (two different objects)
System.out.println(a.equals(b));  // true  (same characters)

// 2) guard against null at the boundary
String name = Objects.requireNonNullElse(maybeName(), "guest");

// 3) don't hand out your mutable internals
List<String> safe = List.of("a", "b");   // immutable list
// safe.add("c");  // would throw UnsupportedOperationException
In practice

Make fields final wherever possible, favor records and List.of/Map.of, and treat every reference from outside as possibly null. Most "mysterious" Java bugs trace back to one of these three.

Practice problem

Why might "hi" == new String("hi") be false while "hi".equals(new String("hi")) is true?

Show solution
== compares references — whether both sides point to the exact same object in memory. new String("hi") deliberately creates a distinct object, so it's a different reference from the pooled literal "hi", making == false. equals compares the actual characters, which match, so it returns true. For value comparison of objects, always use equals.

A complete program (it uses records, enums, collections, streams & switch — all at once)

This ~45-line task tracker pulls together the whole course. Save it as TaskTracker.java and run it with a single command — no build tool needed: java TaskTracker.java (Java 11+).

java · TaskTracker.java — runnable
import java.util.*;
import java.util.stream.*;

public class TaskTracker {

  // enum (Unit 7): a fixed set of states
  enum Status { TODO, DOING, DONE }

  // record (Unit 11): an immutable data carrier with auto everything
  record Task(int id, String title, Status status) {}

  public static void main(String[] args) {
    // collections (Unit 8): a growable list of tasks
    List<Task> tasks = new ArrayList<>();
    tasks.add(new Task(1, "Learn variables", Status.DONE));
    tasks.add(new Task(2, "Practice loops",  Status.DOING));
    tasks.add(new Task(3, "Build a project", Status.TODO));

    // enhanced for (Unit 3) + switch expression (Unit 11)
    System.out.println("All tasks:");
    for (Task t : tasks) {
      String mark = switch (t.status()) {
        case DONE  -> "[x]";
        case DOING -> "[~]";
        case TODO  -> "[ ]";
      };
      System.out.println("  " + mark + " #" + t.id() + " " + t.title());
    }

    // streams (Unit 9): count and collect without manual loops
    long done = tasks.stream()
        .filter(t -> t.status() == Status.DONE)
        .count();

    List<String> remaining = tasks.stream()
        .filter(t -> t.status() != Status.DONE)
        .map(Task::title)
        .collect(Collectors.toList());

    System.out.println("\nDone: " + done + " / " + tasks.size());
    System.out.println("Still to do: " + remaining);
  }
}
