Unit 1 · The core idea

1foundations

What Spring actually solves

In one line: Spring removes the repetitive "plumbing" of a Java app — creating objects, connecting them, managing config — so you write business logic, not glue code.

A real app is dozens of objects that depend on each other: a controller needs a service, the service needs a repository, the repository needs a database connection, which needs config. Without a framework, you write all the code that builds those objects in the right order and hands each one its dependencies. Spring does that wiring for you. You declare what you need; Spring constructs and connects everything at startup.

Everyday analogy

Cooking versus a restaurant kitchen. Without Spring you buy ingredients, prep stations, and hire staff yourself before every dish. Spring is the kitchen that's already staffed and stocked — you just write the recipe (your logic) and the ingredients show up where you need them.

The same dependency graph — built by hand vs. built by the container.

Which to use — Spring vs plain Java vs other frameworks

ApproachBest whenCost
Plain JavaA tiny script or library with almost no dependenciesYou write all wiring & config yourself
Spring / Spring BootAny real backend, API, or service that grows over timeA learning curve, then huge time savings
Quarkus / MicronautYou need millisecond startup & tiny memory (serverless, native)Smaller ecosystem than Spring
Where it's used

Almost every Java backend built in the last decade: banking systems, e-commerce checkouts, internal APIs, microservices. If a company says "Java backend," it almost always means Spring Boot underneath.

Practice problem

You have three classes where A needs B, and B needs C. In plain Java, how many objects must you manually construct in order, and what breaks if you reorder them?

Show solution
Three (C, then B, then A) — each must exist before the thing that needs it. Reorder and you pass a null (or get a compile error) because the dependency isn't built yet. Spring removes this ordering burden entirely: it figures out the order from the dependency graph.

2foundations

Inversion of Control (IoC)

In one line: Normally your code creates and controls its objects with new; with IoC you hand that control to the framework, which creates them and gives them to you.

"Inversion" is about who is in charge of object creation. In ordinary code, a class reaches out and builds what it needs (new UserRepository()). With IoC, the class never says new for its collaborators — it just declares "I need a UserRepository," and the container builds it and pushes it in. Control of creation has been inverted from your class to Spring.

Everyday analogy

The "don't call us, we'll call you" rule. Instead of you phoning around to assemble a team for each task, a manager (the container) already knows the roster and assigns the right people to you. You focus on the work, not the staffing.

IoC = the framework, not your class, decides when and how objects are made.
Where it's used

IoC is the foundation under everything else in Spring — beans, auto-config, testing, transactions. You rarely "use IoC" directly; you feel its effect every time you write a class without new-ing its collaborators.

⚡ What changes when the code changes
You add a new dependency to a class
→
Nothing to wire by hand — Spring sees the new requirement and supplies it next startup
You swap one implementation for another
→
Callers don't change at all — they asked for the type, not the concrete new

This is the practical payoff of IoC: dependencies become a declaration, not a hard-coded construction, so the blast radius of a change stays tiny.

Practice problem

Which line violates IoC inside a Spring service: (a) private final Repo repo; set via constructor, or (b) private Repo repo = new Repo();?

Show solution
(b) violates IoC — the class controls creation with new, hard-coding the concrete class and making it untestable. (a) lets the container supply the dependency, which is the IoC way.

3foundations

Dependency Injection (DI)

In one line: DI is how IoC is delivered — the container "injects" the objects a class needs through its constructor instead of the class building them.

IoC is the principle (framework controls creation); DI is the technique (it passes dependencies in). When Spring sees a class that takes a UserRepository in its constructor, it finds the matching bean and passes it. Your class declares the contract ("give me something that can store users") and never knows or cares which concrete class arrives.

Everyday analogy

A power socket. Your lamp declares "I need 230V" via its plug; it doesn't generate electricity. The wall (container) injects power. Swap the power source — solar, grid, generator — and the lamp doesn't change.

UserService.java — constructor injection
@Service
public class UserService {
  private final UserRepository repo;   // the contract: an interface

  // Spring sees this constructor and injects the matching bean
  public UserService(UserRepository repo) {
    this.repo = repo;
  }

  public User find(long id) {
    return repo.findById(id).orElseThrow();
  }
}
Which to use — when DI pays off
  • Use DI for any collaborator a class doesn't own: repositories, clients, other services, config.
  • Don't inject plain value objects or things you create per-call (a new BigDecimal, a DTO) — those aren't dependencies, they're data.
Where it's used — testing

The biggest day-to-day win: in a unit test you pass a fake/mock repository straight into the constructor — no Spring, no database. The class can't tell the difference because it only knew the interface.

⚡ What changes when the code changes
Add EmailClient to the constructor
→
Spring injects it automatically; tests now need to pass a mock email client too
Two beans match one type
→
Startup fails with NoUniqueBeanDefinitionException — you must add @Qualifier or @Primary

DI shifts errors to startup time, where they're loud and easy to fix, instead of failing mysteriously at runtime.

Practice problem

Why is private final UserRepository repo; with constructor injection better than a mutable field set by @Autowired?

Show solution
final guarantees the dependency is set once and never null after construction, the object is immutable and thread-safe, and you can build it in a test with plain new — no reflection needed. Field injection allows half-built objects and hides dependencies.

4foundations

Beans & the container

In one line: A "bean" is just any object Spring creates and manages for you; the ApplicationContext is the container that holds them all.

There's nothing special about a bean's class — it's an ordinary Java object. What makes it a bean is that Spring instantiated it, wired its dependencies, and keeps a reference to it in a registry. The ApplicationContext is that registry: at startup it scans your code, builds every bean, and from then on it's the single source of all your wired objects.

Everyday analogy

A company directory. Each employee (bean) was hired, given a desk and tools (dependencies), and listed in the directory (context). When someone needs "the person who handles payroll," they look it up rather than hiring a new one each time.

The context is one big registry of fully-wired, mostly-singleton objects.
Where it's used

You almost never touch the context directly — Spring injects beans for you. You only reach for applicationContext.getBean(...) in rare framework-y code or quick experiments. In normal app code, asking the context by hand is a smell; prefer injection.

Practice problem

You have a @Service with a field count that you increment per request. Two requests come in. Why might the count be shared/wrong?

Show solution
Because the service is a singleton — one instance serves every request. Mutable per-request state in a singleton bean is shared across all callers and across threads. Keep singletons stateless; put per-request data in method parameters or a request-scoped bean.

5foundations

Spring vs Spring Boot

In one line: Spring Framework is the engine (IoC, beans, MVC, JDBC); Spring Boot is the engine plus sensible defaults, auto-config, and a built-in server so you can run in minutes.

Spring Framework gives you the building blocks but expects you to assemble and configure them (historically with lots of XML). Spring Boot sits on top and makes opinionated choices for you: it auto-configures common beans, bundles dependencies into "starters," embeds a web server, and externalizes config to one file. You can always override its choices — it just saves you from configuring the 90% case.

Everyday analogy

Buying car parts vs buying a car. Spring is a high-quality parts catalog; you can build exactly what you want but you do the assembly. Spring Boot is the finished car, pre-tuned, that you can still pop the hood and modify.

Which to use

Spring Framework onlySpring Boot
SetupManual config of every beanAuto-configured from dependencies
Web serverDeploy a WAR to Tomcat yourselfEmbedded server, run a single JAR
DependenciesPick & align versions by handStarters bundle compatible versions
Pick it whenEmbedding Spring in something unusualEssentially every new app
Where it's used

New projects in 2026 start from start.spring.io (Spring Boot). You'd choose "plain Spring" only for legacy systems or niche embedding. For learning and for jobs, Spring Boot is the default.

⚡ What changes when the code changes
Add the spring-boot-starter-web dependency
→
Boot auto-configures an embedded Tomcat, JSON support, and MVC — no config written
Add a database driver + JPA starter
→
Boot auto-configures a DataSource and EntityManager from your properties

In Boot, adding a dependency changes behavior. That's the whole model: presence of a library on the classpath triggers its auto-configuration.

Unit 2 · Wiring beans

6wiring

Stereotype annotations

In one line: @Component and its specialized cousins (@Service, @Repository, @Controller) are how you tell Spring "make this class a bean."

Put one of these on a class and component scanning will pick it up at startup and register it. They're functionally almost identical — all of them register a bean — but the specialized names document the role and occasionally add behavior. @Repository also translates database exceptions; @Controller/@RestController mark a web layer.

Everyday analogy

Name badges at a conference. Everyone with a badge (component) is registered. But "Speaker," "Staff," and "Press" badges signal role and unlock different doors. Same registration, clearer meaning.

Which to use — pick by layer

AnnotationPut it onExtra behavior
@ComponentAny generic bean (utility, helper)None — the plain default
@ServiceBusiness-logic classesNone; pure documentation of intent
@RepositoryData-access classesTranslates DB exceptions to Spring's
@Controller / @RestControllerWeb request handlersEnables request mapping
Three layers, three stereotypes
@RestController          // web layer — handles HTTP
class UserController { /* ... */ }

@Service                 // business layer — rules & orchestration
class UserService { /* ... */ }

@Repository              // data layer — talks to the DB
class UserRepository { /* ... */ }
⚡ What changes when the code changes
You add @Service to a class
→
It becomes injectable everywhere; remove the annotation and any class that injects it fails to start
Move a class outside the scanned package
→
Annotation is ignored — no bean is created (a classic "why is this null?" bug)

An annotation only matters if the class sits under a package Spring scans (next lesson but one).

Practice problem

A teammate annotates a data-access class with @Service instead of @Repository. The app runs fine. What did they lose?

Show solution
It still registers as a bean, so the app works — but they lose the role documentation and, more concretely, @Repository's exception translation that converts vendor-specific SQL exceptions into Spring's consistent DataAccessException hierarchy.

7wiring

Constructor vs field vs setter injection

In one line: There are three ways to receive a dependency — through the constructor, a field, or a setter — and constructor injection is the recommended default.

All three end with Spring supplying the bean; they differ in safety and testability. Constructor injection makes dependencies required, final, and visible in one place. Field injection (@Autowired on a field) is shortest but hides dependencies and can't be final. Setter injection suits genuinely optional dependencies.

Try it — switch the injection style
Everyday analogy

Hiring. Constructor injection is checking required skills at the interview — you can't onboard without them. Field injection is hiring first and quietly hoping the skills appear. Setter injection is "nice-to-have skills you can add later."

⚡ What changes when the code changes
Add a 4th constructor parameter
→
Every test that builds the class by hand must pass the new arg — the compiler forces you to notice
Two beans depend on each other's constructors
→
Startup fails with a circular-dependency error — a design smell constructor injection surfaces early

Field injection hides both problems until runtime. Constructor injection turns them into compile/startup errors — which is exactly where you want them.

Which to use
  • Constructor — your default for required dependencies (almost always).
  • Setter — for truly optional dependencies you can live without.
  • Field — avoid in app code; tolerable only in quick tests or @Configuration where the tradeoffs don't bite.
Practice problem

Why does constructor injection make a class easier to unit-test than field injection?

Show solution
You can instantiate it with plain new MyService(mockRepo, mockClient) — no Spring, no reflection. Field injection has no constructor to pass mocks into, so you'd need a Spring context or reflection hacks to set the private fields.

8wiring

@Configuration & @Bean

In one line: When you can't annotate a class (it's from a library) or need custom construction, you write a @Bean method inside a @Configuration class to build it yourself.

Stereotype annotations work only on your classes. For a third-party class — an HTTP client, a JSON mapper, a thread pool — you can't add @Component. Instead you write a method that returns the object, mark it @Bean, and put it in a @Configuration class. Whatever the method returns becomes a managed bean, just as if Spring had created it.

AppConfig.java
@Configuration
public class AppConfig {

  @Bean
  public RestClient restClient() {        // a library class you can't annotate
    return RestClient.builder()
        .baseUrl("https://api.example.com")
        .build();
  }

  // method params are injected too — Spring passes the restClient bean above
  @Bean
  public PaymentGateway gateway(RestClient client) {
    return new PaymentGateway(client);
  }
}
Everyday analogy

A recipe card pinned in the kitchen. You can't change how a store-bought sauce is made, but your recipe card says exactly how to prepare and serve it. @Bean is that card for objects you didn't write.

Which to use — @Component vs @Bean

@Component (& friends)@Bean method
Works onClasses you own & can annotateAny class, incl. third-party
ConstructionSpring calls the constructorYou write the construction logic
Best forYour services, repos, controllersLibrary objects, custom setup, conditional beans
⚡ What changes when the code changes
Change the baseUrl in the @Bean method
→
Every bean that injects this RestClient uses the new URL — one edit, app-wide
Define a second RestClient @Bean
→
Ambiguity — injection now needs @Qualifier to say which one
Practice problem

You need a bean of a class from a Maven library. You can't edit that class. Which mechanism do you reach for?

Show solution
A @Bean method in a @Configuration class — it's the only option, since you can't add @Component to source you don't control.

9wiring

Component scanning — where beans come from

In one line: At startup Spring scans the package of your main class (and its sub-packages) for stereotype annotations and registers what it finds.

Annotations don't register themselves — something has to find them. That's component scanning. By default @SpringBootApplication scans its own package downward. Anything in a sibling or parent package is invisible unless you widen the scan. This is the single most common reason a correctly-annotated bean ends up null.

Beans only exist if they live under a scanned package.
⚡ What changes when the code changes
Move a @Service to a package outside the main one
→
It silently stops being a bean; injection fails with "no qualifying bean"
Add @ComponentScan("com.other")
→
That package is now scanned too, and its components register

When a bean is mysteriously missing, check its package before anything else.

Practice problem

Your @Service is annotated correctly but injection throws "no qualifying bean of type." The class compiles. What's the most likely cause?

Show solution
It lives outside the scanned package tree (a sibling/parent of the main class's package). Move it under the main package or add an explicit @ComponentScan for its package.

10wiring

Bean scopes

In one line: Scope controls how many instances exist — by default one shared singleton, but you can ask for a fresh one each time (prototype) or per web request.

The default singleton means the container creates one instance and shares it everywhere — efficient, but it must be stateless and thread-safe. prototype gives a brand-new object on each injection/lookup, good for stateful helpers. Web scopes (request, session) tie a bean's life to an HTTP request or user session.

Everyday analogy

Office equipment. The printer is a singleton — one shared by all. Scratch paper is prototype — grab a fresh sheet each time. A visitor badge is request-scoped — issued for one visit, then discarded.

Which to use

ScopeInstancesUse for
singleton (default)One, app-wideStateless services, repos, config — 99% of beans
prototypeNew every timeStateful, short-lived helpers
requestOne per HTTP requestPer-request data in web apps
sessionOne per user sessionPer-user state across requests
⚡ What changes when the code changes
Add a mutable counter field to a singleton service
→
It's now shared across all threads — race conditions and wrong values under load
Switch a bean to @Scope("prototype")
→
Each injection gets a fresh instance; state no longer leaks between callers

Most "weird shared state" bugs are a stateful field on a singleton. The fix is usually "make it stateless," not "change the scope."

Practice problem

True or false: making a service prototype is the right fix for thread-safety in a web app.

Show solution
Usually false. Singletons are injected once into other singletons, so a prototype injected into a singleton is still only created once. The real fix is to remove mutable shared state and keep the singleton stateless.

11wiring

Bean lifecycle

In one line: Each bean goes through create → inject dependencies → init callback → live → destroy callback, and you can hook the init/destroy steps.

After Spring constructs a bean and injects its dependencies, it calls any @PostConstruct method — the right place to warm a cache, validate config, or open a resource once everything is wired. On shutdown it calls @PreDestroy for cleanup. Doing this work in the constructor is risky because dependencies may not be set yet.

Hook init after wiring, cleanup before shutdown.
⚡ What changes when the code changes
Move resource setup from constructor to @PostConstruct
→
It runs after injection, so injected config is available — fixes "null dependency in constructor"
Open a connection but forget @PreDestroy
→
Resource leaks on shutdown/restart
Practice problem

You need to validate that an injected API key is non-blank when the app starts. Constructor or @PostConstruct?

Show solution
@PostConstruct — it runs after dependency injection, so the key has been set. In the constructor with field injection the value might still be null.

Unit 3 · Spring Boot

12spring boot

@SpringBootApplication

In one line: One annotation on your main class turns on everything: component scanning, auto-configuration, and config support.

@SpringBootApplication is a shortcut that bundles three annotations: @ComponentScan (find your beans), @EnableAutoConfiguration (configure libraries on the classpath), and @Configuration (let this class define beans). It marks the entry point and the root of scanning.

Application.java — the whole bootstrap
@SpringBootApplication   // = @ComponentScan + @EnableAutoConfiguration + @Configuration
public class ShopApplication {
  public static void main(String[] args) {
    SpringApplication.run(ShopApplication.class, args);
  }
}
Everyday analogy

A "morning routine" macro on your phone: one tap turns on the lights, starts the coffee, and reads the news. One annotation, three jobs.

⚡ What changes when the code changes
Place this class in a deeper package
→
Scanning starts lower, so beans in higher packages disappear
Exclude an auto-config: @SpringBootApplication(exclude=...)
→
That library's default beans are no longer created — you take over its config
Practice problem

Name the three annotations @SpringBootApplication stands in for.

Show solution
@ComponentScan, @EnableAutoConfiguration, and @Configuration.

13spring boot

Auto-configuration — how the magic works

In one line: Boot looks at what's on your classpath and conditionally creates the beans those libraries need — unless you've already defined them yourself.

Auto-config isn't magic, it's a giant set of "if present, then configure" rules. Each rule is guarded by conditions like @ConditionalOnClass ("only if this library is on the classpath") and @ConditionalOnMissingBean ("only if you haven't already defined this bean"). So adding a dependency turns features on, and defining your own bean turns Boot's default off.

Auto-config = conditions on the classpath + "don't override what you wrote".
⚡ What changes when the code changes
Add a Redis starter
→
A Redis connection + template bean appear automatically
Define your own ObjectMapper @Bean
→
Boot stops creating its default JSON mapper — yours is used everywhere

"Add a library to get a feature; define a bean to override the default" is the mental model for all of Boot.

Practice problem

You define your own DataSource bean, yet you worried Boot's default would conflict. Why doesn't it?

Show solution
@ConditionalOnMissingBean — Boot's auto-configured DataSource only fires if none exists. Since you defined one, Boot backs off.

In one line: A starter is one dependency that pulls in a whole curated, version-aligned set of libraries for a capability.

Instead of hand-picking Tomcat, Jackson, validation, and MVC — and praying their versions agree — you add spring-boot-starter-web and get all of them, tested together. Boot's dependency management pins compatible versions, so you usually don't write version numbers at all.

A meal kit. Rather than shopping for 12 ingredients in the right quantities, you order "taco night" and everything arrives measured and compatible.

Why do you rarely specify version numbers for Spring libraries in a Boot project?

15spring boot

Embedded server & main()

In one line: Boot puts the web server inside your app, so you run a single executable JAR with java -jar instead of deploying to an external server.

The classic Java model was: build a WAR, install Tomcat separately, deploy into it. Boot flips this — the server (Tomcat by default) is a library bundled in your JAR and started by main(). The result is a self-contained app: same artifact runs on your laptop, CI, and production, no server to install.

Everyday analogy

A food truck vs a restaurant. The old way needs a building (server) to move into. The food truck (embedded server) brings its own kitchen — park it anywhere and serve.

Which to use — embedded server choice

ServerPick when
Tomcat (default)Standard blocking web apps — the safe default
JettyMany idle long-lived connections, lighter footprint
Netty (WebFlux)Reactive, non-blocking, high-concurrency streams

Note (mid-2026): Undertow support was removed in Spring Boot 4. With Java 21+ and virtual threads, plain Tomcat handles very high concurrency without going reactive.

⚡ What changes when the code changes
Set server.port=8081 in properties
→
App listens on 8081 next run — no server reconfig, it's all in your app
Swap to the WebFlux starter
→
Boot uses Netty and a non-blocking model — your controllers must return reactive types
Practice problem

Why can the same Boot JAR run identically on your laptop and in production with no server install?

Show solution
The web server is embedded inside the JAR. java -jar app.jar starts the app and its server together, so there's nothing external to install or match versions with.

16spring boot

Configuration & properties

In one line: Keep settings out of code in application.yml, then pull them in with @Value for one-offs or @ConfigurationProperties for grouped, type-safe config.

Hard-coded URLs, timeouts, and keys make an app un-shippable. Boot reads application.properties/.yml (and environment variables, command-line args, etc.) into the environment. @Value("${...}") injects a single value; @ConfigurationProperties binds a whole prefix to a typed object — cleaner and validated.

application.yml + typed binding
# application.yml
shop:
  currency: USD
  max-items: 50

// typed, grouped config — preferred for related settings
@ConfigurationProperties(prefix = "shop")
public record ShopProps(String currency, int maxItems) {}

// one-off value
@Value("${shop.currency}") String currency;

Which to use

@Value@ConfigurationProperties
Best forOne or two stray valuesA group of related settings
Type safetyString-ish, manualBound to typed fields, validatable
Refactor-friendlyNo — magic stringsYes — real object
⚡ What changes when the code changes
Set SHOP_CURRENCY=EUR as an env var
→
It overrides the yml value with no code change — the deploy controls config
Rename a property but not its @Value string
→
Silent failure / startup error — magic strings don't refactor. Typed props would catch it
Practice problem

You have eight related mail settings. @Value eight times or one @ConfigurationProperties?

Show solution
One @ConfigurationProperties(prefix="mail") — grouped, type-safe, refactor-friendly, and supports validation. Eight @Value strings are brittle and scattered.

17spring boot

Profiles

In one line: Profiles let one codebase behave differently per environment — dev uses an in-memory DB, prod uses the real one — by activating a named set of config and beans.

You name environments (dev, test, prod) and put environment-specific config in application-dev.yml, application-prod.yml, etc. Activating a profile (via spring.profiles.active or an env var) layers that file on top of the base and switches on any @Profile-annotated beans.

Everyday analogy

Outfits for occasions. Same you, different clothes for the gym, the office, a wedding. Profiles are "outfits" your app puts on depending on where it's running.

Profile-specific beans & config
@Bean @Profile("dev")
DataSource h2() { /* in-memory DB for local dev */ }

@Bean @Profile("prod")
DataSource postgres() { /* real DB for production */ }

# activate at runtime — no rebuild
# SPRING_PROFILES_ACTIVE=prod java -jar app.jar
⚡ What changes when the code changes
Launch with --spring.profiles.active=prod
→
application-prod.yml loads and prod-only beans activate; dev beans don't exist
Forget to set any profile
→
Only the base config runs; profile-only beans are missing — a common deploy mistake
Practice problem

How do you point your app at a production database without changing or recompiling code?

Show solution
Activate the prod profile (e.g. SPRING_PROFILES_ACTIVE=prod) so application-prod.yml and prod beans load. The same artifact behaves differently per environment.

Unit 4 · Web & REST

18web

@RestController & mapping

In one line: @RestController turns a class into an HTTP endpoint handler; mapping annotations connect URLs + verbs to methods, and return values become JSON automatically.

@RestController = @Controller + @ResponseBody: every method's return value is serialized to the response body (JSON by default) instead of being treated as a view name. @GetMapping, @PostMapping, etc. bind a URL path and HTTP method to a handler method.

UserController.java
@RestController
@RequestMapping("/users")        // shared prefix for all methods
class UserController {
  private final UserService svc;
  UserController(UserService svc) { this.svc = svc; }

  @GetMapping("/{id}")       // GET /users/42
  User get(@PathVariable long id) {
    return svc.find(id);     // auto-serialized to JSON
  }
}
Everyday analogy

A receptionist with a directory. A visitor (request) says "GET /users/42"; the receptionist routes them to the right office (method) and hands back a printed result (JSON).

Which to use — @Controller vs @RestController
  • @RestController — JSON/data APIs (almost everything today).
  • @Controller — server-rendered HTML pages (Thymeleaf views); methods return a template name, not data.
⚡ What changes when the code changes
Add a field to the returned User

Two sides of one link
@Entity class Order {

@Entity class User {
  @OneToMany(mappedBy = "user")       // mirror side: no extra column
  private List<Order> orders;

@Transactional

In one line: Put @Transactional on a service method and Spring wraps it in begin → do the work → commit, rolling back automatically if it throws.

A transaction makes a group of database operations all-or-nothing. Spring uses an AOP proxy: when you call the method, the proxy opens a transaction, runs your code, then commits — or rolls back if a runtime exception escapes. Because it's proxy-based, it only applies to public methods called from outside the bean (not self-invocation).

The proxy commits on clean return and rolls back on a runtime exception.
Atomic money transfer
@Service
class BankService {

  @Transactional                      // begin .. commit/rollback around this
  void transfer(Long from, Long to, BigDecimal amt) {
    accounts.debit(from, amt);
    accounts.credit(to, amt);        // if this throws, the debit is rolled back too
  }
}

Which to use

@Transactional@Transactional(readOnly = true)
Allows writesYesNo (read path)
Use forAny method that modifies dataPure reads — lets the DB optimize

Call a @Transactional method from another method in the same class
→
The proxy is bypassed — no transaction starts. Move it to another bean
Throw a checked exception
→
By default it does not roll back; add rollbackFor = ... if you need it to

Default rollback triggers on unchecked (runtime) exceptions only. This surprises people constantly.

The data flow

In one line: A request flows down through Controller → Service → Repository → Database and the result flows back up — each layer with one job.

This is the backbone of almost every Spring Boot app. The controller handles HTTP and validation, the service holds business logic and the transaction boundary, the repository talks to the database, and the entity is the data carried between them. Aspects & advice

In one line: An aspect bundles a pointcut (where) with advice (what to run) — @Before, @After, or @Around the matched method.

Mark a bean @Aspect, write advice methods, and give each a pointcut expression that selects the join points (method executions) to wrap. The proxy intercepts those calls and runs your advice. @Around is the most powerful — it can run code before and after, and decides whether to proceed.

The proxy runs advice around the real method call.
A timing aspect
@Aspect
@Component
class TimingAspect {

  @Around("execution(* com.shop.svc..*(..))")   // all service methods
  Object time(ProceedingJoinPoint pjp) throws Throwable {
    long t = System.nanoTime();
    try { return pjp.proceed(); }            // run the real method
    finally { log.info("took {}ns", System.nanoTime() - t); }
  }
}

Which to use

AdviceRunsUse for
@BeforeBefore the methodValidation, logging entry
@AfterReturning / @AfterThrowingOn success / on errorResult logging, error metrics
@AroundBefore + after, controls proceedTiming, caching, retries
⚡ What changes when the code changes
Add a new service class under the matched package
→
The pointcut auto-matches it — no aspect change needed
Narrow the pointcut expression
→
Advice instantly stops applying to the now-excluded methods

Like @Transactional, aspects only work through the proxy — self-invoked calls within the same bean aren't advised.

@Around — only it can call proceed() in a loop and decide whether to retry based on the outcome. The others can't re-invoke the method.

Spring Security is, mechanically, a chain of servlet filters. A request walks through them in order: figure out who's authenticated, check CSRF/CORS, then enforce authorization. Actuator & observability

In one line: Add the Actuator starter and your app exposes ready-made endpoints for health, metrics and info — the eyes and ears of a production service.

You can't operate what you can't see. Spring Boot Actuator adds endpoints like /actuator/health (is it alive?), /actuator/metrics and Prometheus output (how is it performing?). Metrics flow through Micrometer; Spring Boot 4 makes OpenTelemetry-based tracing first-class so you can follow a request across services. Expose only what you need, and protect these endpoints.

Expose selected endpoints
# application.properties
management.endpoints.web.exposure.include=health,info,metrics,prometheus
management.endpoint.health.show-details=when-authorized

# health: /actuator/health  → { "status": "UP" }
🌿 Where it's used

Kubernetes hits /actuator/health/liveness and /readiness to decide whether to restart or route traffic to a pod. Dashboards scrape /actuator/prometheus for graphs and alerts.

⚡ What changes when the code changes
Add a dependency like a datasource
→
Its health contributor auto-appears in /health — DB down now shows the app as DOWN
Expose all endpoints with include=*
→
Sensitive operational data becomes reachable — secure them behind auth first

Never expose Actuator wide-open on the public internet. Limit the set and require authentication.

Practice problem

Your container keeps getting killed and restarted even though the app "works." What Actuator feature should the orchestrator be checking, and how can the app fail it incorrectly?

Show solution
Liveness/readiness probes via /actuator/health. A failing dependency health contributor (e.g. a flaky DB) can mark the app DOWN, causing restarts — tune which contributors affect liveness vs readiness.

Caching, async & scheduling

In one line: Three annotation-driven powers — @Cacheable to skip repeat work, @Async to run off the request thread, @Scheduled to run jobs on a timer.

All three are AOP-based, so you turn behavior on with an annotation. @Cacheable stores a method's result keyed by its arguments and returns the cached value next time. @Async runs a method on a separate thread so the caller doesn't wait. @Scheduled runs a method on a cron or fixed-rate schedule. Each needs its feature enabled (@EnableCaching, @EnableAsync, @EnableScheduling).

Three powers, three annotations
@Cacheable("users")                  // 2nd call with same id skips the DB
User find(Long id) { ... }

@Async                              // returns immediately; runs on another thread
void sendWelcomeEmail(User u) { ... }

@Scheduled(cron = "0 0 2 * * *")     // every day at 02:00
void nightlyCleanup() { ... }

Which to use

AnnotationSolvesWatch out for
@CacheableExpensive/repeated readsStale data — plan eviction (@CacheEvict)
@AsyncSlow side-work (email, calls)Exceptions vanish unless handled; needs its own thread pool
@ScheduledRecurring background jobsRuns on every instance unless you add locking
⚡ What changes when the code changes
Add @Cacheable to a method
→
Updates elsewhere can serve stale results until you add @CacheEvict on the writes
Add @Async
→
The return is no longer immediate-value; use void or CompletableFuture, and self-invocation won't work (proxy)
Deploy @Scheduled to 3 instances
→
The job fires 3 times — add a distributed lock (e.g. ShedLock)

Like @Transactional, these only work through the proxy — calling them from within the same bean skips the behavior.

Practice problem

You add @Cacheable to findUser, but after an admin edits a user the old data keeps showing. What's missing?

Show solution
Cache eviction. Annotate the update method with @CacheEvict(value="users", key="#user.id") (or @CachePut) so the stale entry is removed/refreshed when data changes.

Package, deploy & scale

In one line: Build one self-contained jar (server included), wrap it in a container image, and run many copies behind a load balancer.

Spring Boot packages your app as an executable "fat jar" — every dependency and the embedded server inside one file you run with java -jar. For containers, you don't even need a Dockerfile: ./mvnw spring-boot:build-image uses Cloud Native Buildpacks to produce an optimized image. Because each instance is stateless (state lives in the DB / cache), you scale horizontally — just run more identical copies. For fast startup and low memory, Boot 4's AOT/GraalVM native images compile ahead of time.

Horizontal scaling: identical stateless instances, one shared source of truth.
⚡ What changes when the code changes
Store session/state in app memory
→
Horizontal scaling breaks (each instance has different state) — move state to DB/Redis
Switch to a GraalVM native image
→
Startup drops to milliseconds and memory shrinks, but reflection-heavy libs may need AOT hints
Change a setting per environment
→
Use profiles + externalized config/env vars, not code changes — the same jar runs everywhere

"Build once, configure per environment" — the same artifact runs in dev, staging and prod via config.

Practice problem

You scale from 1 to 4 instances and users randomly get logged out. What design choice is causing it?

Show solution
In-memory HTTP sessions — each instance has its own, so requests routed elsewhere lose the session. Fix by going stateless (JWT) or storing sessions in a shared store (e.g. Spring Session + Redis).

A complete minimal app (Controller → Service → Repository, the flow from Lesson 30)

This is the whole layered stack in one file's worth of code: an entity, a repository you don't implement, a service with the transaction boundary, and a REST controller. Generate a project on Initializr with Web + JPA + H2, drop this in, and run ./mvnw spring-boot:run.

java · DemoApplication.java
package com.demo;

import jakarta.persistence.*;
import org.springframework.boot.*;
import org.springframework.boot.autoconfigure.*;
import org.springframework.data.jpa.repository.*;
import org.springframework.stereotype.*;
import org.springframework.transaction.annotation.*;
import org.springframework.web.bind.annotation.*;
import java.util.*;

@SpringBootApplication                         // the 3-in-1 from Lesson 12
public class DemoApplication {
  public static void main(String[] args) {
    SpringApplication.run(DemoApplication.class, args);  // embedded server starts
  }
}

@Entity                                          // Lesson 24: one object = one row
class Todo {
  @Id @GeneratedValue(strategy = GenerationType.IDENTITY) Long id;
  String title;
  boolean done;
}

interface TodoRepo extends JpaRepository<Todo, Long> {  // Lesson 25: free CRUD
  List<Todo> findByDoneFalse();                  // Lesson 26: derived query
}

@Service                                          // Lesson 30: business + tx boundary
class TodoService {
  private final TodoRepo repo;
  TodoService(TodoRepo repo) { this.repo = repo; }  // constructor injection, Lesson 7

  @Transactional(readOnly = true)
  List<Todo> open() { return repo.findByDoneFalse(); }

  @Transactional
  Todo add(Todo t) { return repo.save(t); }    // Lesson 29: atomic
}

@RestController                                   // Lesson 18: HTTP layer
@RequestMapping("/todos")
class TodoController {
  private final TodoService service;
  TodoController(TodoService service) { this.service = service; }

  @GetMapping
  List<Todo> open() { return service.open(); }    // GET /todos

  @PostMapping
  Todo add(@RequestBody Todo t) { return service.add(t); }  // POST /todos
}
🌿 What to notice

Every layer from this course is here: @SpringBootApplication (Unit 3), constructor injection (Unit 2), the REST controller (Unit 4), the entity + repository + derived query (Unit 5), and the @Transactional service boundary (Lesson 29). It's the data flow from Lesson 30 made runnable.
