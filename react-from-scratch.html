Unit 1 · Foundations

1foundation

Why React exists

In one line: You describe what the UI should look like for the current data, and React figures out how to update the screen — instead of you editing the DOM by hand.

In plain JavaScript you change the page imperatively: find an element, change its text, toggle a class, add a node. As an app grows, keeping every piece of the screen in sync with your data becomes a tangle of manual updates. React is declarative: you write a function that returns the UI for some data, and whenever the data changes, React re-runs that function and updates only what actually changed.

Everyday analogy

Imperative is giving turn-by-turn driving directions ("turn left, now right, now stop"). Declarative is telling a GPS the destination ("take me home") and letting it work out the route. With React you state the destination — the UI for this data — and it drives the DOM there.

Comparison

Vanilla JS (imperative)React (declarative)
You writeSteps to mutate the DOMWhat the UI looks like for given data
Keeping UI in syncYour job, manuallyReact's job, automatically
As app growsUpdate logic multipliesAdd components, data flows
Reusing UICopy/paste, fragileReusable components
Example

A "like" button showing a count. Vanilla: on click, read the number, add one, write it back to the element. React: store the count in state, write <button>{count} likes</button>, and update the count — React re-draws the button for you.

Practice problem

You have a list of 50 todos and toggle one as "done". In plain JS, what's the risk versus React?

Show solution
In plain JS you must find that exact row and update its checkbox, class, and maybe a "3 of 50 done" counter — three manual updates that can drift apart. In React you flip the item's done in state and re-render; the row, its style, and the counter all reflect the new data because they're all derived from it.

2foundation

Components

In one line: A component is a JavaScript function that returns some UI — and you build apps by nesting components inside components.

Components are the unit of reuse and the unit of thought in React. A component is just a function whose name starts with a capital letter and that returns markup (JSX). You use it like a custom HTML tag: <Profile />. Components nest to form a tree — one root <App /> contains the whole UI.

Everyday analogy

LEGO bricks. A small brick (Button) clicks into a bigger brick (Toolbar), which clicks into a wall (Page). The same brick can appear many times. You design the brick once, then snap copies together.

jsx · a component is a function that returns UI
// Capitalized name → React treats it as a component
function Welcome() {
  return <h1>Hello, React</h1>;
}

// Use it like a tag — and reuse it freely
function App() {
  return (
    <div>
      <Welcome />
      <Welcome />
    </div>
  );
}
Practice problem

Why must a component name be capitalized (Welcome, not welcome)?

Show solution
JSX uses capitalization to tell components from HTML tags. Lowercase <welcome /> is treated as an unknown DOM tag; capitalized <Welcome /> is read as your component.

3foundation

JSX

In one line: JSX lets you write HTML-like markup inside JavaScript; a build step turns it into normal function calls.

JSX is not HTML and not a string — it's syntax sugar. <h1>Hi</h1> compiles to a function call that creates a React element (a plain object describing the UI). Inside JSX, anything in { } is live JavaScript: variables, expressions, function calls.

Everyday analogy

Like a fill-in-the-blank form letter. The fixed text is the markup; the {blanks} are where you drop in live values — a name, a total, today's date.

Practice problem

This errors: return <h1>A</h1><p>B</p>;. Why, and how do you fix it?

Show solution
A component must return a single root element. Wrap both in a fragment: return <><h1>A</h1><p>B</p></>; (or a single <div>).

4core mechanism

Render & the Virtual DOM

In one line: When your data changes, React re-runs your components to build a fresh description of the UI, compares it to the previous one, and updates only the parts of the real screen that actually differ.

This is the heart of "how the UI changes when your code/state changes." A render is just React calling your component function. The output is a lightweight tree of objects (the virtual DOM). React keeps the previous tree, diffs the new one against it (reconciliation), and commits only the minimal set of real DOM changes. You never touch the DOM yourself.

Everyday analogy

An editor with "track changes." You hand in a fully rewritten draft; the editor compares it to the old version and only marks the sentences that actually changed — they don't reprint the whole book. React reprints only what differs.

Live: state change → re-render → screen updates

Each click sets state, which makes React re-run the component (watch the border flash = one render) and patch the number on screen.

Practice problem

People often say "the virtual DOM is fast." What's the more precise truth?

Show solution
Building the virtual tree isn't free. The win is that React commits the minimal real-DOM changes after diffing — and real DOM mutations are the expensive part. It also frees you from writing manual, error-prone updates. "Fast enough, and correct by default" is the honest claim.

5mental model

Elements vs components

In one line: A component is a function (a blueprint); an element is a plain object it produces (a description of what to show); React turns elements into real DOM.

Three things get conflated. The component is your function Button. An element is what <Button /> evaluates to — a cheap object like {type: Button, props: {...}}. Elements are descriptions, not the actual screen; React reads them and creates/updates the real DOM. Understanding this explains why rendering is cheap to describe and why React can diff so freely.

Everyday analogy

A recipe (component) vs a written-down order ticket for one dish (element) vs the plated food (DOM). The kitchen can re-issue tickets cheaply; only plating costs real work.

TermWhat it isExample
ComponentA function that returns elementsfunction Button() {…}
ElementA plain object describing UI<Button /> → an object
DOM nodeThe real thing on screenan actual <button>
Example

<Button /> is sugar for React.createElement(Button), which returns an object. Returning that object doesn't run anything visible yet — React decides when to call the component and when to commit DOM.

Practice problem

Is calling Button() directly the same as writing <Button />?

Show solution
No. <Button /> creates an element React manages (it owns state, lifecycle, and re-renders). Calling Button() just inlines its output as if it were part of the parent, losing its identity as a separate component. Use JSX, not direct calls.

Unit 2 · Props & data flow

6data flow

Props

In one line: Props are the inputs a parent passes down to a child component — read-only arguments that customize it.

A component receives one object, props, holding everything the parent passed. You read props but never change them — they belong to the parent. Different props produce different output, which is exactly why the same component can render many ways.

Everyday analogy

Function arguments. makeCoffee("oat", "large") — the recipe is fixed, the inputs vary the result. Props are arguments for UI.

jsx · pass props down, read them in the child
// Child destructures the props it needs (with a default)
function Avatar({ name, size = 48 }) {
  return <img width={size} alt={name} />;
}

// Parent passes props like HTML attributes
function App() {
  return <Avatar name="Sam" size={64} />;
}
Practice problem

A child does props.count = props.count + 1. Why is this wrong?

Show solution
Props are read-only. To change a value, the data must live as state in some component (often the parent), and the parent updates it — the child can't mutate its props. Changing props directly does nothing useful and breaks React's model.

7pattern

Composition & children

In one line: The special children prop lets a component wrap whatever you put between its tags — so you build flexible layouts by nesting, not by inheritance.

Whatever you write between a component's opening and closing tags arrives as props.children. This makes generic "wrapper" components — cards, modals, layouts — that don't care what's inside. React favors composition (nesting components) over class-style inheritance.

Everyday analogy

A picture frame. The frame (component) is reusable; the photo you slide in (children) changes. Same frame, any photo.

jsx · a reusable Card that accepts any children
function Card({ children }) {
  return <div className="card">{children}</div>;
}

function App() {
  return (
    <Card>
      <h2>Title</h2>
      <p>Anything can go inside.</p>
    </Card>
  );
}
Example

One Modal with a children slot can host a login form, a confirmation, or an image gallery — without the modal knowing anything about its contents.

Practice problem

You need a layout with a fixed sidebar and a swappable main area. Composition or copy-paste per page?

Show solution
Composition: build <Layout>{pageContent}</Layout> where Layout renders the sidebar plus {children}. Each page passes its own main content as children — no duplication of the shell.

8core mechanism

One-way data flow

In one line: Data flows down via props; changes flow up via callback functions — children ask the parent to update, the parent owns the data.

This is the rule that keeps React predictable, and it directly answers "where does data live and how does it change?" State lives in a parent (the source of truth). The parent passes the value down as a prop and passes a function down too. When the child wants a change, it calls that function; the parent updates its state; new props flow back down.

Everyday analogy

A vending machine. The display (child) shows what the machine (parent) decides. You press a button (event up); the machine updates its state and changes the display (data down). The display never edits itself.

Practice problem

Two sibling components must show the same counter. Where does the count live?

Show solution
In their closest common parent. The parent holds the count in state and passes it (and an update function) to both siblings. This is "lifting state up" — covered in Lesson 15.

9core mechanism

Lists & keys

In one line: Render a list by mapping data to elements, and give each a stable key so React can track which item is which across renders.

You turn an array into UI with .map(). The key prop is how React matches items between the old and new render — so when the list reorders, inserts, or deletes, React moves the right DOM nodes instead of rebuilding everything. Keys must be stable and unique among siblings (usually an id from your data, not the array index).

Everyday analogy

Name tags at a conference. If everyone wears a permanent name, the organizer can reseat people correctly. If tags say "Seat 1, Seat 2" (the index) and you reorder the row, the tags now lie — and React gets confused the same way.

jsx · map data to elements with stable keys
function TodoList({ todos }) {
  return (
    <ul>
      {todos.map((todo) => (
        <li key={todo.id}>{todo.text}</li>
      ))}
    </ul>
  );
}
Practice problem

You used key={index}. After inserting an item at the top, an input's typed text "jumps" to the wrong row. Why?

Show solution
Index keys shift when the list changes order: the item that was index 0 is now index 1, so React reuses the old DOM (and its input state) for the wrong item. Use a stable id as the key so identity follows the data, not the position.

Unit 3 · State & reactivity

10core mechanism

useState

In one line: useState gives a component a piece of memory that survives re-renders — and changing it tells React to re-render.

A normal variable resets every render and changing it doesn't redraw anything. State is different: React remembers it across renders, and calling the setter both stores the new value and schedules a re-render. const [value, setValue] = useState(initial) returns the current value and the function to update it.

Everyday analogy

A scoreboard with a remote. The number stays put between plays (memory). Pressing the remote (the setter) both changes the number and lights up the board (re-render). A scribble on a sticky note (a plain variable) neither persists nor updates the board.

jsx · the canonical counter
import { useState } from "react";

function Counter() {
  const [count, setCount] = useState(0);
  return (
    <button onClick={() => setCount(count + 1)}>
      Clicked {count} times
    </button>
  );
}
Live: a plain variable vs state

The left value is real React-style state (updates the screen). The right pretends to be a plain variable — it changes in memory but the screen never refreshes, so you never see it.

Practice problem

Why won't let count = 0; count++ inside a component update the screen?

Show solution
Two reasons: a local variable is recreated as 0 on every render (no memory), and changing it doesn't tell React to re-render. State solves both — it persists and it triggers a re-render.

11mental model

State is a snapshot

In one line: Within one render, state holds a fixed value; calling the setter doesn't change that value mid-render — it schedules the next render with the new value.

A surprising-but-crucial idea. The count you read during a render is a snapshot, frozen for that render. So calling setCount(count + 1) three times in a row from the same snapshot all compute from the same old value — you get +1, not +3. To build on the latest value, pass an updater function: setCount(c => c + 1).

Everyday analogy

A photo taken at the start of the render. Everyone in that render looks at the same photo. Saying "add one to the photo's number" three times still reads the same photo each time. Asking "add one to whatever it becomes" (the updater) chains correctly.

Live: snapshot (+1) vs updater (+3)

Practice problem

You call setOpen(true) then immediately console.log(open). It prints false. Bug?

Show solution
No — expected. open is the snapshot for the current render; the new value only appears on the next render. Read the new value in the next render (or in an effect), not on the line after the setter.

12core mechanism

Updating state immutably

In one line: Never mutate state in place — create a new object/array with the change, so React sees a new reference and knows to re-render.

React decides whether to re-render partly by comparing references. If you push into the same array or set a property on the same object, the reference is unchanged and React may skip the update — or behave unpredictably. Always produce a new value: spread the old one and override what changed.

Everyday analogy

Editing a shared document by sending a fresh copy with edits, rather than scribbling on the original behind everyone's back. The fresh copy is obviously "new"; the secret scribble might go unnoticed.

jsx · copy-then-change for objects and arrays
// ❌ mutates the same object — React may not notice
user.name = "Sam";
setUser(user);

// ✅ new object with the change
setUser({ ...user, name: "Sam" });

// arrays: build new ones, don't push/splice in place
setItems([...items, newItem]);              // add
setItems(items.filter(i => i.id !== id));    // remove
setItems(items.map(i =>
  i.id === id ? { ...i, done: true } : i));    // update one

Practice problem

You todos.push(newTodo); setTodos(todos) and the list doesn't update. Why?

Show solution
You handed React the same array reference. It compares old vs new, sees the same array, and bails. Use setTodos([...todos, newTodo]) so the reference changes.

13mechanism

Batching & updater functions

In one line: React groups multiple state updates in the same event into a single re-render (batching); use updater functions when each update depends on the previous one.

If you call several setters in one click handler, React doesn't re-render after each — it batches them and re-renders once with all the changes. That's faster and avoids flickering. The catch from Lesson 11 returns: if updates build on each other, use the updater form so each sees the latest queued value.

Everyday analogy

A waiter taking your whole table's order before going to the kitchen, instead of running back per dish. One trip (one render) with everything.

jsx · three sets, one render
function handleClick() {
  setName("Sam");
  setAge(30);
  setActive(true);
  // React batches these → ONE re-render, not three
}
Example

Since React 18, batching also applies inside promises, timeouts, and native handlers ("automatic batching"), not just React events — so the one-render behavior is consistent everywhere.

Practice problem

In one handler you call setCount(count+1) twice expecting +2 but get +1. Fix?

Show solution
Both reads use the same snapshot of count. Use updater functions so they chain: setCount(c => c + 1); setCount(c => c + 1); → +2 in one batched render.

14which to use

Derived vs stored state

In one line: If a value can be calculated from existing state or props, calculate it during render — don't store it as separate state.

A common beginner mistake is storing things that are really just computations of other state — a filtered list, a total, a "isValid" flag. Stored copies drift out of sync. The rule: keep the minimal source of truth in state, and derive everything else inline while rendering.

Everyday analogy

A receipt's total. You don't store the total and the line items separately and hope they match — you add up the items each time. The total is derived, not stored.

const visible = items.filter(i => i.text.includes(query));

Practice problem

You keep fullName in state alongside first and last, updating it in three places. What goes wrong?

Show solution
They drift apart — miss one update site and fullName is stale. Drop the state; derive it: const fullName = first + " " + last; during render. One source of truth.

15where to put it

Lifting state up

In one line: When two components need the same data, move that state up to their closest common parent and pass it down to both.

This is the standard answer to "where should this state live?" Put state in the lowest component that all the components needing it share as an ancestor. The parent holds the value and an updater; each child receives the value as a prop and calls the updater to change it. Now they're always in sync.

Everyday analogy

Two kids arguing over the TV remote. The fix isn't two remotes — it's giving the remote to the parent (common ancestor), who changes the channel for both. One source of truth.

Practice problem

A temperature app has a Celsius input and a Fahrenheit input that should stay in sync. Where's the state?

Show solution
Lift it to the parent — store one temperature (say, Celsius) and a setter there. Both inputs are controlled by props derived from it and call the setter on change. The parent converts as needed, keeping both fields consistent.

Unit 4 · Events & forms

16core mechanism

Handling events

In one line: Pass a function to event props like onClick — React calls it when the event happens.

Event handlers are functions you attach via camelCase props (onClick, onChange, onSubmit). Pass the function itself, not its result. To pass arguments, wrap it in an arrow function. Most handlers end by updating state, which triggers a re-render.

Everyday analogy

Leaving instructions with a receptionist: "when this person arrives, do X." You hand over the instructions (the function), you don't perform X right now.

jsx · pass the function, not the call
// ✅ pass the function reference
<button onClick={handleClick}>Go</button>

// ✅ wrap to pass an argument
<button onClick={() => remove(id)}>Delete</button>

// ❌ this CALLS handleClick during render, not on click
<button onClick={handleClick()}>Go</button>
Practice problem

Your button's handler runs once on page load and never on click. What did you write?

Show solution
You wrote onClick={handleClick()} with parentheses — that calls it during render and passes the return value (often undefined) as the handler. Remove the parentheses, or wrap: onClick={() => handleClick()}.

17which to use

Controlled vs uncontrolled

In one line: A controlled input's value lives in React state (React is the source of truth); an uncontrolled input keeps its own value in the DOM and you read it only when needed.

Controlled: value={state} + onChange — every keystroke flows through state, so the UI always reflects state and you can validate or transform live. Uncontrolled: let the DOM hold the value and grab it via a ref on submit — simpler, less re-rendering, but you can't react to each change.

Everyday analogy

Controlled is a live teleprompter — every word you type is mirrored on the screen instantly because the screen drives it. Uncontrolled is scribbling on paper and reading it aloud only at the end.

You want to disable a "submit" button until the email field contains "@". Controlled or uncontrolled?

18practical

Forms

In one line: Handle a form with one onSubmit (call preventDefault), and manage fields with controlled state — often a single object for many fields.

The browser's default form behavior reloads the page; event.preventDefault() stops that so you handle submission in JS. For many fields, store them in one object and update by key, so you don't write a separate handler per input.

Everyday analogy

A single intake clipboard with many lines. One pen updates whichever line you're on; one "submit" hands the whole clipboard in at once.

jsx · one object, one change handler, one submit
const [form, setForm] = useState({ name: "", email: "" });

function change(e) {
  setForm({ ...form, [e.target.name]: e.target.value });
}
function submit(e) {
  e.preventDefault();          // stop the page reload
  save(form);
}

<form onSubmit={submit}>
  <input name="name"  value={form.name}  onChange={change} />
  <input name="email" value={form.email} onChange={change} />
  <button>Save</button>
</form>
Practice problem

Your form briefly flashes and the data vanishes on submit. Likely cause?

Show solution
You forgot e.preventDefault(), so the browser performed its default submit and reloaded the page, wiping state. Add it as the first line of your submit handler.

Unit 5 · Effects

19core mechanism

useEffect

In one line: useEffect runs code after render to synchronize your component with something outside React — a network request, a subscription, a timer, the document title.

Rendering should be pure: compute UI from props/state, no surprises. Anything that reaches the outside world (called a "side effect") goes in useEffect, which React runs after committing the DOM. Think of it as "keep this external thing in sync with my state," not "do this on a button click" (that's an event handler).

Everyday analogy

After you finish redecorating a room (render), you then water the plants and adjust the thermostat (effects) — the after-the-fact chores that connect the room to the outside world.

jsx · sync the document title with state
import { useState, useEffect } from "react";

function Title() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    document.title = `Count: ${count}`;  // side effect
  }, [count]);                          // re-sync when count changes

  return <button onClick={() => setCount(count + 1)}>+</button>;
}
Practice problem

Should "save the form when the user clicks Save" go in a useEffect?

Show solution
No. That's a response to a specific user action — put it in the button's onClick handler. Effects are for synchronizing with external systems based on state, not for handling discrete events.

20core mechanism

The dependency array

In one line: The second argument to useEffect lists the values it depends on; React re-runs the effect only when one of them changes between renders.

This array is exactly "how the effect changes when your data changes." [] = run once after mount. [a, b] = run after mount and whenever a or b changes. No array = run after every render. List every prop/state the effect reads, or it'll work with stale values — the linter's "exhaustive-deps" rule catches this.

Everyday analogy

A standing instruction with a trigger: "re-check the weather only when the city changes." If you never list the trigger, you never re-check; if you list the wrong one, you re-check at the wrong times.

21core mechanism

Cleanup

In one line: Return a function from an effect to undo it — React runs that cleanup before the effect re-runs and when the component unmounts.

If your effect starts something ongoing — a timer, a subscription, an event listener — you must stop it, or you leak resources and stack duplicates. The returned cleanup function runs before each re-run (to tear down the old version) and on unmount. Setup and cleanup come in pairs.

Everyday analogy

Open a window for fresh air (setup); close it before you leave (cleanup). Forget to close, and you've left windows open all over the house every time you re-decorate.

jsx · subscribe on setup, unsubscribe on cleanup
useEffect(() => {
  const id = setInterval(tick, 1000);   // setup
  return () => clearInterval(id);       // cleanup
}, []);                                  // one timer, cleared on unmount
Practice problem

You add a resize listener in an effect with no cleanup. After many re-renders the page slows to a crawl. Why?

Show solution
Each run adds another listener without removing the old ones, so dozens fire on every resize. Return a cleanup that calls removeEventListener so each run replaces, not stacks.

Effects are overused. Reserve effects for genuine external synchronization: network, subscriptions, the DOM, browser APIs.

Don't hire a courier (an effect) to walk a note across your own desk. Just pick it up (derive it). Couriers are for things outside the building.

A building's PA system. Instead of relaying a message person-to-person down every hallway (prop drilling), you announce once and everyone who's listening hears it directly.

In one line: useMemo caches an expensive computed value and useCallback caches a function, recomputing only when their dependencies change — use them to skip needless work, not by default.

Both are performance tools. useMemo(() => compute(a), [a]) avoids recomputing on unrelated renders. useCallback(fn, [a]) keeps the same function reference so a memoized child (Lesson 30) doesn't re-render. They have a cost, so reach for them when you have a measured, real expense — a heavy calculation or a stable reference a memoized child needs.

Everyday analogy

Saving the result of a slow calculation on a sticky note so you don't redo it next time — useful for genuinely slow sums, pointless for "2 + 2."

  []
);

27pattern

Custom hooks

In one line: A custom hook is a function starting with use that bundles reusable stateful logic by calling other hooks — sharing behavior, not UI.

When two components need the same logic (track window size, debounce a value, fetch a resource), extract it into a function named useSomething that calls the built-in hooks and returns what callers need. Each component that uses it gets its own independent state — you're sharing the logic, not the data.

Everyday analogy

A recipe card you can hand to any cook. Everyone follows the same steps, but each makes their own dish in their own kitchen. The card (hook) is shared; the meals (state) are separate.

jsx · extract logic into a reusable hook
// reusable: tracks the window width
function useWindowWidth() {
  const [w, setW] = useState(window.innerWidth);
  useEffect(() => {
    const on = () => setW(window.innerWidth);
    window.addEventListener("resize", on);
    return () => window.removeEventListener("resize", on);
  }, []);
  return w;
}

// any component can now use it in one line
function Layout() {
  const width = useWindowWidth();
  return width < 700 ? <Mobile/> : <Desktop/>;
}
Practice problem

Two custom hooks both call useState(0). Do they share that count?

Show solution
No. Each call to a hook creates its own independent state. Custom hooks share logic, not state — every component (and every hook call) gets a fresh copy.

28rules

Rules of Hooks

In one line: Call hooks only at the top level of a component (or another hook), in the same order every render — never inside conditions, loops, or nested functions.

React tracks hook state by call order, not by name. If a hook is sometimes skipped (inside an if), the order shifts and React mixes up which state belongs to which hook. So hooks must run unconditionally and only from React functions. The official ESLint plugin enforces this for you.

Everyday analogy

Numbered lockers handed out in order. If you sometimes skip a person in line, everyone after gets the wrong locker. Always go in the same order so the numbering stays aligned.

jsx · keep hooks unconditional
// ❌ conditional hook — order can change between renders
if (loggedIn) {
  const [name, setName] = useState("");
}

// ✅ hook at top level; put the condition inside
const [name, setName] = useState("");
if (loggedIn) { /* use name */ }
Practice problem

You wrap useEffect in if (data) and get "rendered fewer hooks than expected." Why?

Show solution
On renders where data is falsy the effect is skipped, changing the number/order of hook calls. Move the hook to the top level and put the if inside the effect (or use the dependency array).

29core mechanism

What triggers a re-render

In one line: A component re-renders when its own state changes, when its parent re-renders, or when a context it reads changes — and re-rendering a component re-renders its children by default.

This is the precise version of "how the UI changes when code changes." Three triggers: (1) you call a state setter in this component; (2) the parent re-rendered (so this child re-runs too); (3) a context value this component consumes changed. Crucially, when a component re-renders, all its descendants re-render too — unless you stop that with memoization (next lesson).

Everyday analogy

Updating a slide near the top of a deck re-prints it and everything nested under it. The slides above are untouched; the branch below refreshes.

By default a child re-renders whenever its parent does. A "do not disturb" sign that says "only knock if something actually changed for me." If nothing relevant changed, the worker keeps their previous result instead of redoing it.

Lazy loading & Suspense

In one line: React.lazy loads a component's code only when it's first needed, and <Suspense> shows a fallback while it loads — shrinking your initial bundle.

Everyday analogy

A buffet that cooks dishes to order instead of laying out everything at once. You wait briefly for the dish you actually want, and the kitchen isn't overwhelmed upfront.

jsx · load on demand with a fallback

      <Chart />   // its code loads only now
    </Suspense>
  );
}

At the route level — lazy-load each page so visitors download only the route they're on, not every page upfront. Heavy, rarely-used widgets (editors, charts) are the next best candidates.

Conditional rendering

In one line: JSX is just expressions, so you choose what to show with ordinary JavaScript — a ternary, a &&, or returning null to render nothing.

There's no special "if" syntax in JSX. You decide what an element should be before you put it in the tree. The three everyday tools: cond ? <A/> : <B/> to pick between two things, cond && <A/> to show something or nothing, and if (…) return … at the top of a component for whole-screen branches like loading states.

Everyday analogy

A traffic light. The same intersection shows different signals depending on state — you don't build a new road, you pick which light is on right now.

Why does {count && <Badge/>} render a stray 0 on screen when count is zero?

Show solution
Because 0 is falsy, 0 && … evaluates to 0 — and React renders the number 0 as text. false, null, and undefined render nothing, but 0 and NaN are real values. Fix it with a real boolean: {count > 0 && <Badge/>}.

Where should state live?

In one line: put each piece of state in the lowest component that needs it — and if two siblings need it, move it up to their closest common parent, no higher.

This is the question that decides whether an app stays sane. The rule of thumb: colocate — keep state as close as possible to where it's used. Only "lift" it when a second component needs to read or change the same value. Lifting too high makes a parent re-render on every keystroke deep below; keeping it too low means siblings can't share it. Aim for the smallest scope that still works.

Everyday analogy

Storing things at home. Keep the salt in the kitchen, not the garage. Shared stuff (the thermostat) goes somewhere both roommates can reach — but you don't move the thermostat to the roof "just in case."

Unit 9 · Talking to a server

35data

Fetching data with Effects

In one line: fetching is a side effect, so it lives in useEffect — kick off the request on mount (or when inputs change), and guard against responses arriving out of order.

To load data when a component appears, run a fetch inside useEffect and store the result in state. The subtle part is race conditions: if a prop changes fast, an older request can resolve after a newer one and overwrite fresh data with stale data. The fix is an ignore flag set in the effect's cleanup — when a new effect runs, the old one is told to discard its result.

Everyday analogy

Ordering takeout, then changing your order. If the first kitchen is slower, its meal might show up last. The ignore flag is you telling the door: "if the old order arrives, refuse it — I only want the latest."

jsx · fetch on change, ignore stale responses
useEffect(() => {
  let ignore = false;            // this run's "is it still current?" flag
  setData(null);               // show loading while we fetch
  fetch(`/api/user/` + id)
    .then(r => r.json())
    .then(json => { if (!ignore) setData(json); });
  return () => { ignore = true; };  // next run disowns this one
}, [id]);                       // re-fetch whenever id changes
Practice problem

Without the ignore flag, you type "a", then quickly "ab". Sometimes the screen shows results for "a". Why?

Show solution
Network responses aren't guaranteed to return in the order they were sent. The "a" request can resolve after "ab", calling setData last and clobbering the newer results. The cleanup flag makes the stale callback a no-op, so only the latest request can update state.

Loading, error, success

In one line: any async data has at least three on-screen states — loading, error, and success — and your UI should explicitly render all three, not just the happy path.

Beginners render only the success case, so the screen is blank while loading and crashes (or stays blank) on failure. A resilient component tracks status and branches: show a spinner or skeleton while pending, a friendly message plus retry on error, and the data on success. Don't forget the empty success case — "loaded fine, but there are zero items" deserves its own message.

Everyday analogy

An elevator panel. It tells you it's coming (loading), it's out of service (error), or which floor it's on (success). A panel that only lights up when the elevator arrives leaves you staring at a blank wall.

jsx · render every state explicitly
if (status === "loading") return <Spinner />;
if (status === "error")   return <Error onRetry={refetch} />;
if (items.length === 0) return <Empty />;
return <List items={items} />;   // success with data
Why it matters

Treating status as a single value ("idle" | "loading" | "error" | "success") instead of juggling separate isLoading / isError booleans prevents impossible combinations like "loading and error at once."

Practice problem

Your component uses isLoading and data. On error it sets isLoading=false but leaves data=null, so it renders a blank list forever. What's a sturdier model?

Show solution
Use one status enum plus separate data and error fields. Then each render path is explicit and you can't fall into the "not loading, no data, no error shown" gap. Libraries like React Query (next lesson) give you this for free.

Server state & React Query

In one line: data that lives on a server isn't really your component's state — it's a cache of someone else's state, and tools like TanStack Query (React Query) manage that cache so you stop hand-rolling fetch + loading + error + refetch every time.

The big mental shift: server state is different from UI state. It can go stale, it's shared across components, and it needs caching, deduplication, retries, and background refresh. Doing all that by hand with useEffect gets repetitive and buggy. A query library gives you a useQuery hook that returns data, isLoading, and error, caches by key, dedupes identical requests, and refetches in the background to keep things fresh.

Everyday analogy

A library branch holding copies of books owned by a central archive. The branch caches popular books, lends the same copy to many readers, and quietly swaps in updated editions — so you don't drive to the archive every time.

Hand-rolled useEffect fetchQuery library (TanStack Query / SWR)

Two components fetching same URL → two requestsDeduped & shared from one cache

jsx · the whole fetch lifecycle in a few lines
function Profile({ id }) {
  const { data, isLoading, error } = useQuery({
    queryKey: ["user", id],          // cache identity
    queryFn: () => fetch(`/api/user/` + id).then(r => r.json()),
  });
  if (isLoading) return <Spinner />;
  if (error)     return <Error />;
  return <h1>{data.name}</h1>;
}

Show solution
You'd be duplicating the cache into local state and then fighting to keep them in sync — exactly the bug-prone work the library exists to remove. Read directly from the query's data; only put genuinely local UI state (a toggle, a draft input) in useState.

Vite vs. Next.js — which to reach for

In one line: Vite builds a fast plain-React single-page app and gets out of your way; Next.js is a full framework that adds server rendering, routing, and data conventions — pick based on whether you need a server.

React itself is just the rendering library; you need a tool around it. Vite gives you a lightning-fast dev server and a build step for a client-rendered SPA — ideal for dashboards, internal tools, and apps that live behind a login. Next.js is opinionated: file-based routing, server-side rendering and static generation for SEO, and React Server Components that run on the server and ship less JavaScript. If you need great SEO, server data fetching, or a marketing-plus-app site, a framework earns its weight; if you just need a snappy app shell, Vite is lighter.

Vite is a high-quality engine you drop into your own chassis. Next.js is a finished car — engine, wheels, dashboard, and a recommended way to drive — faster to ship, but you adopt its conventions.

In one line: most apps don't need a global state library — start with local state and Context; reach for Zustand, Redux, or Jotai only when truly shared client state gets painful to pass around.

First, separate the two kinds of state: server state (cached data from an API — use a query library, L37) and client state (UI things like a sidebar toggle, a wizard step, a shopping cart). The honest default is: useState for local, lift it up (L15) when shared, Context (L25) for low-frequency global values like theme or auth. Only when client state is large, frequently updated, and spread across the tree does a dedicated store pay off.

You don't rent a warehouse to store a few boxes. A closet (local state) handles most things; a shared garage (Context) handles the rest; you only rent the warehouse (a state library) when you genuinely have warehouse-scale stuff.

Styling approaches

In one line: there's no single "React way" to style — CSS Modules, Tailwind, and CSS-in-JS all work; they trade off between familiarity, scoping, and how close styles sit to your markup.

React renders className strings; how you produce those is up to you. The main families: plain CSS / CSS Modules (real CSS files, with Modules giving you locally-scoped class names so styles don't leak), utility-first (Tailwind) (compose tiny class names right in the JSX — fast to write, styles live with markup), and CSS-in-JS (write styles in JavaScript, co-located and dynamic, e.g. styled-components — though runtime CSS-in-JS has fallen out of favor for performance). Pick for your team's comfort and your need for scoping and dynamism, not for fashion.

Everyday analogy

Ways to label moving boxes: a separate inventory sheet (CSS files), stickers right on each box (Tailwind utilities), or a smart label that changes text based on contents (CSS-in-JS). All get the boxes labeled; they differ in where the labels live and how dynamic they are.

ApproachTrade-off
Plain CSS / CSS ModulesFamiliar, zero runtime; Modules add automatic scoping
Tailwind (utility-first)Very fast to build, styles co-located; verbose class strings
CSS-in-JS (styled-components)Dynamic & co-located; runtime cost — prefer zero-runtime options
Component libraries (MUI, shadcn/ui)Pre-built styled components; less control, faster to ship
Practice problem

A solo developer wants to ship a small app fast, doesn't enjoy naming CSS classes, and is fine with verbose markup. Which approach fits?

Show solution
Tailwind suits that profile well: no class-naming, styles right in the JSX, quick iteration. A team that prefers separation of concerns and has a designer might lean to CSS Modules instead. There's no wrong answer — match it to who's building and maintaining it.
Freshness note: as of mid-2026 Tailwind and CSS Modules dominate new projects, runtime CSS-in-JS has cooled (especially with Server Components), and "zero-runtime" CSS-in-JS tools are rising. Verify the current landscape before standardizing a stack.
