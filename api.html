Foundations: what an API even is

Start from zero. By the end you can explain client–server, HTTP, and the word REST without hand-waving.

l-what-apiWhat is an API?

Key takeawayAn API is a contract: a defined set of requests one program can make to another, and the responses it will get back.
Everyday analogy

A restaurant menu. You don't walk into the kitchen and cook. You read the menu (the documented options), tell the waiter what you want (a request), and get a plate back (a response). The kitchen can change completely and your experience stays the same, as long as the menu keeps its promises.

The idea, in plain language

API stands for Application Programming Interface. It's the agreed-upon way for two pieces of software to talk. One side offers capabilities; the other side uses them, without needing to know how they're built internally.

You already use APIs constantly as a developer: calling a function from a library is using that library's API. A web API is the same idea but over a network — the two programs live on different machines and talk by sending messages.

The key word is contract. An API says: "send me a request shaped like this, and I'll respond with something shaped like that." As long as both sides honor the contract, either side can be rewritten freely.

Diagram
Why it exists

Software needs to reuse other software without copying it or knowing its internals. APIs make that possible across teams, companies, and machines.

Where it's used

Everywhere: payment (Stripe), maps (Google), auth (Auth0), your own frontend talking to your own backend, microservices talking to each other.

How to use it

Read the provider's docs to learn the contract, then send requests from your code (with a library like fetch, axios, or requests) and handle the responses.

Code

Calling a public API with the built-in fetch. This is the whole idea in five lines.

JavaScript · a real API call
// Ask GitHub's API for info about a user.
// We send a request to a URL; we get JSON back.
const res = await fetch("https://api.github.com/users/torvalds");
const data = await res.json();        // parse the JSON response body
console.log(data.name, data.public_repos);
// -> "Linus Torvalds" 7  (numbers vary)
Worked example
Walk-through

Imagine a weather app on your phone. It doesn't contain weather data. When you open it, it sends a request like GET /forecast?city=Hyderabad to a weather company's API.

The weather company's servers measure, model, and store the data. They expose an API contract: "ask me for a city, I'll return temperature and conditions as JSON." The app receives that JSON and draws a nice screen. Two completely separate systems, cooperating through one agreed contract.

Practice

Your team builds an online store. The frontend (website) needs the list of products, which live in a database the frontend can't touch directly. Describe, in API terms, how the frontend should get the products.

SolutionShow solution

The backend exposes an API endpoint, e.g. GET /products. The frontend (client) sends a request to that endpoint. The backend (server) reads the database and returns the products as a response, typically JSON. The frontend never touches the database directly — it only knows the API contract. If the database changes (new engine, new schema), the API contract can stay the same, so the frontend keeps working.

Most asked interview questions
QIn one sentence, what is an API?

A defined contract that lets one program request capabilities or data from another program, and receive a structured response, without knowing how the other program is implemented.

QWhat's the difference between an API and a database?

A database stores data. An API is an interface in front of capabilities or data. A client should talk to the API, not the database directly — the API enforces the contract, adds validation, authentication, and business rules, and hides the storage details so they can change freely.

l-client-serverThe client–server model

Key takeawayIn every web API call there is a client that asks and a server that answers — and the network in between only carries messages.
Everyday analogy

Ordering food delivery. You (client) place an order through an app. The restaurant (server) prepares and sends it. The delivery road network just carries the order and the food — it doesn't cook. The same road can carry thousands of unrelated orders.

The idea, in plain language

The web runs on a client–server split. The client initiates: it sends a request. The server waits, receives requests, does work, and sends responses. The roles are about who starts the conversation, not about who is more important.

A client can be a browser, a mobile app, a command-line tool like curl, or another server. A server is just a program listening for requests on a network address and port.

This separation matters because each side can scale and evolve independently. You can have one server answering millions of clients, or many servers behind a load balancer answering the same client over time.

Compare
RoleWho starts?ExamplesJob
ClientAlways initiates the requestBrowser, mobile app, curl, another serviceAsk for something, then use the response
ServerWaits, never initiates (in plain HTTP)Web server, API backendReceive requests, do work, respond
Why it exists

Separating who-asks from who-answers lets the two sides be built by different teams, in different languages, and scaled independently.

Where it's used

Every website, every mobile app fetching data, every microservice call inside a company's backend.

How to use it

Pick a client library to send requests; run a server framework (Express, FastAPI, Spring) to receive them. They only share the API contract.

Code

curl is one of the simplest HTTP clients. Here it asks a server for a page.

Terminal · curl is a client
# curl is the client; example.com is the server.
# -i shows the response headers + status line too.
curl -i https://example.com

# You'll see something like:
# HTTP/2 200
# content-type: text/html; charset=UTF-8
# ... then the HTML body ...
Worked example
Walk-through

You type a URL in your browser. The browser (client) opens a connection to the server at that domain and sends an HTTP request. The server finds or builds the page, then sends back a response with a status code and the HTML. The browser renders it.

Crucially, the server didn't push anything first. It sat idle until your request arrived. That's the heart of the model: request in, response out.

Practice

True or false: in classic HTTP, a server can send a message to a client whenever it wants, without the client asking first. Explain.

SolutionShow solution

False. In classic HTTP, the server only responds to requests the client initiates — it cannot start the conversation. To get server-initiated messages you need a different mechanism layered on top, like WebSockets, Server-Sent Events, or polling/long-polling. Plain REST over HTTP is strictly request → response.

Most asked interview questions
QWhat defines the 'client' vs the 'server' in a web request?

The client is whoever initiates the request; the server is whoever receives it and responds. It's about who starts the conversation, not about hardware. A backend service calling another backend service is acting as a client for that call.

QCan one machine be both a client and a server?

Yes. A backend API server receives requests from browsers (acting as a server) while also calling a payment provider's API (acting as a client). Roles are per-connection, not per-machine.

l-httpHTTP: the language of the web

Key takeawayHTTP is the plain-text request/response protocol REST is built on; learn its shape and REST stops being mysterious.
Everyday analogy

A standard postal envelope. There's an address line (the URL), a few lines of metadata on the outside (headers), and the letter inside (the body). Every post office in the world understands the envelope format, so any sender and receiver can communicate.

The idea, in plain language

HTTP (HyperText Transfer Protocol) is the set of rules for how clients and servers exchange messages on the web. Almost all REST APIs ride on HTTP.

An HTTP request has four parts: a method (the verb, like GET), a path/URL (what you want), headers (metadata), and an optional body (data you're sending).

An HTTP response has three parts: a status code (did it work?), headers (metadata about the response), and an optional body (the data you asked for).

Use HTTPS — HTTP encrypted with TLS — for anything real. Same protocol, with privacy and tamper-protection added.

Diagram
Why it exists

Without a shared protocol, every pair of programs would invent its own way to talk. HTTP is the universal envelope everyone agreed on.

Where it's used

Browsers, mobile apps, IoT devices, server-to-server calls, webhooks — nearly all of them speak HTTP/HTTPS.

How to use it

You rarely write raw HTTP by hand; libraries build the request and parse the response. But knowing the four request parts and three response parts makes everything else click.

Code

This is what a request literally looks like on the wire (libraries generate this for you).

Raw HTTP request
GET /users/1 HTTP/1.1
Host: api.shop.com
Accept: application/json
User-Agent: my-app/1.0

And the server's reply:

Raw HTTP response
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 41

{
  "id": 1,
  "name": "Ada Lovelace"
}
Worked example
Walk-through

Say your code runs fetch("https://api.shop.com/users/1"). Under the hood the library builds the raw request shown above: method GET, path /users/1, a Host header, and an Accept header saying "I'd like JSON."

The server replies with 200 OK, a Content-Type: application/json header so your code knows how to parse it, and the JSON body. Your library reads the status, checks the content type, and hands you a parsed object.

Practice

Name the four parts of an HTTP request and the three parts of an HTTP response.

SolutionShow solution

Request: (1) method/verb, (2) path or URL, (3) headers, (4) optional body. Response: (1) status code (with reason text), (2) headers, (3) optional body. A handy memory hook: a request says what to do and with what data; a response says what happened and here's the result.

Most asked interview questions
QWhat's the difference between HTTP and HTTPS?

HTTPS is HTTP carried over a TLS-encrypted connection. Same messages and semantics, but the traffic is encrypted (privacy), integrity-checked (tamper-evident), and the server is authenticated via a certificate. Always use HTTPS for real APIs, especially anything with credentials.

QIs REST tied to HTTP?

REST is an architectural style described independently of any protocol, but in practice REST APIs are almost always built on HTTP and lean heavily on HTTP's methods, status codes, and headers. When people say 'REST API' they nearly always mean 'HTTP + JSON following REST conventions.'

l-what-restWhat REST actually means

Key takeawayREST is a style for designing networked APIs around resources you act on with standard HTTP methods — not a protocol or a library.
Everyday analogy

REST is like a set of architectural building codes, not a brand of house. The code says 'put outlets every few feet, stairs at this slope.' Many different houses follow it. REST says 'model things as resources, use standard verbs, be stateless.' Many different APIs follow it.

The idea, in plain language

REST stands for REpresentational State Transfer. It was described by Roy Fielding in 2000 as an architectural style — a set of constraints — for building scalable networked systems. The web itself is the biggest REST system.

The core mental shift: think in resources (nouns) instead of actions. A user, an order, a product — each is a resource with a URL. You don't call getUser or deleteUser; you apply standard HTTP methods to the user's URL: GET /users/1, DELETE /users/1.

"Representational state transfer" means: the server holds a resource's state, and when you ask for it you get a representation of that state (often JSON). You transfer representations back and forth; you don't manipulate the server's internals directly.

An API that follows these conventions well is called RESTful.

Why it exists

A common, predictable style means any developer can guess how your API works. It scales well and caches well because it reuses the web's own design.

Where it's used

The default style for public and internal web APIs for two decades: Stripe, GitHub, Twitter/X, Shopify, and most company backends.

How to use it

Model your domain as resources, give each a clean URL, expose them through GET/POST/PUT/PATCH/DELETE, return JSON, and use HTTP status codes honestly.

Code

In REST, one resource URL plus different HTTP methods covers most operations.

The same resource, four verbs
GET    /users/1     # read user 1
PUT    /users/1     # replace user 1 entirely
PATCH  /users/1     # update part of user 1
DELETE /users/1     # remove user 1
GET    /users       # list users
POST   /users       # create a new user
Worked example
Walk-through

Suppose you're designing an API for a blog. The action-oriented instinct is to make endpoints like /createPost, /editPost, /removePost. That's a verb-per-feature explosion.

The REST instinct: there is a resource called posts. The collection is /posts and a single post is /posts/42. Creating is POST /posts. Editing is PATCH /posts/42. Removing is DELETE /posts/42. Reading one is GET /posts/42. Six well-known verbs cover everything, and any developer can predict the URLs.

Practice

Rewrite these RPC-style endpoints in RESTful form: POST /fetchOrders, POST /addOrder, POST /cancelOrder?id=9.

SolutionShow solution

POST /fetchOrders → GET /orders (read the collection). POST /addOrder → POST /orders (create in the collection). POST /cancelOrder?id=9 → this one is interesting: a pure REST take is DELETE /orders/9 if cancelling means removing, or PATCH /orders/9 with body {"status":"cancelled"} if the order still exists but changes state. Cancelling usually means a state change, so PATCH is often the better fit.

Most asked interview questions
QWhat does REST stand for and what is it?

Representational State Transfer. It's an architectural style (a set of constraints) for networked APIs, introduced by Roy Fielding in 2000. It centers on resources identified by URLs, manipulated through a uniform set of methods, with stateless interactions and representations (like JSON) transferred between client and server.

QWhat makes an API 'RESTful' rather than just HTTP?

Following REST's conventions: resources as nouns in URLs, correct use of HTTP methods for their defined semantics, meaningful status codes, statelessness, and a uniform interface. Many 'REST APIs' are only partly RESTful (e.g. they ignore HATEOAS), which is fine in practice — full compliance is a spectrum.

l-constraintsThe six REST constraints

Key takeawayREST is defined by six constraints; the ones that bite daily are statelessness, uniform interface, and cacheability.
Everyday analogy

Think of a franchise like a coffee chain. Every branch follows the same rulebook (uniform interface), any branch can serve you because they don't rely on remembering your last visit (stateless), and popular drinks are pre-made for speed (cache). The rulebook is what makes the chain scale to thousands of branches.

The idea, in plain language

Fielding defined REST with six constraints. Five are required; one (code-on-demand) is optional. Together they produce systems that scale and stay simple.

1. Client–server. Separate the UI/client concerns from data/storage concerns. Each evolves independently.

2. Stateless. Every request carries everything the server needs. The server keeps no client session between requests. (Deep-dived later — it's the big one.)

3. Cacheable. Responses must say whether they can be cached, so clients and intermediaries can reuse them and cut traffic.

4. Uniform interface. The same small set of conventions everywhere: resources with URLs, standard methods, self-descriptive messages. This is what makes REST predictable.

5. Layered system. A client can't tell whether it's talking to the real server or a proxy/load balancer/cache in front of it. Lets you insert infrastructure freely.

6. Code-on-demand (optional). The server may send executable code (e.g. JavaScript) to extend the client. Rarely emphasized in APIs.

CacheableResponses declare cacheabilityLess traffic, faster apps
Uniform interfaceConsistent resources + methodsPredictable, learnable APIs
Layered systemHidden intermediariesAdd proxies, LBs, gateways freely
Code-on-demand(optional) ship code to clientRarely used in APIs
Why it exists

These constraints aren't bureaucracy — each one buys a concrete scaling or simplicity benefit, which is why the web works at planetary scale.

Where it's used

They shape how load balancers, CDNs, and API gateways are deployed in front of real services every day.

How to use it

You mostly 'use' them by not violating them: keep requests self-contained, set cache headers, and stick to standard methods and URLs.

Worked example
Walk-through

Why does statelessness enable scaling? Picture a service behind a load balancer with 10 identical server instances. If each request carries its own auth token and all needed data, the load balancer can send each request to any instance. No instance has to 'remember' you.

If instead the server stored your session in memory, request #2 must go back to the exact instance that handled request #1, or it won't recognize you. That's sticky, fragile, and hard to scale. The stateless constraint is what removes that headache.

Practice

Your API stores each logged-in user's 'shopping cart' in server memory tied to the specific server that handled login. Which REST constraint does this violate, and what problem will it cause in production?

SolutionShow solution

It violates statelessness. The cart state lives on one specific server instance, so later requests must be routed back to that same instance (sticky sessions). When you scale to multiple instances or one crashes, carts vanish or requests fail. The fix: store cart state in a shared store (database/Redis) keyed by the user, and have the client send its identity (a token) on every request so any instance can serve it.

Most asked interview questions
QList the REST constraints.

Client–server, stateless, cacheable, uniform interface, layered system, and the optional code-on-demand. The five required ones plus one optional.

QWhich constraint most directly enables horizontal scaling, and why?

Statelessness. Because each request is self-contained, any server instance can handle any request, so you can add or remove instances freely behind a load balancer without sticky sessions or shared in-memory state.

QWhat is the 'uniform interface' and why is it valuable?

It's the agreement to use the same conventions everywhere: resources identified by URLs, a standard set of methods with defined semantics, and self-descriptive messages. Its value is predictability and decoupling — clients can interact with any RESTful resource the same way without bespoke knowledge.

l-resourcesResources and representations

Key takeawayA resource is the thing (a noun); a representation is one snapshot of that thing in a format like JSON.
Everyday analogy

A person is a resource. Their passport photo, their LinkedIn page, and a verbal description are all representations of that same person — different formats, same underlying thing. The person doesn't change when you take a new photo; you just transferred a new representation.

The idea, in plain language

In REST, a resource is any meaningful concept your API exposes: a user, an order, a product, a comment, even an abstract thing like 'today's weather.' Each resource has a stable identifier — its URL.

A representation is a concrete encoding of a resource's current state at a moment in time. The most common format is JSON, but the same resource could be returned as XML, CSV, or HTML. The client and server negotiate the format via headers (content negotiation).

This is the 'RST' in REST: you transfer representations of state. The server keeps the authoritative state; you receive copies (representations) and send proposed new states back.

Resources come in two flavors: collections (e.g. /users — many) and single items (e.g. /users/1 — one). This pairing drives almost all REST URL design.

Diagram
Why it exists

Separating the thing from its encoding lets one API serve many clients (a JSON app, a CSV export tool) without changing the underlying resource.

Where it's used

Every REST endpoint you'll ever design names a resource; content negotiation shows up when an API offers JSON and CSV exports of the same data.

How to use it

Name resources as plural nouns, give collections and items their own URLs, return JSON by default, and let the Accept header request other formats if you support them.

Code

Content negotiation: the client asks for a format with Accept; the server answers with Content-Type.

Same resource, two representations
GET /users/1 HTTP/1.1
Accept: application/json
# -> 200 OK, Content-Type: application/json
#    { "id": 1, "name": "Ada" }

GET /users/1 HTTP/1.1
Accept: application/xml
# -> 200 OK, Content-Type: application/xml
#    <user><id>1</id><name>Ada</name></user>
Worked example
Walk-through

Consider a report endpoint /reports/q3. A dashboard might request it with Accept: application/json to draw charts. An analyst might request the exact same URL with Accept: text/csv to open it in a spreadsheet.

Same resource (the Q3 report), same URL, two representations. The server picks the format based on the Accept header and labels its reply with Content-Type. The resource's identity never changed.

Practice

A teammate says 'the URL /users/1.json and /users/1.xml are two different resources.' Are they right?

SolutionShow solution

Conceptually, no — they're two representations of the same resource (user 1). The cleaner REST approach is one URL (/users/1) with content negotiation via the Accept header. That said, file-extension URLs are a common pragmatic shortcut some APIs use to make formats explicit. The key point: the resource is the user; JSON and XML are just encodings of its state.

Most asked interview questions
QWhat is the difference between a resource and a representation?

A resource is the conceptual thing the API exposes (e.g. user #1), identified by a URL. A representation is a concrete encoding of that resource's state at a point in time, such as a JSON or XML document. You transfer representations; the resource itself is abstract and lives on the server.

QWhat's the difference between a collection resource and an item resource?

A collection (e.g. /users) represents a group of items and typically supports listing (GET) and creating (POST). An item (e.g. /users/1) represents a single member and supports reading (GET), replacing (PUT), updating (PATCH), and deleting (DELETE).

HTTP methods & messages

The verbs (GET/POST/PUT/PATCH/DELETE), headers, and content types — the working vocabulary of every REST API.

l-methodsHTTP methods: the verbs

Key takeawayThe HTTP method tells the server what kind of action you intend; pick it by meaning, not by habit.
Everyday analogy

At a library counter, the verb is your intent. 'I'd like to read this' (GET), 'please add this new book' (POST), 'replace this damaged copy' (PUT), 'just fix the title typo' (PATCH), 'remove this book' (DELETE). The book (resource) is the same; your verb changes what happens.

The idea, in plain language

HTTP defines a handful of methods (also called verbs). REST maps each to a kind of operation on a resource. The five you'll use constantly:

  • GET — read a resource. Never changes anything.
  • POST — create a new resource, or trigger an operation.
  • PUT — replace a resource entirely.
  • PATCH — update part of a resource.
  • DELETE — remove a resource.

Two more you'll see occasionally: HEAD (like GET but returns headers only, no body) and OPTIONS (asks what methods are allowed; central to CORS preflight).

The method is the single most important design decision per request, because clients, proxies, and caches all behave differently based on it. Using GET to delete something, for example, breaks the web's assumptions and can cause real damage.

Why it exists

A shared verb vocabulary means caches, proxies, browsers, and other developers all understand your intent without reading docs.

Where it's used

Every CRUD app maps Create→POST, Read→GET, Update→PUT/PATCH, Delete→DELETE. This single table covers most endpoints you'll ever write.

How to use it

Choose the method by its defined meaning; let the URL name the resource. Don't bury verbs in URLs (/users/1/delete is wrong).

Try it live

Mini REST API playground

A real (in-memory) /users API running entirely in your browser. Pick a method, set a path, send. Watch the status codes and semantics behave exactly like a real REST service.

Ideas to try: GET /users, then GET /users/1, then GET /users/99 (404). Switch to POST /users and send twice — notice you get two users (POST isn't idempotent). Then DELETE /users/1 twice — the second returns 404 but the end state is identical (idempotent).

Code
curl · one resource, five verbs
# Read
curl https://api.shop.com/users/1

# Create (note -X POST and a JSON body)
curl -X POST https://api.shop.com/users \
  -H "Content-Type: application/json" \
  -d '{"name":"Grace","role":"user"}'

# Replace entirely
curl -X PUT https://api.shop.com/users/1 \
  -H "Content-Type: application/json" \
  -d '{"name":"Ada","role":"admin"}'

# Partial update
curl -X PATCH https://api.shop.com/users/1 \
  -H "Content-Type: application/json" \
  -d '{"role":"admin"}'

# Delete
curl -X DELETE https://api.shop.com/users/1
Worked example
Walk-through

You have a to-do item at /todos/7. To mark it done you have two reasonable choices: PATCH /todos/7 with body {"done": true} (change just that field), or PUT /todos/7 with the entire to-do object including done: true (replace the whole thing).

PATCH is usually the better fit because you only want to change one field and you don't want to accidentally wipe fields you didn't send. We'll dig into the PUT-vs-PATCH difference in its own lesson.

Practice

A developer creates GET /users/5/delete to remove a user 'because it's easy to call from a browser link.' List two concrete problems this causes.

SolutionShow solution

(1) Caches and crawlers may issue GET requests automatically (prefetching, search bots), so the user could be deleted without anyone intentionally clicking — GET must be safe and side-effect-free. (2) It violates the uniform interface: the destructive intent is hidden in the URL instead of the method, so proxies, logs, and other developers can't tell it's a mutation. The correct design is DELETE /users/5.

Most asked interview questions
QName the five main HTTP methods and what each does.

GET reads, POST creates (or triggers an action), PUT replaces a whole resource, PATCH partially updates a resource, DELETE removes a resource.

QWhy shouldn't GET have side effects?

GET is defined as safe — it must not change server state. Browsers, proxies, prefetchers, and crawlers may issue GET requests freely and cache them, so any side effect could be triggered unexpectedly or repeatedly. Mutations belong on POST/PUT/PATCH/DELETE.

l-getGET — reading data

Key takeawayGET retrieves a representation and must never change server state; everything you read goes through it.
Everyday analogy

Reading a notice board. You can look as many times as you like, from as many people as you like, and nothing on the board changes because you looked. That's GET: pure reading, repeatable, harmless.

The idea, in plain language

GET asks the server for a representation of a resource. It's the most common method by far — every page load, every data fetch.

GET is safe (no side effects) and idempotent (calling it many times is the same as once). It's also the most cacheable method, which is why GETs can be served from CDNs and browser caches.

GET requests don't carry a request body. Inputs go in the URL: the path identifies the resource (/users/1) and the query string carries options like filtering, sorting, and pagination (/users?role=admin&sort=name).

Because the URL is the whole input, two identical GET URLs should return the same kind of result — which is exactly what makes caching possible.

Why it exists

Reading is the most frequent operation, so it needs to be cheap, cacheable, and safe to repeat. GET's guarantees deliver that.

Where it's used

Loading a profile, listing products, searching, fetching a dashboard — all GETs. CDNs cache GET responses to serve millions of users cheaply.

How to use it

Put the resource in the path and options in the query string. Never put secrets in the URL (they get logged); use headers for those.

The resource identifier goes in the URL path; filtering/sorting/pagination options go in the query string. GET has no body. Never put secrets in the URL — use headers (e.g. Authorization) instead, because URLs get logged and cached.

l-postPOST — creating data

Key takeawayPOST creates a new resource (or triggers an action) and is the one common method that is NOT idempotent.
Everyday analogy

Submitting a new application form at a counter. Each time you hand in a form, a new record is created. Submit the same form twice and you get two records — that's the non-idempotent nature of POST.

The idea, in plain language

POST sends data to a collection to create a new resource, or asks the server to perform a non-standard action. The data goes in the request body, usually as JSON.

POST is not safe (it changes state) and not idempotent (sending it twice usually creates two resources). This is the key difference from PUT.

On success, the server typically returns 201 Created, a Location header pointing to the new resource's URL, and often the created resource in the body (so the client gets the server-assigned id and timestamps).

POST is the 'catch-all' verb for operations that don't fit the others — e.g. POST /orders/9/cancel or POST /emails/send — but prefer modeling things as resources when you can.

Why it exists

Creation needs a verb where the server assigns the identity (you don't know the new id yet). POST fits because the client posts to the collection, not a specific item URL.

Where it's used

Sign-ups, placing orders, uploading files, submitting forms, sending messages — all POSTs. Also one-off actions that aren't pure CRUD.

How to use it

POST to a collection (/users), put the data in a JSON body with the right Content-Type, and read the Location header and returned body for the new id.

Code
JavaScript · POST to create
const res = await fetch("https://api.shop.com/users", {

  const url = res.headers.get("Location"); // e.g. /users/42
  console.log("Created at", url, created);
}
Worked example
Walk-through

You POST {"name":"Grace Hopper"} to /users. The server assigns id 42, stores the record, and replies:

If your network hiccups and you blindly retry the same POST, you'll likely get another user (id 43) — two records for one intended action. That's why POST needs care on retries (covered in the idempotency lesson, often solved with an idempotency key).

Practice

Root cause: POST is not idempotent, so the retry created a second order. The server records the key with the result; a retry with the same key returns the original result instead of creating a duplicate. (Stripe popularized this pattern.)

Because POST typically creates a new resource each time it's called. Two identical POSTs to a collection generally produce two distinct resources, so the server state after N calls differs from after one call.

QWhat status code and header should a successful POST that creates a resource return?

201 Created, plus a Location header with the URL of the newly created resource. Returning the created resource in the body is also common and convenient (the client gets the server-assigned id).

l-put-patchPUT vs PATCH — updating data

Key takeawayPUT replaces the whole resource (send everything); PATCH changes only the fields you send.
Everyday analogy

Editing a document. PUT is 'delete the old page and put down this entirely new page' — whatever you don't include is gone. PATCH is 'on the existing page, change just this one line' — everything else stays.

The idea, in plain language

Both update an existing resource, but they differ in scope.

PUT replaces the resource entirely with the body you send. If you omit a field, the server treats it as removed or reset. PUT is idempotent: sending the same full representation twice leaves the resource in the same state.

PATCH applies a partial change — only the fields you include are modified; the rest are untouched. PATCH may or may not be idempotent depending on the operation (replacing a value is idempotent; an operation like 'increment' is not).

The classic bug: using PUT but only sending the field you changed. Because PUT means 'replace the whole thing,' the missing fields get wiped. If you only want to change part, use PATCH.

Updates come in two shapes — 'here's the new version of the whole thing' vs 'just change this bit' — and conflating them causes data loss. Two verbs keep the intent explicit.

Where it's used

Profile edits and settings toggles are usually PATCH; replacing a config document wholesale or syncing a known full object is PUT.

QIs PATCH idempotent?

It depends on the operation. A PATCH that sets fields to fixed values ({role:'admin'}) is idempotent. A PATCH expressing a relative operation (e.g. 'increment count by 1') is not, because repeating it changes the state each time.

l-deleteDELETE — removing data

Key takeawayDELETE removes a resource and is idempotent: deleting twice leaves the same end state (gone).
Everyday analogy

Shredding a specific document. The first shred removes it. Shredding the same (now-missing) document again doesn't make it 'more gone' — the end state is identical. That's idempotency.

The idea, in plain language

DELETE removes the resource at a given URL. It's not safe (it changes state) but it is idempotent: after the first successful delete, repeating it doesn't change the outcome — the resource is already gone.

On success, servers usually return 204 No Content (deleted, nothing to send back) or 200 OK with a small confirmation body. A second DELETE of the same id commonly returns 404 Not Found — and that's fine; the state (resource absent) is unchanged, which is what idempotency is about.

Two flavors exist in practice: hard delete (actually removed) and soft delete (marked deleted/archived but kept in storage). Soft deletes are common for audit trails and 'undo,' and are usually implemented behind the same DELETE endpoint.

Why it exists

Removal needs a dedicated, recognizable verb so caches and intermediaries know the resource is gone, and so retries are safe.

Where it's used

Removing an account, cancelling a subscription, clearing an upload, emptying a cart item. Soft deletes power 'Trash'/'Archive' features.

How to use it

Send DELETE /resource/id; expect 204 on success and tolerate 404 on a repeat. For destructive actions, require auth and consider confirmation.

Code
JavaScript · DELETE and handle both cases
async function deleteUser(id) {
  const res = await fetch(`https://api.shop.com/users/${id}`, {
    method: "DELETE",
  });
  // 204 = deleted now; 404 = already gone. Both mean "it's not there."
  if (res.status === 204 || res.status === 404) return true;
  throw new Error(`Unexpected status ${res.status}`);
}
Worked example
Walk-through

You DELETE /users/1. Server removes it → 204 No Content. Your retry logic fires again (maybe the response was slow) → second DELETE /users/1 → 404 Not Found.

Is that a bug? No. The goal of DELETE is 'this resource should not exist,' and it doesn't. Treat 404 on a delete-retry as success. The end state is identical whether you called it once or twice — that's idempotency in action.

Practice

Your app needs an 'undo delete' feature for 30 days. How would you implement DELETE to support that, and what changes for GET?

SolutionShow solution

Use a soft delete: instead of removing the row, set a deletedAt timestamp (or status: 'deleted'). The DELETE endpoint stays the same from the client's view (returns 204). GET endpoints must now exclude soft-deleted records by default (filter deletedAt IS NULL), and you might add a way to list/restore deleted items. A background job hard-deletes records older than 30 days.

Most asked interview questions
QIs DELETE idempotent even though a repeat returns 404?

Yes. Idempotency is about the resulting server state, not the status code. After one DELETE the resource is gone; a second DELETE leaves it gone. The 404 just reports that it was already absent — the state is unchanged, so the method is idempotent.

QWhat's the difference between hard and soft delete?

Hard delete physically removes the data. Soft delete marks it as deleted (e.g. a deletedAt field) but keeps it, enabling undo, audit trails, and referential safety. Soft-deleted records should be filtered out of normal reads.

l-safe-idemSafe and idempotent methods

Key takeawaySafe = no state change; idempotent = repeating it has the same effect as doing it once. These two properties decide what's safe to retry and cache.
Everyday analogy

A light switch labeled 'OFF' is idempotent — flip it to OFF ten times, the light is just off. A doorbell is non-idempotent — ring it ten times, it rings ten times. 'Safe' is like reading the switch's label without touching it: nothing changes at all.

The idea, in plain language

Two formal properties classify HTTP methods, and they drive real engineering decisions about retries and caching.

Safe: the method doesn't change server state. Only GET, HEAD, and OPTIONS are safe. Safe requests can be issued freely, prefetched, and cached.

Idempotent: making the identical request N times leaves the server in the same state as making it once. GET, HEAD, OPTIONS, PUT, and DELETE are idempotent. POST is not (and PATCH is conditionally idempotent).

Why it matters: networks fail. If a request is idempotent, a client (or proxy) can safely retry it after a timeout without risking duplicates. If it's not (POST), retries can create duplicate orders, charges, or records — which is exactly why patterns like idempotency keys exist.

Every safe method is idempotent, but the reverse isn't true: PUT and DELETE are idempotent yet change state, so they're not safe.

Why it exists

Retry logic and caches need to know which requests are safe to repeat. These two properties are the formal contract that makes that automatic.

HTTP client libraries, service meshes, and load balancers auto-retry idempotent methods. CDNs cache safe methods. Idempotency keys protect POSTs.

Design each endpoint to honor the property its method promises. If a POST must be retry-safe, add an idempotency key so duplicates collapse.

Worked example
Walk-through

A payment service times out on POST /charges. Did the charge happen? Unknown. Because POST isn't idempotent, blindly retrying risks double-charging.

This is why robust HTTP clients retry GET/PUT/DELETE automatically but treat POST specially.

Practice

The increment is the gotcha: PATCH is only idempotent when it sets fixed values, not relative ones.

Most asked interview questions
QDifference between safe and idempotent?

Safe = the request causes no state change (read-only). Idempotent = repeating the identical request yields the same server state as doing it once. All safe methods are idempotent; not all idempotent methods are safe (PUT, DELETE change state but are idempotent).

QWhich methods are idempotent, and why does it matter for retries?

GET, HEAD, OPTIONS, PUT, DELETE (and PATCH conditionally). It matters because idempotent requests can be safely retried after network failures without side effects, while retrying a non-idempotent POST can create duplicates — hence idempotency keys.

l-headersHTTP headers

Key takeawayHeaders are key–value metadata about a request or response — content type, auth, caching, and more — separate from the body.
Everyday analogy

The information printed on the outside of a parcel: what's inside, who it's for, whether it's fragile, the tracking number. The contents (body) are inside; the labels (headers) tell everyone how to handle it.

The idea, in plain language

Headers are lines of metadata sent with every request and response, as Name: value pairs. They describe the message rather than being the data itself.

Request headers you'll use constantly: Content-Type (what format the body is), Accept (what formats you can handle back), Authorization (credentials/token), User-Agent (who's calling).

Response headers you'll meet: Content-Type (what's in the body), Cache-Control / ETag (caching), Location (where a created resource lives), Set-Cookie, and rate-limit headers like Retry-After.

Custom headers traditionally started with X- (e.g. X-Request-Id); that convention is now discouraged but still widespread. Header names are case-insensitive.

Most asked interview questions
QDifference between Content-Type and Accept?

Content-Type declares the format of the body being sent in this message. Accept (a request header) tells the server which formats the client is willing to receive in the response. One describes outgoing data; the other negotiates incoming data.

QWhere do authentication credentials usually go, and why not in the URL?

In the Authorization request header (e.g. Bearer <token>). Not in the URL because URLs are logged by servers, proxies, and browser history, and may be cached or shared — leaking the credential. Headers aren't logged by default and aren't part of the cache key in the same way.

l-content-negMedia types & content negotiation

Key takeawayMedia types name a body's format (like application/json); content negotiation lets client and server agree on which format to use.
Everyday analogy

Booking a translator. You say which languages you can understand (Accept), the speaker says which one they'll use (Content-Type), and you settle on a common language. If there's no overlap, the conversation can't happen (406).

The idea, in plain language

A media type (a.k.a. MIME type) labels a body's format: application/json, text/html, text/csv, application/xml, multipart/form-data (file uploads), application/x-www-form-urlencoded (HTML form posts).

Content negotiation is how the client and server agree on a representation. The client sends Accept: text/csv, application/json;q=0.8 (with optional quality weights), and the server picks the best match it supports, replying with the matching Content-Type.

If the server can't satisfy the Accept header, it returns 406 Not Acceptable. If the client sends a body in a type the server can't handle, the server returns 415 Unsupported Media Type.

Most JSON APIs keep this simple: they always speak JSON and don't bother negotiating. Negotiation shines when one resource genuinely needs multiple formats (e.g. data + export).

l-status-familiesStatus codes: the five families

Key takeawayThe first digit of a status code tells the whole story: 1xx info, 2xx success, 3xx redirect, 4xx your fault, 5xx my fault.
Everyday analogy

A traffic-light system with five colors. The color (first digit) tells you the broad situation instantly; the exact number is the fine print. You can navigate by the color alone in an emergency.

The idea, in plain language

Every HTTP response carries a three-digit status code. The first digit classifies it into one of five families, and that classification is the single most useful thing to memorize.

  • 1xx — Informational. Interim, rare in REST (e.g. 100 Continue).
  • 2xx — Success. It worked (200, 201, 204).
  • 3xx — Redirection. Go elsewhere or use your cache (301, 304).
  • 4xx — Client error. The request was wrong; the client must change something (400, 401, 403, 404, 409, 422, 429).
  • 5xx — Server error. The server failed; the client did nothing wrong (500, 502, 503).

The 4xx-vs-5xx split is crucial: 4xx says 'don't blindly retry the same request, fix it,' while 5xx says 'it might work if you retry later.' Choosing the right family is the most common status-code mistake juniors make (e.g. returning 200 with an error inside, or 500 for a validation problem).

l-2xx2xx success codes

Key takeawayDon't reflexively return 200 for everything: 201 for created, 204 for 'done, nothing to send', 202 for accepted-but-async.
Everyday analogy

Different 'yes' answers at a counter. 'Here's your document' (200), 'Your new account is set up, here's the number' (201), 'Done, nothing to hand back' (204), 'We've received your request, we'll process it' (202). All are yes, but each carries different information.

The idea, in plain language

The 2xx family means the request succeeded. Using the specific code adds meaning for free:

200 OK — general success with a body. The default for successful GETs and many PUT/PATCH responses.

201 Created — a new resource was created (usually after POST). Include a Location header pointing to it.

202 Accepted — the request was accepted for asynchronous processing; it isn't finished yet. Common for jobs, emails, video encoding.

204 No Content — success, but there's intentionally nothing in the body. Common for DELETE and for PUT/PATCH when you don't return the updated resource.

Returning the right one lets clients act without guessing — e.g. a 201 tells the client 'a thing now exists, here's where.'

201 after sign-up/order creation; 204 after deletes and toggles; 202 in job queues, batch imports, and anything processed in the background.

l-3xx3xx redirection & caching codes

Key takeaway3xx says 'the answer is elsewhere or unchanged' — 301/308 move a resource, 304 tells a cache its copy is still good.
Everyday analogy

A forwarding address and a 'nothing new in your mailbox' note. 301 is 'this office moved permanently, update your records.' 304 is the mail carrier saying 'no new mail since last time — keep using what you have.'

The idea, in plain language

The 3xx family redirects the client or confirms a cached copy is still valid.

301 Moved Permanently / 308 Permanent Redirect — the resource has a new URL forever. Update links. (308 preserves the method; 301 historically let clients switch POST→GET.)

302 Found / 307 Temporary Redirect — temporarily elsewhere; don't update your records. (307 preserves the method.)

304 Not Modified — used with conditional requests. The client sends If-None-Match: <etag> (or If-Modified-Since); if the resource hasn't changed, the server replies 304 with no body, and the client reuses its cached copy. This saves bandwidth on unchanged data.

In JSON APIs, 301/302 are less common than on websites, but 304 + ETag is a valuable, underused optimization.

Most asked interview questions
QWhat is a 304 Not Modified and how is it triggered?

It signals the client's cached copy is still valid, so no body is sent. It's triggered by a conditional request: the client sends If-None-Match with a previously received ETag (or If-Modified-Since with a date). If the resource is unchanged, the server responds 304 and the client reuses its cache, saving bandwidth.

QDifference between 301 and 308?

Both mean the resource moved permanently. The difference is method handling: 308 guarantees the original method (e.g. POST) is preserved on the redirect, whereas 301 historically permitted clients to change the method to GET. Use 308 when preserving the method matters.

l-4xx4xx client errors

Key takeaway4xx means the caller must change something; pick the specific one — 400 malformed, 401 not logged in, 403 not allowed, 404 not found, 409 conflict, 422 validation, 429 rate-limited.
Everyday analogy

Reasons a bank teller turns you away. Your form is gibberish (400), you haven't shown ID (401), your ID is fine but you're not authorized for this account (403), the account number doesn't exist (404), someone already claimed that username (409), or you've asked too many times this minute (429).

The idea, in plain language

The 4xx family says the request was wrong. The client must fix something before it can succeed — retrying the identical request won't help.

400 Bad Request — the request is malformed (bad syntax, invalid JSON, missing required structure).

401 Unauthorized — authentication is missing or invalid. Really means 'unauthenticated — who are you?'

403 Forbidden — authenticated, but not permitted. Logging in again won't help.

404 Not Found — no resource at this URL. Also used to hide the existence of protected resources.

405 Method Not Allowed — the URL exists but not for this verb. Include an Allow header.

409 Conflict — the request conflicts with current state (duplicate, or an edit based on a stale version).

422 Unprocessable Entity — syntax is fine but the data fails validation (e.g. email already taken, negative price). Very common in REST APIs.

429 Too Many Requests — rate limit hit; check Retry-After.

The 401-vs-403 and 400-vs-422 distinctions are the two most-asked in interviews.

QWhat's the difference between 401 and 403?

QWhen would you use 400 vs 422?

To avoid leaking existence. A 403 confirms the resource exists but you can't see it; a 404 hides whether it exists at all. For sensitive resources, returning 404 to unauthorized callers prevents attackers from enumerating valid ids.

l-5xx5xx server errors

Key takeaway5xx means the server failed on a valid request — 500 generic crash, 502 bad upstream, 503 down/overloaded, 504 upstream timeout — and these are the ones worth retrying.
Everyday analogy

Problems on the restaurant's side, not yours. The kitchen caught fire (500), the supplier sent spoiled goods (502), the restaurant is temporarily closed for maintenance (503), or the supplier took too long to deliver (504). Your order was fine; come back and try again.

The idea, in plain language

The 5xx family means the server (or something it depends on) failed to handle an otherwise valid request. The client did nothing wrong, so a retry — ideally with exponential backoff — may succeed.

500 Internal Server Error — a generic, unexpected failure (usually an unhandled exception). The catch-all.

502 Bad Gateway — a server acting as a proxy/gateway got an invalid response from an upstream server.

503 Service Unavailable — the server is temporarily unable to handle the request (overloaded or in maintenance). Often paired with Retry-After.

504 Gateway Timeout — a gateway didn't get a response from upstream in time.

Critical rule: never leak internal details (stack traces, SQL, secrets) in a 5xx body. Log the details server-side with a correlation id; return a safe, generic message plus that id so support can trace it.

Servers and their dependencies fail; a standard failure family lets clients retry intelligently and lets ops teams alert on real outages.

  console.error(id, err);                 // full detail in logs only

// CLIENT: retry 5xx with exponential backoff
async function getWithRetry(url, tries = 4) {
  for (let i = 0; i < tries; i++) {
    const res = await fetch(url);
    if (res.status < 500) return res;     // 2xx/4xx: don't retry
    const wait = Math.min(1000 * 2 ** i, 8000); // 1s,2s,4s,8s
    await new Promise(r => setTimeout(r, wait));
  }
  throw new Error("Server unavailable after retries");
}

Walk-through

Because it's a 5xx, the client's retry logic waits 1s, then 2s, then 4s (exponential backoff). By the third attempt the upstream has recovered and the request succeeds.

Had you instead returned a raw 500 with the upstream's stack trace, you'd leak internal architecture and the client couldn't tell it was a transient upstream issue worth retrying. The right code plus a clean body makes the whole system resilient.

QWhy should you never return a stack trace in a 5xx response?

It leaks internal implementation details — file paths, library versions, SQL, sometimes secrets — which helps attackers and confuses clients. QDifference between 502, 503, and 504?

502 Bad Gateway: a proxy received an invalid response from an upstream server. 503 Service Unavailable: the server itself is temporarily down or overloaded (often with Retry-After). 504 Gateway Timeout: a proxy waited for an upstream response and it never arrived in time. All are transient and generally retryable with backoff.

QWhich status family should trigger automatic retries and how?

5xx (and sometimes 429). Retry with exponential backoff and jitter, a capped number of attempts, and only for idempotent operations (or POSTs guarded by an idempotency key). Honor Retry-After when present. Don't auto-retry 4xx — the request must change first.

l-jsonJSON: the data format

Key takeawayJSON is the near-universal REST payload format: human-readable text with objects, arrays, strings, numbers, booleans, and null.
Everyday analogy

A labeled fill-in form. Each blank has a name (key) and a value, and blanks can contain sub-forms (nested objects) or lists (arrays). Anyone can read it, and any language can parse it into its own data structures.

The idea, in plain language

JSON (JavaScript Object Notation) is a lightweight text format for structured data. It became the REST default because it's compact, human-readable, and parseable by every language.

It has exactly six value types: object { } (key–value pairs), array [ ] (ordered list), string (double-quoted), number, boolean (true/false), and null. That's it — no dates, no comments, no trailing commas.

Because JSON has no native date type, dates travel as strings, conventionally in ISO 8601 / RFC 3339 format (2026-07-01T09:30:00Z). Keys are strings and must be double-quoted.

Watch the type mismatch trap: JSON numbers are IEEE-754 doubles, so very large integers (like 64-bit IDs or money in cents beyond ~2^53) can lose precision. Represent big IDs and currency as strings to be safe.

l-body-designDesigning request & response bodies

Key takeawayKeep payloads consistent: predictable field names, a stable shape for success and errors, and only the data the client needs.
Everyday analogy

A well-designed form letter. Every letter uses the same layout, same field names, same date format — so the reader always knows where to look. Inconsistent letters (sometimes 'firstName', sometimes 'fname') force the reader to re-learn each one.

The idea, in plain language

The body is your API's data surface. Good bodies are consistent and predictable — that's worth more than clever.

Naming: pick one convention and never deviate. camelCase is common in JS-facing APIs; snake_case in many others. Consistency beats the specific choice.

Envelope or not? Two schools: return the raw resource ({ "id": 1, ... }), or wrap it ({ "data": { ... }, "meta": { ... } }). Envelopes give room for pagination/metadata and a uniform shape for success and errors; raw is simpler. Pick one and apply it everywhere.

Errors deserve a stable shape too — a machine-readable code, a human message, and optional field details. Never return an error inside a 200.

Only what's needed: don't dump internal columns. Expose a deliberate representation; never leak password hashes, internal flags, or secrets.

l-statelessStatelessness in practice

Key takeawayEach request must carry everything the server needs; the server remembers nothing about the client between requests.
Everyday analogy

A vending machine. It doesn't remember you between purchases. Every time, you insert full payment and make a selection — the machine needs no memory of your last visit. That's why one machine can serve everyone, and any identical machine works the same.

The idea, in plain language

Statelessness is REST's most consequential constraint. Every request must be self-contained: it includes the identity (a token), the target, and all inputs. The server does not keep per-client session state in memory between requests.

This does not mean the system has no state — the database absolutely holds state. It means the server process doesn't hold client session state. Authentication travels on each request (e.g. a bearer token), not a server-side session tied to one instance.

The payoff is huge: any server instance can handle any request, so you scale horizontally, restart instances freely, and put a load balancer in front without sticky sessions. It also makes requests easier to cache, log, and reason about in isolation.

The cost: each request is a bit larger (it re-sends auth/context), and you push shared state into a database or cache (like Redis) instead of process memory.

Diagram
Why it exists

Horizontal scaling and resilience require that any instance can serve any request. Server-held session state breaks that; statelessness enables it.

Where it's used

Cloud autoscaling groups, Kubernetes pods, serverless functions — all assume stateless request handling. Token-based auth exists largely because of it.

How to use it

Send identity/context on every request (bearer token); keep shared state in a DB/cache, not process memory. Avoid features that require the same instance to handle consecutive requests.

Walk-through

A user adds an item to their cart, then checks out a second later. With statelessness, request #1 might land on Server A and request #2 on Server C — and it just works, because each request carried the user's token and the cart lives in the shared database keyed by that user.

Contrast a stateful design where Server A stored the cart in its own memory. Now request #2 on Server C sees an empty cart, or you're forced into sticky sessions that break when Server A restarts. Statelessness is what lets the load balancer spread traffic freely.

Practice

A teammate proposes storing the logged-in user object in a server-side in-memory session after login, then relying on it for later requests. Why is this a problem for a service that runs 5 instances behind a load balancer, and what's the stateless alternative?

SolutionShow solution

The session lives only in the memory of the one instance that handled login. Later requests routed to a different instance won't find it, so the user appears logged out — unless you add sticky sessions, which break on scaling and restarts. Stateless alternative: issue a signed token (e.g. JWT) or a session id backed by a shared store (Redis/DB). The client sends it on every request via Authorization; any instance can validate it and load the user. No instance-local memory required.

Most asked interview questions
QWhat does 'stateless' mean in REST, and what does it NOT mean?

It means the server keeps no client session state between requests — each request carries all context it needs (identity, inputs). It does not mean the application has no state; persistent data still lives in a database. The constraint is specifically about not storing per-client session state in the server process.

QHow does statelessness enable scaling?

Because any request is self-contained, any server instance can handle it. You can add/remove instances behind a load balancer without sticky sessions, restart or replace instances freely, and even go serverless — all because no instance holds session state the next request depends on.

QWhat's the downside of statelessness and how is it mitigated?

Each request re-sends context (slightly larger requests) and shared state must live in an external store, adding a dependency. Mitigations: compact tokens (JWT) to carry identity cheaply, and fast shared caches (Redis) for session-like data. The scaling benefits generally outweigh these costs.

l-uri-designResource naming & URI design

Key takeawayUse plural nouns for collections, put ids in the path, keep verbs out of URLs, and stay lowercase-with-hyphens and consistent.
Everyday analogy

A well-organized filing cabinet. Drawers are labeled with plural nouns ('Invoices', 'Clients'), each folder has a number, and you never see a drawer labeled 'GetInvoice'. The structure is so predictable that anyone can find a file without instructions.

The idea, in plain language

Good URIs are guessable. Once someone sees /users/1, they should be able to guess /users, /users/1/orders, and how to create or delete. That predictability is the payoff of following conventions.

l-relationshipsRelationships & nested resources

Key takeawayExpress ownership with nesting (/users/1/orders) but don't nest more than a level or two; link deep relationships instead.
Everyday analogy

Folders inside folders. One or two levels ('Clients / Acme / Invoices') is navigable. Ten levels deep and nobody can find anything — you'd rather keep a flat index and cross-reference.

The idea, in plain language

Resources relate to each other, and URLs can express that. Nesting shows containment/ownership: /users/1/orders means 'the orders belonging to user 1.' This reads naturally and scopes queries.

But deep nesting gets unwieldy: /users/1/orders/9/items/3/reviews/7 is fragile and hard to consume. The common guideline: nest at most one level to show a relationship, then switch to a top-level resource with a filter or id.

Alternatively, link related resources in the response body ({"customer": "/users/1"}) so clients can follow references without deep URLs.

l-filteringFiltering, sorting & searching

Key takeawayRefine a collection with query parameters — filter by field, sort with a sign convention, and search with a q parameter — keeping the resource URL unchanged.
Everyday analogy

An online store's sidebar filters. The page is still 'shoes' (the resource), but you tick 'size 9', 'blue', and 'sort by price'. You didn't change which store section you're in — you refined the view. Query parameters are those sidebar controls.

The idea, in plain language

A collection like /products can return a lot. Query parameters let clients narrow and shape the result without new endpoints.

Filtering: match fields via ?field=value, e.g. /products?category=shoes&color=blue. Multiple params AND together by convention.

Sorting: a common convention is ?sort=field ascending and ?sort=-field descending (leading minus). Multiple keys: ?sort=-price,name.

Searching: free-text search usually uses ?q=term (or ?search=), separate from exact-match filters.

Two rules keep this clean: the resource URL never changes (it's still /products), and parameters must be URL-encoded. Document which fields are filterable/sortable so the API stays predictable and safe (never pass raw params into SQL).

function buildQuery(q) {
  const where = {};
  for (const [k, v] of Object.entries(q)) {
    if (FILTERABLE.has(k)) where[k] = v;        // ignore unknown params
  }
  let sort = q.sort || "createdAt";
  const dir = sort.startsWith("-") ? "DESC" : "ASC";
  sort = sort.replace(/^-/, "");
  if (!SORTABLE.has(sort)) sort = "createdAt";  // fall back safely
  return { where, orderBy: { [sort]: dir } };

l-paginationPagination

Key takeawayNever return unbounded lists; page results with offset/limit (simple) or cursor (stable & scalable), and include metadata to fetch the next page.
Everyday analogy

A book with pages vs one endless scroll. Offset pagination is 'go to page 5' — easy but if someone inserts pages while you read, page numbers shift. Cursor pagination is a bookmark ('continue after this exact line') — stable even as the book grows.

The idea, in plain language

Returning an entire collection is a performance and memory hazard. Pagination serves results in chunks.

Offset/limit (a.k.a. page-based): ?page=2&pageSize=20 or ?offset=20&limit=20. Simple and lets you jump to any page. Downsides: it gets slow on huge tables (the DB still scans the skipped rows) and can skip or duplicate items if data changes between page loads.

Cursor/keyset: the server returns an opaque nextCursor pointing just past the last item; the client sends ?cursor=...&limit=20 to continue. It's stable under inserts/deletes and stays fast at scale, but you can't jump to an arbitrary page.

Always include metadata so clients can continue: total (for offset) or a next cursor/next link. Enforce a max page size so clients can't request a million rows.

l-versioningVersioning

Key takeawayOnce others depend on your API, breaking changes need a new version; URL versioning (/v1/) is the most common and visible approach.
Everyday analogy

Editions of a textbook. Students mid-course keep using the 1st edition; new students get the 2nd. You don't silently change page 50 of everyone's book overnight — you publish a new edition and let people migrate.

The idea, in plain language

The moment external clients depend on your API, you can't freely change response shapes or remove fields — you'd break them. Versioning lets you evolve while old clients keep working.

The most common approach is URL versioning: /v1/users, /v2/users. It's explicit and easy to route, cache, and reason about. Alternatives: a header (Accept: application/vnd.myapi.v2+json) or a query param (?version=2).

Crucially, know what counts as a breaking change: removing/renaming a field, changing a type, changing status-code behavior, or making an optional field required. Additive changes (new optional fields, new endpoints) are usually safe and don't need a new version — as long as clients ignore unknown fields.

Best practice: version from day one (start at v1), avoid breaking changes when you can, and when you must, ship a new major version and give clients a deprecation window.

l-idempotencyIdempotency keys in depth

Key takeawayTo make a POST safe to retry, have the client send a unique idempotency key; the server records it and returns the original result on duplicates.
Everyday analogy

A confirmation number for an order. If you resubmit with the same confirmation number, the clerk says 'you already placed that — here's the same order,' instead of creating a second one. The number ties all retries to a single logical operation.

The idea, in plain language

Recall that POST isn't idempotent, so retries after a timeout can create duplicates (double orders, double charges). The industry-standard fix is an idempotency key.

The client generates a unique key per logical operation (e.g. a UUID) and sends it, typically as an Idempotency-Key header. On the first request, the server processes normally and stores the key with the result. If the same key arrives again (a retry), the server skips reprocessing and returns the original result.

Key details: the key must be stable across retries of the same operation (generate it once, before the first send, not per attempt). The server should store keys with an expiry (e.g. 24h) and guard against races (two identical requests arriving at once) with a lock or unique constraint. This turns an unsafe POST into something a client can retry confidently.

Diagram
Why it exists

Networks fail after the server acted but before the client heard back. Without idempotency keys, safe retries are impossible for POST — you'd risk duplicates.

Where it's used

Payments (Stripe, PayPal), order creation, money transfers, sending messages/emails — anywhere a duplicate would be costly or embarrassing.

How to use it

Client: generate one key per operation and reuse it on every retry. Server: store key→result with a TTL, return the stored result on repeats, and lock to handle concurrent duplicates.

Code

Generate the key ONCE, before the first attempt.

Client: one key, retried safely
async function charge(amount) {
  const key = crypto.randomUUID();          // one key per logical op
  for (let attempt = 0; attempt < 4; attempt++) {
    const res = await fetch("https://api.pay.com/charges", {
      method: "POST",
      headers: {
        "Content-Type": "application/json",
        "Idempotency-Key": key,             // SAME key every retry
      },
      body: JSON.stringify({ amount }),
    });
    if (res.status < 500) return res.json(); // success or client error
    await new Promise(r => setTimeout(r, 500 * 2 ** attempt));
  }
  throw new Error("charge failed after retries");
}

Return the original result for a repeated key.

  if (!key) return res.status(400).json({ error: "missing_idempotency_key" });

  const seen = await store.get(key);          // shared store (e.g. Redis/DB)
  if (seen) return res.status(seen.status).json(seen.body); // replay result

  // Process once, then remember the outcome (with a TTL, e.g. 24h)
  const charge = await createCharge(req.body);
  const result = { status: 201, body: charge };
  await store.set(key, result, { ttlSeconds: 86400 });
  res.status(201).json(charge);
});
Worked example
Walk-through

A checkout POSTs a $42 charge with Idempotency-Key: k_9f2a. The server creates charge c_1, stores k_9f2a → c_1, and replies 201 — but the response is lost to a dropped connection.

The client retries with the same key k_9f2a. The server sees the key already exists and returns the stored result (charge c_1) instead of creating a new charge. The customer is charged exactly once, even though the client sent the request twice. That's the entire value: exactly-once semantics on top of an at-least-once network.

Practice

A developer generates a fresh idempotency key inside the retry loop (a new UUID each attempt). Why does this defeat the purpose?

SolutionShow solution

The whole mechanism relies on retries sharing the same key so the server can recognize them as one operation. Generating a new key per attempt makes each retry look like a brand-new request, so the server processes each one and you get duplicates — exactly what idempotency keys are meant to prevent. The fix: generate the key once before the first attempt and reuse it for every retry of that logical operation.

Most asked interview questions
QHow do idempotency keys make POST safe to retry?

The client sends a unique key per logical operation (e.g. via an Idempotency-Key header). The server processes the first request, stores the key with its result, and for any later request bearing the same key returns the stored result instead of reprocessing. This gives exactly-once behavior for creates/charges even when the client must retry after a timeout.

QWhat server-side concerns arise when implementing idempotency keys?

Storing keys with a TTL/expiry; handling concurrency (two identical requests racing) with a lock or a unique constraint so only one is processed; deciding what to store (status + body) to replay; and scoping keys correctly (per endpoint/account) so different operations don't collide. The store must be shared across instances to remain stateless-compatible.

Authentication & security

Who are you, and what may you do? API keys, Basic auth, JWT, OAuth 2.0, HTTPS, and CORS — the security layer every real API needs.

l-authn-authzAuthentication vs authorization

Key takeawayAuthentication proves who you are; authorization decides what you're allowed to do. They're different steps, and the codes differ (401 vs 403).
Everyday analogy

An office building. Authentication is showing your ID badge at the door to prove you're an employee. Authorization is whether that badge opens the server room. You can be authenticated (a valid employee) yet unauthorized (no server-room access).

The idea, in plain language

Two distinct questions, often confused:

Authentication (authN): 'Who are you?' The client proves its identity — typically by presenting a credential (API key, token, username/password). If it fails, the server returns 401 Unauthorized (really 'unauthenticated').

Authorization (authZ): 'What are you allowed to do?' Given a known identity, the server checks permissions/roles/scopes. If the identity is valid but lacks permission, the server returns 403 Forbidden.

Order matters: authenticate first, then authorize. A request with no credentials to a protected endpoint is 401; a request with valid credentials but insufficient rights is 403.

Everything else in this unit (API keys, JWT, OAuth) is about how to do authentication (and, for OAuth scopes, part of authorization). Keep the two concepts separate in your head and it all stays clear.

Why it exists

Conflating identity with permission leads to security holes (e.g. any logged-in user editing anyone's data). Separating them makes access control explicit.

Where it's used

Every protected endpoint runs both: middleware authenticates the token, then a policy check authorizes the specific action on the specific resource.

Authenticate the caller (validate their credential), then authorize the action (check roles/ownership/scopes) before doing the work. Return 401 vs 403 accordingly.

  if (!user) return res.status(401).json({ error: "unauthenticated" });
  req.user = user;
  next();
}

// 2) Authorization: are you allowed to do THIS?
function requireOwnerOrAdmin(req, res, next) {
  const targetId = req.params.id;
  const ok = req.user.role === "admin" || req.user.id === targetId;
  if (!ok) return res.status(403).json({ error: "forbidden" });
  next();
}

Walk-through

If Alice had sent no token at all, she'd never reach the authorization step — she'd get 401 Unauthorized because we can't even establish who she is. Same endpoint, two different failures depending on which question failed.

Practice

A junior dev writes middleware that returns 401 whenever a user lacks permission for an action. What's wrong, and what should each case return?

SolutionShow solution

Using 401 for a permission failure is incorrect. 401 means the request lacks valid authentication — the server doesn't know who you are. If the user is authenticated but not allowed to perform the action, the correct code is 403 Forbidden. So: no/invalid credentials → 401; valid credentials but insufficient rights → 403. Mixing them up misleads clients (a 401 suggests 're-login might help,' which is false for a permissions issue).

Most asked interview questions
QWhat's the difference between authentication and authorization?

Authentication verifies identity ('who are you?') using credentials/tokens; authorization determines permissions ('what are you allowed to do?') using roles, ownership, or scopes. Authentication happens first. In HTTP, failed authentication is 401 Unauthorized; failed authorization (authenticated but not permitted) is 403 Forbidden.

QWhy is 401 called 'Unauthorized' when it really means unauthenticated?

It's a naming quirk from the HTTP spec. Despite the word 'Unauthorized,' 401 signals missing or invalid authentication credentials. Genuine authorization failures — authenticated but lacking permission — use 403 Forbidden.

l-api-keysAPI keys

Key takeawayAn API key is a simple shared secret that identifies the calling application; easy to use, but it must travel over HTTPS in a header and be treated like a password.
Everyday analogy

A building's numeric keypad code shared with a contractor. It gets them in and identifies 'the contractor,' but anyone who learns the code can enter. So you keep it secret, rotate it if leaked, and don't tape it to the door.

The idea, in plain language

An API key is a long, random string the server issues to a client. The client sends it on each request to identify itself. It's the simplest auth scheme, common for server-to-server calls and public data APIs.

Send it in a header (e.g. Authorization: Bearer <key> or a custom X-API-Key), never in the URL (URLs get logged and cached). Always over HTTPS, or it's exposed in transit.

Keys mainly do authentication of the app (and enable rate-limiting/usage tracking per key). They're a bearer credential: whoever holds the key can use it, so leakage is serious. Mitigations: scope keys to least privilege, allow rotation, support multiple keys, and let users revoke a compromised key.

Limitation: a plain API key doesn't identify an end user or carry fine-grained permissions well — that's where tokens/OAuth come in.

Practice

QHow should an API key be transmitted, and why not in the URL?

API keys are bearer secrets that mainly identify an application, not an end user, and carry only coarse permissions. l-basic-authHTTP Basic authentication

Key takeawayBasic auth sends base64(username:password) in the Authorization header — simple, but only safe over HTTPS, since base64 is encoding, not encryption.
Everyday analogy

Writing your username and password on a postcard in a mild code anyone can read, then mailing it. Fine if the postcard travels in a sealed, tamper-proof tube (HTTPS). Risky on an open postcard (plain HTTP), because base64 'code' is trivial to reverse.

The idea, in plain language

Basic authentication is built into HTTP. The client sends Authorization: Basic <credentials>, where credentials are base64(username:password).

The crucial caveat: base64 is encoding, not encryption. Anyone who sees the header can decode it back to the raw username and password instantly. So Basic auth is only acceptable over HTTPS, where TLS encrypts the whole exchange.

It's stateless (credentials sent every request, matching REST) and dead simple, but it has real downsides: the password travels on every request; there's no easy logout/expiry; and it's poor for user-facing apps. It's mostly seen in internal tools, quick scripts, and simple server-to-server setups.

When the server wants Basic auth, it can reply 401 with a WWW-Authenticate: Basic header, prompting the client (or browser) for credentials.

Why it exists

Sometimes you need auth with zero infrastructure — an internal endpoint, a script, a quick prototype. Basic auth is built into HTTP and needs no token system.

Where it's used

Internal dashboards, CI health checks, simple server-to-server calls behind HTTPS, and legacy systems.

How to use it

Send Authorization: Basic base64(user:pass) over HTTPS only. For anything user-facing or public, prefer tokens; treat Basic as a minimal, HTTPS-gated option.

Walk-through

You build an internal metrics endpoint. You add Basic auth: the client sends Authorization: Basic YWRhOnMzY3JldA==. On the wire over HTTPS this is safe because TLS encrypts everything, including the header.

l-jwtJWT (JSON Web Tokens)

Key takeawayA JWT is a signed, self-contained token carrying claims (like user id and roles); the server verifies its signature instead of looking up a session — perfect for stateless auth.
Everyday analogy

A tamper-evident festival wristband. It states who you are and which areas you can access (claims), and it's sealed so staff can verify it's genuine at a glance (signature) without phoning the box office (no DB lookup). But anyone can read what's printed on it — so don't print secrets.

The idea, in plain language

A JWT (pronounced 'jot') is a compact, URL-safe token with three parts separated by dots: header.payload.signature.

The header says the signing algorithm. The payload holds claims — data like sub (subject/user id), exp (expiry), iat (issued-at), roles, and scopes. The signature is computed over header+payload using a secret (HMAC) or a private key (RSA/EC).

The magic: because the token is signed, the server can verify it wasn't tampered with using its key — no database session lookup needed. That makes JWTs a natural fit for REST's statelessness: the token carries the identity, any instance can validate it.

Two things beginners miss: (1) the payload is only base64url-encoded, not encrypted — anyone can read it, so never put secrets in it. (2) You can't easily revoke a JWT before it expires (it's valid until exp), so keep access tokens short-lived and pair them with a refresh-token flow.

import jwt from "jsonwebtoken";
const SECRET = process.env.JWT_SECRET;   // keep this secret!

// On successful login: issue a short-lived access token
function issueToken(user) {
  return jwt.sign(
    { sub: user.id, role: user.role, scope: "read:orders" }, // claims
    SECRET,
    { expiresIn: "15m" }                 // short-lived on purpose
  );
}

// On each protected request: verify signature + expiry
function authenticate(req, res, next) {
  const token = (req.header("Authorization") || "").replace("Bearer ", "");
  try {
    req.user = jwt.verify(token, SECRET); // throws if tampered/expired
    next();
  } catch {
    res.status(401).json({ error: "invalid_or_expired_token" });
  }
}

Practice

A developer stores the user's password and a 'is_premium' flag inside the JWT payload to avoid DB lookups. Name two problems.

SolutionShow solution

(1) Password in the payload is a serious leak — JWT payloads are base64url-encoded, not encrypted, so anyone who obtains the token can read the password instantly. Never put secrets in a JWT. (2) Stale/immutable data: the token can't be updated until it expires, so if the user's premium status changes (upgrade/downgrade/refund), the JWT keeps asserting the old value. Most asked interview questions
QWhat are the three parts of a JWT?

Header (algorithm/type), payload (the claims — e.g. sub, exp, roles), and signature (computed over header+payload with a secret or private key). They're base64url-encoded and joined by dots. The signature lets the server verify integrity without a database lookup.

QIs a JWT encrypted? What does that imply?

No — a standard signed JWT is only base64url-encoded, so its payload is readable by anyone who holds the token. It's signed for integrity, not encrypted for secrecy. The implication: never store sensitive data in the payload; use it for identity/claims only, and rely on TLS to protect it in transit.

QHow do you handle JWT revocation and expiry?

JWTs can't easily be revoked before their exp because they're validated by signature, not lookup. Mitigations: keep access tokens short-lived (minutes) and use a refresh-token flow to obtain new ones; maintain a server-side denylist of revoked token ids (jti) for emergencies; or rotate signing keys. Short lifetimes limit the damage window of a stolen token.

l-oauthOAuth 2.0 (high level)

Key takeawayOAuth 2.0 lets a user grant a third-party app limited access to their data on another service — without sharing their password — via a token with scopes.
Everyday analogy

A hotel key card. The front desk (authorization server) checks your ID once and issues a card (access token) that opens only your room and the gym (scopes) for a limited time. You never hand your master identity to the cleaning app; you hand it a scoped, expiring card.

The idea, in plain language

OAuth 2.0 is a framework for delegated authorization. It answers: 'How can App B access some of my data on Service A without me giving App B my Service A password?' (Think 'Log in with Google,' or letting a photo-printing app read your cloud photos.)

The four roles: the resource owner (you, the user), the client (the third-party app), the authorization server (issues tokens after you consent), and the resource server (the API holding your data).

The modern flow for web/mobile apps is the Authorization Code flow (with PKCE): the app redirects you to the auth server, you log in and consent to specific scopes (e.g. read:photos), the auth server returns a short code, and the app exchanges that code for an access token (often plus a refresh token). The app then calls the API with Authorization: Bearer <access token>.

Key ideas: the user's password never touches the third-party app; access is scoped and expiring; and it can be revoked. (OpenID Connect, or OIDC, layers authentication/identity on top of OAuth's authorization.)

Users shouldn't hand their password to every third-party app, and apps shouldn't hold god-mode access. OAuth grants narrow, revocable, expiring access instead.

Where it's used

'Log in with Google/GitHub/Apple,' apps that read your calendar/photos/repos, and enterprise SSO. It's the backbone of third-party API access.

How to use it

As an API provider, run (or integrate) an authorization server, define scopes, and accept bearer access tokens. As a client, use the Authorization Code + PKCE flow and store tokens securely.

l-httpsHTTPS & TLS

Key takeawayHTTPS wraps HTTP in TLS to give encryption, integrity, and server authentication — non-negotiable for any API that carries credentials or personal data.
Everyday analogy

Sending a letter in a sealed, tamper-evident, verified-courier envelope instead of an open postcard. TLS seals the contents (encryption), shows if anyone opened it (integrity), and proves the courier is really who they claim (authentication via certificate).

The idea, in plain language

HTTPS is just HTTP running over a TLS (Transport Layer Security) connection. TLS provides three guarantees:

Encryption: the request and response — including headers like Authorization and the body — are unreadable to anyone intercepting the traffic.

Integrity: tampering in transit is detected; an attacker can't silently modify the data.

Authentication: the server presents a certificate (issued by a trusted Certificate Authority) proving it's really api.shop.com and not an impostor. This defends against man-in-the-middle attacks.

Everything else in this unit — API keys, Basic auth, JWTs, OAuth tokens — is a bearer credential that would be trivially stolen over plain HTTP. TLS is the foundation that makes them safe to send. Modern practice: HTTPS everywhere, redirect HTTP to HTTPS, and use HSTS to force it.

Walk-through

You send Authorization: Bearer <jwt> to your API. Over HTTPS, an attacker sharing the coffee-shop Wi-Fi sees only encrypted bytes — they can't read the token, alter the request, or impersonate the server (the certificate wouldn't match).

Practice

Your API is served over both HTTP and HTTPS 'for compatibility,' and clients sometimes hit the HTTP endpoint. What's the risk and the fix?

Risk: any request to the HTTP endpoint sends credentials/tokens and data in cleartext, where they can be intercepted or modified — and an attacker can strip/downgrade clients to HTTP. Supporting HTTP undermines all your auth. QWhat three guarantees does TLS provide?

Encryption (traffic is unreadable to interceptors), integrity (tampering is detected), and server authentication via a certificate (you're talking to the real server, not an impostor). Together they prevent eavesdropping and man-in-the-middle attacks.

l-corsCORS (cross-origin requests)

Key takeawayCORS is a browser security rule; your API opts in to being called from other origins by sending Access-Control-Allow-* headers. It's server-configured, not a client bug.
Everyday analogy

A guest list at a private club. By default, the browser's bouncer won't let a script from site-A read responses from api-B. CORS is api-B publishing a guest list ('site-A is allowed') so the bouncer lets it through. The bouncer is the browser; the guest list is your API's headers.

The idea, in plain language

Browsers enforce the same-origin policy: by default, JavaScript on https://app.com can't read responses from a different origin like https://api.other.com. An 'origin' is scheme + host + port. This protects users from malicious cross-site reads.

CORS (Cross-Origin Resource Sharing) is how a server opts in to allowing specific other origins. The server sends headers like Access-Control-Allow-Origin: https://app.com, and the browser then permits the JS to read the response.

For 'non-simple' requests (custom headers, methods like PUT/DELETE, JSON with certain content types), the browser first sends a preflight OPTIONS request asking 'is this allowed?' The server answers with the allowed origins, methods, and headers before the real request is sent.

Two things trip people up: CORS is enforced by browsers only (server-to-server and curl ignore it), and CORS errors are fixed on the server (by sending the right headers), not by changing client code. Avoid the lazy Access-Control-Allow-Origin: * with credentials — it's insecure.

Diagram
Why it exists

Browsers must stop a random malicious site from silently reading your bank's API using your logged-in session. CORS is the controlled exception that lets legitimate cross-origin apps work.

Where it's used

Any single-page app on one domain calling an API on another (e.g. app.com → api.com); public APIs meant for browser use configure CORS deliberately.

How to use it

On the server, set Access-Control-Allow-Origin to your allowed origins (not * if you use credentials), plus allowed methods/headers, and handle the OPTIONS preflight. Client code doesn't 'fix' CORS.

Code

Allow specific origins; handle preflight.

Configuring CORS on the server (Express)
const ALLOWED = new Set(["https://app.com", "https://admin.app.com"]);

app.use((req, res, next) => {
  const origin = req.header("Origin");
  if (ALLOWED.has(origin)) {
    res.set("Access-Control-Allow-Origin", origin);   // echo the allowed one
    res.set("Vary", "Origin");                        // correct caching
    res.set("Access-Control-Allow-Methods", "GET,POST,PUT,PATCH,DELETE");
    res.set("Access-Control-Allow-Headers", "Content-Type,Authorization");
    // res.set("Access-Control-Allow-Credentials", "true"); // if using cookies
  }
  if (req.method === "OPTIONS") return res.sendStatus(204); // preflight
  next();
});
Worked example
Walk-through

Your SPA on https://app.com calls PUT https://api.other.com/users/1 with an Authorization header. The browser first fires a preflight OPTIONS asking whether app.com may send a PUT with an Authorization header. If the API responds with the matching Access-Control-Allow-* headers, the browser proceeds with the real PUT and lets your JS read the result.

If the API doesn't send those headers, you get the classic console error: 'blocked by CORS policy.' Note what happened — the request may have even reached the server, but the browser refused to expose the response to your JS. The fix is on the API (send the headers), never a client hack.

Practice

A frontend dev gets 'blocked by CORS policy' calling your API and asks you to 'fix it in the fetch call.' Explain why that's the wrong place and what actually fixes it.

SolutionShow solution

CORS is enforced by the browser based on server response headers; no client-side fetch option can grant cross-origin read access that the server hasn't allowed. The fix is on the API: send Access-Control-Allow-Origin (set to the frontend's origin, or a validated allow-list) plus Access-Control-Allow-Methods and Access-Control-Allow-Headers, and respond to the preflight OPTIONS request. If cookies/credentials are used, also set Access-Control-Allow-Credentials: true and do not use * for the origin. (A dev-only workaround is a proxy, but the real fix is server CORS config.)

Most asked interview questions
QWhat is CORS and what problem does it address?

CORS (Cross-Origin Resource Sharing) is a browser mechanism that relaxes the same-origin policy. By default, browser JavaScript can't read responses from a different origin; CORS lets a server explicitly opt in to allowing specific origins via Access-Control-Allow-* headers. It balances enabling legitimate cross-origin apps against protecting users from malicious cross-site reads.

QWhat is a CORS preflight request?

For 'non-simple' requests (e.g. methods like PUT/DELETE, custom headers, or certain content types), the browser first sends an OPTIONS request asking the server which origins, methods, and headers are permitted. Only if the server's response approves does the browser send the actual request. It's an automatic browser step, not something the developer issues.

QWhere are CORS errors fixed — client or server?

On the server, by returning the correct Access-Control-Allow-* headers and handling preflight. CORS is enforced by the browser based on server headers, so no client-side change can bypass it (aside from routing through a same-origin proxy). l-error-handlingError handling & error responses

Key takeawayReturn the right status code plus a consistent, machine-readable error body with a code, message, and (for validation) field details — and never leak internals.
Everyday analogy

A good rejected-form letter. It says exactly which box was wrong and how to fix it ('date of birth must be in the past'), in plain language, without dumping the clerk's private notes. A bad one just says 'ERROR' or attaches the whole internal manual.

The idea, in plain language

Errors are part of your API's contract, not an afterthought. Great error handling has three ingredients.

Correct status code: use the family and specific code that matches reality (4xx for client mistakes, 5xx for server failures). This drives client retry logic and monitoring.

Consistent body: every error — across every endpoint — uses the same shape: a stable code (machine-readable, e.g. validation_error), a human message, optional details (per-field issues), and a requestId for support/tracing.

Safe by default: never expose stack traces, SQL, file paths, or secrets. Log the gory details server-side keyed by requestId; return a clean message to the caller.

A widely-adopted standard for error bodies is RFC 9457 Problem Details (application/problem+json). You don't have to use it, but a consistent, documented shape is the point.

               details: err.details, requestId },
    });
  }
  // Unknown error: log full detail, return safe generic message
  console.error(requestId, err);

  });
});

l-rate-limitingRate limiting & throttling

Key takeawayCap how many requests a client can make in a window; signal remaining quota with headers and return 429 with Retry-After when exceeded.
Everyday analogy

A theme-park ride with an hourly rider limit. It protects the ride from breaking under a crush, keeps wait times fair for everyone, and staff tell you 'come back in 10 minutes' rather than letting the line collapse. Rate limits protect your API the same way.

The idea, in plain language

Rate limiting restricts how many requests a client may make in a time window (e.g. 100 requests/minute). It protects the API from overload, abuse, and runaway clients, and enforces fair use and billing tiers.

Common algorithms: fixed window (simple counter per minute), sliding window (smoother), and token bucket (allows short bursts up to a cap, then a steady refill rate — the most popular).

Communicate limits with headers so good clients can self-regulate: X-RateLimit-Limit, X-RateLimit-Remaining, and X-RateLimit-Reset (or the standardized RateLimit headers). When a client exceeds the limit, respond 429 Too Many Requests with a Retry-After header telling them how long to wait.

Limits are usually keyed by API key, user id, or IP. Well-designed clients read the remaining count and back off before hitting 429.

Why it exists

Without limits, one buggy or malicious client can exhaust capacity for everyone, spike your bill, or take the service down. Limits enforce fairness and stability.

Where it's used

Every public API (GitHub, Twitter/X, Stripe) rate-limits; internal APIs limit to contain incidents; login endpoints limit to slow brute-force attacks.

How to use it

Choose a key (API key/user/IP) and an algorithm (token bucket is a safe default), expose remaining-quota headers, and return 429 + Retry-After when exceeded.

Back off using the server's own guidance.

async function callWithRateLimit(url, opts) {
  for (let i = 0; i < 5; i++) {
    const res = await fetch(url, opts);
    if (res.status !== 429) return res;
    const retry = Number(res.headers.get("Retry-After") || 2 ** i);
    await new Promise(r => setTimeout(r, retry * 1000));
  }
  throw new Error("Still rate-limited after retries");
}

Walk-through

Your plan allows 100 requests/minute. A client makes 88; each response advertises X-RateLimit-Remaining counting down. A well-behaved client sees it's near zero and slows down. A greedy client keeps firing and, on request 101, gets 429 Too Many Requests with Retry-After: 30.

The client's retry logic reads Retry-After and waits 30s instead of hammering. Meanwhile, the theme-park effect: because you capped this one client, every other client's requests still go through. The limit turned a potential outage into a mild, self-correcting slowdown for one caller.

Practice

A client complains they 'randomly' get 429s. You look and see they fire 200 requests in a 1-second burst every minute against a 100/minute limit. Explain what's happening and how both sides could handle it better.

SolutionShow solution

They're exceeding the limit in a burst: 200 in one second blows past 100/minute immediately, so the excess returns 429 — not random at all. Client side: read X-RateLimit-Remaining and spread requests out; on 429, honor Retry-After with backoff instead of retrying instantly. Server side: a token bucket could permit a modest burst while enforcing the average rate, and clear rate-limit headers help the client self-regulate. Also ensure the 429 body/headers clearly state the limit and reset time.

Most asked interview questions
QWhy do APIs rate-limit, and what status/header signal it?

To protect against overload and abuse, ensure fair usage, and enforce plan tiers. When a client exceeds the limit the API returns 429 Too Many Requests, ideally with a Retry-After header indicating how long to wait, and often X-RateLimit-* headers advertising the limit, remaining quota, and reset time.

QCompare token bucket and fixed window rate limiting.

Fixed window counts requests per calendar interval — simple but allows a burst at the boundary (e.g. full quota at 0:59 and again at 1:00). Token bucket holds up to N tokens that refill at a steady rate; each request spends a token, allowing controlled bursts while enforcing the long-run average. Token bucket is more flexible and widely used.

l-cachingHTTP caching

Key takeawayUse Cache-Control to say how long a response stays fresh, and ETags to revalidate cheaply with 304 — cutting latency, bandwidth, and load.
Everyday analogy

A library reference desk. Some answers are posted on a board and good for a week (Cache-Control: don't ask again yet). For others you show the librarian your old copy's version stamp and they say 'still current, keep it' (ETag + 304) instead of re-copying the whole book.

The idea, in plain language

Caching stores responses so repeat requests can be served without redoing work or re-sending data. REST is designed to be cacheable, which is a big reason the web scales.

Freshness (Cache-Control): the server declares how long a response can be reused, e.g. Cache-Control: max-age=300 (fresh for 5 minutes). While fresh, clients/CDNs serve the cached copy with no request at all. Use no-store for sensitive data and private to keep responses out of shared caches.

Validation (ETag): the server tags a response with an ETag (a version fingerprint). Later, the client sends If-None-Match: <etag>; if unchanged, the server replies 304 Not Modified with no body, and the client reuses its copy. This revalidates cheaply when freshness has lapsed.

Caching applies mainly to GET (safe). Mutations must bust or update caches. Done right, caching turns many expensive requests into near-free ones.

Why it exists

Recomputing and re-sending identical data for every request wastes CPU, bandwidth, and time. Caching is one of the cheapest, biggest performance wins available.

Where it's used

CDNs cache public GETs at the edge; browsers/mobile apps cache and revalidate; conditional requests slash payload sizes for polling clients.

How to use it

Set Cache-Control freshness on cacheable GETs, emit ETags and honor If-None-Match for cheap revalidation, and use no-store/private for anything user-specific or sensitive.

Code
Server: freshness + ETag validation (Express)
app.get("/articles/:id", async (req, res) => {
  const article = await db.getArticle(req.params.id);
  const etag = `"v${article.version}"`;

  // Cheap revalidation: if client's copy matches, send 304 (no body)
  if (req.header("If-None-Match") === etag) return res.status(304).end();

  res.set("ETag", etag);
  res.set("Cache-Control", "public, max-age=60"); // fresh for 60s
  res.json(article);
});
Worked example
Walk-through

A mobile app polls GET /articles/88 every minute. First call: 200 with the full article, ETag: "v7", and Cache-Control: max-age=60. For the next 60 seconds the app serves its cached copy with no request at all.

After a minute, the copy is stale, so the app revalidates: If-None-Match: "v7". The article hasn't changed, so the server returns 304 Not Modified — just headers, no body. The app keeps its copy. Across thousands of devices, you've turned heavy repeated downloads into tiny 304s, saving bandwidth and server load while keeping data reasonably fresh.

Practice

You add Cache-Control: public, max-age=3600 to GET /me (the current user's private profile). Why is this dangerous, and what should you use instead?

SolutionShow solution

public lets shared caches (proxies, CDNs) store the response, and a 1-hour max-age means one user's private profile could be cached and then served to a different user hitting the same URL through a shared cache — a serious data leak. Fix: for per-user/sensitive data use Cache-Control: private, no-store (or at least private with a short max-age and proper Vary/auth handling) so it's never held in shared caches. Reserve public caching for genuinely shared, non-sensitive resources.

Most asked interview questions
QWhat's the difference between freshness and validation caching?

Freshness (via Cache-Control: max-age) lets a cache reuse a response without contacting the server until it expires. Validation (via ETag + If-None-Match, or Last-Modified + If-Modified-Since) lets the client cheaply check whether its (now-stale) copy is still current; if so the server returns 304 with no body. Freshness avoids requests entirely; validation avoids re-sending the body.

QHow does an ETag save bandwidth?

The server sends an ETag (a version fingerprint) with the response. On a later request the client includes If-None-Match with that ETag; if the resource is unchanged, the server replies 304 Not Modified with an empty body, so only headers cross the network instead of the full payload.

l-hateoasHATEOAS & the maturity model

Key takeawayHATEOAS means responses include links to related actions so clients navigate by following links; it's REST's top maturity level, and mostly optional in practice.
Everyday analogy

A well-designed website vs a phone tree. A website gives you clickable links on each page — you discover where to go next without memorizing URLs (HATEOAS). A phone tree makes you know 'press 4 then 2 then 7' in advance (a client hardcoding every endpoint).

The idea, in plain language

HATEOAS — Hypermedia As The Engine Of Application State — is the idea that API responses include links telling the client what it can do next, so clients discover actions by following links rather than hardcoding URLs.

It's easiest to see via the Richardson Maturity Model, which grades how 'RESTful' an API is:

  • Level 0: one URL, one verb (usually POST) — RPC over HTTP, not REST.
  • Level 1: multiple resources (URLs), still mostly POST.
  • Level 2: resources + proper HTTP methods + status codes. Most real 'REST' APIs live here.
  • Level 3: Level 2 + HATEOAS (hypermedia links). The 'full' REST.

In practice, most successful APIs stop at Level 2. HATEOAS is elegant — it lets servers change URLs without breaking clients and makes APIs self-describing — but it adds complexity and few clients fully exploit it. Know what it is and why it exists; you'll rarely be required to implement Level 3, though a links section (like pagination links) is a light, pragmatic taste of it.

Diagram
Why it exists

Hardcoding every URL couples clients tightly to the server. Hypermedia links let servers evolve and guide clients dynamically — the original REST vision.

Where it's used

Some enterprise/hypermedia APIs (HAL, JSON:API links); pagination next/prev links are the most common real-world use even in Level-2 APIs.

How to use it

At minimum, include useful links (pagination, related resources) in responses. Full HATEOAS is optional — adopt it when clients truly benefit from discoverability.

Once the order is paid, the server drops the pay link and maybe adds a refund link. The client's UI adapts automatically. That's the promise of HATEOAS: the server drives available actions. In reality most teams find this overkill and settle for Level 2 plus a few practical links (like pagination), which is a perfectly respectable, common choice.

l-validationInput validation & safety

Key takeawayNever trust input: validate type, presence, and range on the server, reject with 400/422, and use parameterized queries to block injection.
Everyday analogy

Airport security screening every bag regardless of who's carrying it. It doesn't matter how trustworthy the passenger looks (or what the client claims to have already checked) — everything is screened at the gate, because that's the last line of defense.

The idea, in plain language

Every request from outside is untrusted, even from your own frontend (attackers can bypass it and call the API directly). Server-side validation is mandatory; client-side checks are only for UX.

Validate three things for each field: presence (required?), type/format (is it a number, a valid email?), and range/constraints (0–120 for age, max length, allowed enum values). Reject bad input early with 400 (malformed) or 422 (well-formed but invalid) and clear field-level errors.

Beyond shape, guard against attacks: use parameterized queries / ORM bindings (never string-concatenate SQL) to prevent SQL injection; sanitize/encode output to prevent stored XSS; cap sizes and pagination to prevent resource exhaustion; and apply an allow-list for fields you accept (to prevent mass-assignment of things like isAdmin).

Use a schema/validation library (Zod, Joi, Pydantic, Bean Validation) so validation is declarative, consistent, and hard to forget.

Practice

Your endpoint builds SQL as "SELECT * FROM users WHERE name = '" + req.query.name + "'". Show the attack and the fix.

SolutionShow solution

Attack (SQL injection): a client sends ?name=' OR '1'='1, producing ... WHERE name = '' OR '1'='1', which matches every row — dumping the whole table. Worse payloads can modify or delete data. Fix: never concatenate input into SQL. Use a parameterized query: db.query("SELECT * FROM users WHERE name = $1", [req.query.name]) (or an ORM binding). Most asked interview questions
QWhy validate on the server if the client already validates?

Client-side validation is only for user experience — it can be bypassed by calling the API directly (curl, Postman, a modified client). The server is the real trust boundary, so it must independently validate every input. Skipping server validation invites bad data and serious vulnerabilities.

QHow do you prevent SQL injection?

Use parameterized/prepared statements or an ORM that binds parameters, so user input is always passed as data, never concatenated into the SQL string. Combine with input validation, least-privilege database accounts, and selecting only needed columns. The core rule: never build queries by string concatenation with untrusted input.

QWhat is a mass-assignment vulnerability and how do you stop it?

It occurs when the server blindly binds all fields from the request body to a model, letting an attacker set fields they shouldn't (e.g. isAdmin, balance). Prevent it by explicitly allow-listing the fields you accept per endpoint (via a schema/DTO) and ignoring everything else, rather than saving the raw request object.

l-rest-vsREST vs GraphQL, gRPC, SOAP & WebSockets

Key takeawayREST is the versatile default, but GraphQL suits flexible client-driven queries, gRPC suits fast internal services, SOAP lingers in enterprises, and WebSockets handle real-time.
Everyday analogy

Modes of transport. REST is a car — goes almost anywhere, everyone can drive one. GraphQL is a personal shopper who fetches exactly your list in one trip. gRPC is a high-speed freight rail between warehouses. SOAP is an old but strict courier with heavy paperwork. The idea, in plain language

REST isn't the only way to build an API. Knowing the alternatives — and when each shines — is a common interview theme.

GraphQL: a single endpoint where the client specifies exactly which fields it wants in one query. Great for avoiding over-fetching/under-fetching and for varied frontends, at the cost of more server complexity and trickier caching.

gRPC: a high-performance RPC framework using HTTP/2 and binary Protocol Buffers. Excellent for internal microservice-to-microservice calls and streaming; less browser-friendly and not human-readable.

SOAP: an older, strict, XML-based protocol with formal contracts (WSDL) and built-in standards (WS-Security). Verbose, but still found in banking, government, and legacy enterprise systems.

WebSockets: not a REST replacement but a complement — a persistent, two-way connection for real-time data (chat, live prices, multiplayer). REST for normal CRUD, WebSockets for live pushes.

The pragmatic take: REST remains the default for public and general-purpose APIs because it's simple, cacheable, and universally understood. Reach for the others when their specific strength matches your problem.

QWhen would you choose GraphQL over REST?

Most real systems use REST for standard operations and WebSockets (or SSE) specifically where the server must push updates continuously.

Everyday analogy

A standardized nutrition label plus assembly instructions for your API. Anyone — human or machine — can read the same label to know exactly what goes in, what comes out, and how to use each part, without reverse-engineering the product.

The idea, in plain language

An API is only as usable as its documentation. OpenAPI (formerly Swagger) is the industry-standard, machine-readable format for describing REST APIs — a single YAML/JSON document listing every endpoint, its parameters, request/response schemas, status codes, and auth.

Out-of-date, hand-written docs are a top complaint about APIs. A machine-readable spec keeps docs accurate and unlocks auto-generated clients and validation, saving huge effort.

Design: resources are nouns; collections plural; ids in the path; nest one level max; options in the query string; lowercase-hyphenated URLs; versioned (/v1).

Semantics: correct method per operation (GET safe, POST create, PUT replace, PATCH partial, DELETE remove); correct status codes (201+Location on create, 204 on delete, right 4xx vs 5xx); honor idempotency/safety.

Data: consistent JSON shapes and naming; a single error shape with code/message/details/requestId; never leak internals; never hide errors in a 200.

Robustness & security: HTTPS everywhere; authenticate then authorize (401 vs 403); validate all input server-side; parameterized queries; pagination with caps; rate limiting (429 + Retry-After); caching (Cache-Control/ETag) where useful; idempotency keys for unsafe retries.

Documentation & testing: OpenAPI spec as the contract; test happy path, errors, auth, and edge cases. Nail these and you're operating at a professional level.

Practice

Do a lightning code review. An endpoint is GET /v1/getUserOrders?userId=5, returns 200 with { "error": "not found" } when the user doesn't exist, dumps the raw DB rows including an internalNotes column, and has no pagination. List every issue and the fix.

SolutionShow solution

Issues: (1) Verb in URL (getUserOrders) — use GET /v1/users/5/orders (noun, nested one level). (2) id in query string — put it in the path. (3) Error returned as 200 — a missing user should be 404 (or empty list if the user exists but has no orders), never a 200 hiding an error. (4) Inconsistent error shape — use the standard { error: { code, message, requestId } }. (5) Leaking internal data (internalNotes) — map to a deliberate response representation excluding internal columns. (6) No pagination — add ?page/pageSize (or cursor) with a max cap and return paging metadata. Bonus: confirm it requires auth and enforces that the caller may view user 5's orders (ownership/admin → else 403). That's the checklist in action.

Most asked interview questions
QWhat separates a well-designed REST API from a poor one?

Consistency and correct use of HTTP: resource-based, versioned URLs (plural nouns, ids in path); the right method and status code for every operation; consistent JSON payloads and a single error shape that never leaks internals or hides errors in 200s; security (HTTPS, authenticate-then-authorize, server-side validation, parameterized queries); scalability features (pagination, caching, rate limiting, idempotency for unsafe retries); thoughtful versioning; and accurate documentation (OpenAPI) with tested error paths. A poor API violates these — verbs in URLs, wrong/optimistic status codes, inconsistent shapes, trusting the client, unbounded responses, and stale docs.

QYou're reviewing a teammate's new endpoint. What do you check?

Run the checklist: is the URL a noun (plural, versioned, id in path, shallow nesting)? Correct method and status codes (201+Location, 204, proper 4xx/5xx)? Consistent payload and error shape with no leaked internals and no errors buried in 200s? HTTPS, authentication then authorization (401 vs 403)? Server-side input validation and parameterized queries? Pagination with a cap on list endpoints? Idempotency for unsafe retries where relevant? Versioning/backward-compatibility of any change? And tests plus updated docs. Each 'no' is a concrete, principled review comment.
